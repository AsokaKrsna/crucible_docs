    Scenario Authoring Guide¶

    What Is A Scenario?¶

    A scenario defines the blockchain environment at a specific point in time — usually moments before a known exploit. It tells Crucible:

    1. Which chain(s) to fork (e.g., Ethereum at block 15,259,100)
    2. What state to verify (e.g., check that a storage slot is at a specific value)
    3. What setup steps to run (e.g., deploy a helper contract, fund an account)
    4. What assertions to make (e.g., the exploit succeeds, state changes as expected)

    Think of a scenario as the "before" snapshot — it sets up the chessboard so the attack plugin can make its move.

    Why Are Scenarios Separate From Attack Plugins?¶

    Component Purpose Responsible For
    Scenario (scenarios/<name>/scenario.yaml) Environment setup — fork the chain, prepare state, run preconditions Declarative YAML
    Attack Plugin (attacks/<name>/<name>.py) Exploit execution — forge calldata, send transactions, verify success Python code
    Experiment (experiments/<name>_replay.yaml) Orchestration — wires scenario + attack plugin + topology together Declarative YAML

    This separation means you can: - Reuse the same scenario with different attack plugins (test variations) - Reuse the same attack plugin with different scenarios (test on different fork blocks) - Swap topologies without changing the attack code

    Anatomy Of A Scenario¶

    name: my_attack_pre_exploit                        # Must match directory name
description: >-
  One or two paragraphs explaining what this scenario does.
  What chain is being forked? At what block? What state
  is being verified? What contract is the target?
version: "1.0.0"
chains:                                              # Which chains to fork
  - ethereum

phases:                                              # Ordered list of phases
  - name: setup                                      # Phase 1: setup
    steps:
      - type: log
        message: "Forking Ethereum at block 15259100 for Nomad reproduction"
  - name: attack                                     # Phase 2: attack (plugin runs here)
    steps:
      - type: log
        message: "Attack plugin will be executed by the experiment runner"
  - name: verify                                     # Phase 3: verification
    steps:
      - type: assert
        name: exploit_succeeds
        expected: true
        actual: true

    Alternative: Top-Level Steps (Simple Format)¶

    For simpler scenarios that don't need distinct setup, attack, and verify phases, Crucible supports a flat list of steps at the root level using the steps: key:

    name: simple_scenario
chains: [ethereum]
steps:
  - type: log
    message: "This scenario runs as a single block of steps"
  - type: impersonate
    chain: ethereum
    account: "0xAttacker"
    When using this format, the scenario runs entirely before the attack plugin executes.

    Directory Structure¶

    scenarios/
  my_attack_pre_exploit/
    scenario.yaml          # The scenario definition (required)
    steps.py               # Optional Python steps (only if using python_script step type)

    Phases And Execution Order¶

    Scenarios run in ordered phases. Each phase contains one or more steps that execute sequentially.

    flowchart LR
    P1["Phase 1: setup<br>Run setup steps<br>(log, impersonate, send_tx, etc.)"]
    P2["Phase 2: attack<br>Experiment runner loads and executes<br>the attack plugin between phases"]
    P3["Phase 3: verify<br>Run verify steps<br>(assert, log)"]

    P1 --> P2
    P2 --> P3

    Important: The attack plugin runs between the scenario phases, not inside them. The experiment runner: 1. Executes setup phase steps 2. Loads and runs the attack plugin (execute() method) 3. Runs the attack plugin's verify() method 4. Executes the remaining phase steps

    Checkpoint System: Per-Phase Snapshots¶

    As each phase completes, Crucible's CheckpointManager saves a phase checkpoint — a full EVM state snapshot for each running chain, hashed for integrity verification.

    What gets saved: 

    states/checkpoints/
  <run_id>/
    checkpoint.json             # Manifest: run_id, experiment, attack, completed_phases, timestamps
    integrity.json              # SHA-256 hashes of all snapshot files
    setup.ethereum.json         # EVM state snapshot after 'setup' phase
    attack.ethereum.json        # EVM state snapshot after 'attack' phase
    verify.ethereum.json        # EVM state snapshot after 'verify' phase

    How it works: 1. After each phase completes, CheckpointManager.save_phase_checkpoint() dumps anvil_dumpState for each chain in the active topology 2. Each snapshot is validated (JSON parse + re-serialize) before being hashed 3. All SHA-256 hashes are written to integrity.json — the authoritative record of snapshot integrity 4. The checkpoint.json manifest tracks which phases are complete

    On resume: 1. CheckpointManager.verify_checkpoint() loads the manifest and verifies ALL snapshot file hashes against integrity.json 2. If any file is missing or corrupt, CheckpointIntegrityError is raised — preventing silent data corruption 3. The experiment runner restores each chain's state from the most recent phase snapshot via anvil_loadState — this is done by ExperimentRunner.resume(), not by the CheckpointManager itself 4. Execution continues from the next uncompleted phase

    # Resume an interrupted experiment
crucible experiment resume DS-2026-00155

# List all available checkpoints
crucible experiment checkpoints

    Note: Checkpoint snapshots only capture chains in the active topology. Stale .meta.json files from previous topologies are skipped automatically — see Troubleshooting — Stale Meta Files.

    When to use checkpoints: - Long-running experiments — if a multi-hour experiment crashes at phase 3 of 5, you don't need to re-run phases 1-2 - L4 cross-chain experiments — bridge deployment + configuration can take several minutes; checkpointing saves that work - Multiple attack attempts — run attack variant A, reset --mode soft to checkpoint state, run attack variant B

    Checkpoint storage limits: 

    # testbed.yaml
cleanup:
  max_checkpoints: 10            # Max checkpoint runs to keep (oldest removed)

    Checkpoints are cleaned by crucible clean --checkpoints or crucible clean --all. See CLI Reference — Clean for details.

    Step Types (Complete Reference)¶

    1. log — Record A Message¶

    Logs a message to the experiment run log. Useful for documenting what's happening.

    - type: log
  message: "Fork block verified — contract exists at expected address"

    Purpose: Provides human-readable documentation in the experiment report. Every important step should be logged.

    2. assert — Verify A Condition¶

    Asserts that a condition is true. Fails the experiment if the condition is false.

    - type: assert
  name: vulnerability_exists          # Unique name for this assertion
  expected: true                      # What we expect
  actual: true                        # What we observe (can reference state)

    Purpose: Formal verification that the exploit state or outcome meets expectations. These appear in the experiment report as pass/fail.

    3. impersonate — Impersonate An Account¶

    Instructs Anvil to impersonate an account (bypass private key checks). Useful for sending transactions from accounts you don't control.

    - type: impersonate
  chain: ethereum                     # Which chain
  account: "0xRelayerAddress..."      # Address to impersonate

    Purpose: Allows the scenario to send transactions from privileged addresses (relayers, governors, keepers) that are controlled by the protocol, not by you.

    4. send_tx — Send A Transaction¶

    Sends a transaction to a contract. Supports symbolic address resolution for bridge contracts.

    - type: send_tx
  chain: ethereum                     # Which chain
  from: "0xAttackerAddress..."        # Sender (must be impersonated first if not yours)
  to: "bridge.ethereum"               # Recipient (can use symbolic name like "bridge.ethereum")
  value: 1000000000000000000          # Value in wei (1 ETH = 10^18 wei)
  data: "0x123456..."                 # Calldata (hex-encoded)

    Purpose: Execute a transaction on the forked chain. Used to trigger contract functions, send funds, etc.

    5. advance_time — Skip Time Forward¶

    Increases the EVM timestamp and mines a new block. Used to bypass time-based restrictions (challenge windows, vesting periods, etc.).

    - type: advance_time
  chain: polygon                      # Which chain
  seconds: 3600                       # Number of seconds to advance (3600 = 1 hour)

    Purpose: Simulate the passage of time in the EVM. Useful for exploits that rely on time-based state changes.

    6. assert_balance — Check Account Balance¶

    Asserts that an account's balance meets a minimum threshold. Fails if the balance is below the expected value.

    - type: assert_balance
  chain: ethereum                     # Which chain
  account: "0xVaultAddress..."        # Account to check
  min_value: 0                        # Minimum balance in wei

    Purpose: Verify that funds moved as expected after an exploit. For example, assert that the attacker's balance increased.

    7. forge_script — Run A Foundry Script¶

    Executes a Foundry (Solidity) script via subprocess. The RPC URL is automatically injected.

    - type: forge_script
  chain: ethereum                     # Which chain to target
  script: "Exploit.s.sol:ExploitScript"  # Foundry script path
  extra_args:                         # Optional arguments
    - "--sender"
    - "0xAttackerAddress..."

    Purpose: Run sophisticated exploit logic written in Solidity. Use this when the exploit logic is too complex for simple YAML steps.

    Requirements: The Foundry script must be in the scenarios/<name>/ directory (or a scripts/ directory in the project root).

    8. python_script — Run A Python Script¶

    Dynamically imports and executes a steps.py file located in the scenario directory.

    - type: python_script

    Your steps.py must define a run(context: ExperimentContext) function:

    from crucible.rpc import JsonRpcClient
from crucible.sdk import ExperimentContext

def run(context: ExperimentContext):
    # Get the RPC URL for the forked chain
    rpc_url = context.chains.runtime_for("ethereum").rpc_url
    client = JsonRpcClient(rpc_url)

    # Custom logic goes here
    # ...

    context.log.step("Custom Python step executed",
                     phase="attack", action="custom", status="success")

    Purpose: Execute arbitrary Python logic that can't be expressed in YAML steps alone.

    9. verify_tracing — Verify Transaction Trace Capture¶

    Triggers a verification that the instrumentation proxy correctly captured transaction traces for a given chain. Raises an error if trace artifacts are missing or incomplete.

    - type: verify_tracing
  chains:                             # Which chain(s) to verify traces for (list; defaults to all chains if omitted)
    - ethereum
    - polygon

    Purpose: Ensures that all transactions sent during a phase were captured by the instrumentation proxy. Useful in multi-phase scenarios where each phase must produce complete artifacts.

    10. bridge_action — Execute A Bridge Protocol Action¶

    Performs a bridge-specific action through the active bridge plugin. The available actions depend on the bridge plugin loaded for the topology.

    - type: bridge_action
  action: send_message                # Bridge action name (e.g. send_message, relay_messages, check_delivery, verify_state)
  source_chain: ethereum              # Source chain
  dest_chain: polygon                 # Destination chain
  data: "0x..."                       # Action-specific data

    Purpose: Interact with bridge plugins from within a scenario. Used for cross-chain workflows where bridge state needs to be set up before or verified after an attack.

    L4 bridge workflow example:

    # scenarios/my_bridge_scenario/scenario.yaml
name: my_bridge_scenario
chains:
  - ethereum
  - polygon

phases:
  - name: setup
    steps:
      - type: log
        message: "Bridge deployed automatically via lifecycle — ready for attack"

  - name: attack
    steps:
      - type: log
        message: "Attack plugin runs here — both chains and bridge are ready"

  - name: verify
    steps:
      - type: bridge_action
        action: verify_state
        chain: ethereum
      - type: log
        message: "Cross-chain state verified"

    Topology for L4:

    # topologies/nomad_bridge.yaml
name: nomad_bridge
chains: [ethereum, polygon]
bridge: nomad_bridge    # Points to bridges/nomad_bridge/

    # experiments/my_l4_experiment.yaml
name: my_l4_experiment
scenario: my_bridge_scenario
attack: my_l4_attack
topology: nomad_bridge
run:
  traces: true
  storage_diffs: true
tags: [l4, bridge]

    Available bridge plugins: | Plugin | Chains | Vulnerability Model | |---|---|---| | nomad_bridge | ethereum + polygon | committedRoot == 0x00 — forged message accepted | | poly_network_bridge | ethereum + polygon + bsc | Hash collision in _executeCrossChainTx | | wormhole_bridge | ethereum + polygon | Guardian bypass — forged VAA accepted |

    11. plugin — Run A Plugin Step (Deprecated)¶

    Note: This step type is currently a no-op — it only logs a "skipped" status. The documented dispatch behavior (method:/phase:) is not yet implemented. See [#issue] for tracking.

    - type: plugin
  method: "prepare"                   # Not yet dispatched
  phase: 0                            # Not yet handled

    How Scenarios Relate To Experiments¶

    The experiment file wires everything together:

    # experiments/my_attack_replay.yaml
name: my_attack_replay              # Experiment name (used in log directory path)
scenario: my_attack_pre_exploit      # Which scenario to use (from scenarios/<name>/)
attack: my_attack                    # Which attack plugin to load (from attacks/<name>/)
topology: single                     # Which topology to use (from topologies/<name>.yaml)
run:
  traces: true                       # Enable transaction tracing
  storage_diffs: true               # Enable storage diff capture
  gas_profiling: true               # Enable gas profiling
tags: [my_attack, l2]              # Tags for filtering in history queries

    What happens when you run crucible experiment run experiments/my_attack_replay.yaml:

    1. Load the topology (e.g., single = Ethereum only)
    2. Load the scenario (from scenarios/my_attack_pre_exploit/)
    3. Load the attack plugin (from attacks/my_attack/)
    4. Fork the chain(s) at the fork block auto-merged from evidence.yaml — the attack's evidence.yaml historical_scope.fork_blocks drives the fork block automatically. Check the experiment output for: fork_block updated from X to Y (from evidence.yaml).
    5. Run the scenario's setup phase steps
    6. Start instrumentation (trace capture)
    7. Run the attack plugin's execute() method
    8. Run the attack plugin's verify() method
    9. Run the remaining scenario phase steps
    10. Save artifacts and generate report

    [ok] Auto-merge: The fork block is now automatically merged from evidence.yaml at runtime. When you run crucible experiment run, it reads attacks/<name>/evidence.yaml and updates the config's fork_block to match what evidence.yaml specifies. You no longer need to manually edit testbed.local.yaml to change fork blocks between experiments.

    If the chain is already running at a different block, the stale chain is terminated and re-forked at the correct block automatically.

    Only override manually if you need a different fork block than what evidence.yaml declares (e.g., testing a patch at a different block).

    To bypass auto-merge entirely: use the --no-auto-merge flag: 

    crucible experiment run experiments/my_replay.yaml --no-auto-merge
    This keeps the current fork block unchanged, even if evidence.yaml specifies a different one.

    Common Patterns¶

    Pattern 1: Simple — Only Logs (Attack Plugin Does Everything)¶

    When the attack plugin handles all exploit logic, the scenario just documents what's happening:

    name: nomad_pre_exploit
description: >-
  Fork Ethereum at block 15259100 for Nomad bridge reproduction.
  The attack plugin handles all exploit logic.
chains:
  - ethereum
phases:
  - name: setup
    steps:
      - type: log
        message: "Starting Nomad replica fraud reproduction at fork block 15259100"
  - name: attack
    steps:
      - type: log
        message: "Attack plugin executed by experiment runner"

    Pattern 2: Pre-Exploit State Setup¶

    When the exploit requires setting up state before running the attack:

    name: bridge_exploit_pre_exploit
chains:
  - ethereum
  - polygon
phases:
  - name: setup
    steps:
      - type: log
        message: "Setting up pre-exploit state"
      - type: impersonate
        chain: ethereum
        account: "0xRelayerAddress"
      - type: send_tx
        chain: ethereum
        from: "0xRelayerAddress"
        to: "0xBridgeContract"
        data: "0x..."
  - name: attack
    steps:
      - type: log
        message: "Ready for attack plugin execution"

    Pattern 3: Post-Exploit Verification¶

    When the scenario needs to verify state after the attack:

    phases:
  - name: setup
    steps:
      - type: log
        message: "Setup complete"
  - name: attack
    steps:
      - type: log
        message: "Attack in progress"
  - name: verify
    steps:
      - type: assert
        name: attacker_balance_increased
        expected: true
        actual: true
      - type: log
        message: "Verification passed"

    Best Practices¶

    1. Log every meaningful step — Each log step becomes a line in the experiment report. This is how reviewers understand what happened.

    2. Use descriptive assertion names — Names like exploit_succeeds or keeper_key_replaced are much more useful than assertion_1.

    3. Keep scenarios declarative — Prefer YAML steps over python_script or forge_script when possible. YAML is self-documenting.

    4. Set fork blocks in evidence.yaml — The fork_blocks field in attacks/<name>/evidence.yaml is now the source of truth. Crucible auto-merges it at runtime. No need to edit testbed.yaml or testbed.local.yaml unless you need to override.

    5. One scenario per attack — Don't reuse scenarios for different attacks unless they're truly identical environments.

    6. Verify fork block auto-merge — When you run crucible experiment run, the fork block is auto-merged from attacks/<name>/evidence.yaml. Check the output for: fork_block updated from X to Y (from evidence.yaml). If the wrong block is used, update evidence.yaml's fork_blocks, not testbed.yaml.

    7. Document multi-chain scenarios — If your scenario uses multiple chains, list all of them in chains: and explain why each is needed.

    Troubleshooting¶

    Symptom Likely Cause Fix
    Scenario not found Wrong directory name Check scenarios/<name>/scenario.yaml exists
    Unknown step type Typo in step type Use one of the 11 types above
    python_script missing steps.py No steps.py in scenario directory Create scenarios/<name>/steps.py or use a different step type
    Chain X not in topology Scenario references a chain not in the topology Add the chain to the topology or remove it from the scenario
    Fork block mismatch testbed.yaml has wrong fork_block Update evidence.yaml's fork_blocks (auto-merged), or use testbed.local.yaml to override
