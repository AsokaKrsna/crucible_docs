    Attack Pack Authoring Guide¶

    Purpose: A step-by-step guide to create new Crucible attack packs following the established patterns of the 6 completed benchmark reproductions. This guide explains what each component does AND why it exists.

    Quick Overview: What Is An Attack Pack?¶

    An attack pack is a self-contained directory that tells Crucible everything it needs to reproduce a specific exploit. It has three parts:

    attacks/<name>/                  ← The attack plugin (the "what" of the exploit)
scenarios/<name>_pre_exploit/   ← The environment setup (the "where" of the exploit)
experiments/<name>_replay.yaml  ← The wiring (ties everything together)

    Why Three Separate Parts?¶

    Part Purpose Analogy
    Attack Plugin (attacks/) What the attack does — forge calldata, send transactions, verify success The chef's recipe
    Scenario (scenarios/) Where the attack happens — which chain, at what block, what preconditions The kitchen setup
    Experiment (experiments/) How to run it — which topology, what config, what tags The dinner menu

    This separation lets you: - Reuse an attack across different environments (test on different fork blocks) - Reuse a scenario for different attack variants (test multiple exploit approaches) - Swap infrastructure (Docker vs local) without changing attack or scenario code

    File-By-File Guide¶

    plugin.yaml — The Plugin Manifest¶

    Purpose: Tells Crucible how to find, validate, and load your attack code.

    Why it exists: Crucible needs to know your plugin's name, type, entrypoint, chain requirements, and topology requirements before it can load and run it.

    name: "your_attack_name"          # Must match the directory name
type: attack                       # Always "attack" for attack plugins
version: "1.0.0"                  # Semantic version for your attack pack
author: "your-name"               # Your name or team name
entry: "your_attack.py"           # The Python file containing your plugin class
class: "YourAttackClass"          # The class name inside the entry file
requires:
  crucible: ">=1.0.0"            # Crucible version requirement
  chains:                         # Which chains this attack needs
    - ethereum
  topology: single                 # Which topology this attack needs (single | bridge | dual | single_bsc | harmony_bridge)

    Common mistakes: - Using class_name: (legacy format) instead of class: (new format) — both work but new is preferred - Forgetting to update chains: when adding multi-chain support - Setting topology: to a topology that doesn't include all required chains

    evidence.yaml — The Research Evidence¶

    Purpose: Declares all the on-chain data, sources, and assertions that back up your reproduction. This is the most important research artifact — it proves you've done your homework.

    Why it exists: Without evidence, a reproduction is just code. The evidence file proves that: 1. You found the real exploit data (contracts, transactions, blocks) 2. You know what the exploit did and how it worked 3. You can define what "success" looks like

    incident:
  name: "Human-readable exploit name"
  date: "YYYY-MM-DD"                # Date of the actual exploit
  affected_bridge: "Bridge Name"    # Protocol name
  funds_lost_usd: 611000000         # Total funds lost across all chains
  summary: >                        # Technical summary — explain ROOT CAUSE, not impact
    One paragraph explaining the vulnerability.
    What was the bug? How was it triggered?
    What was the result?

sources:
  advisories:                        # Security advisories from the project
    - "https://..."
  postmortems:                       # Technical postmortems (Immunefi, Certik, Halborn)
    - "https://..."
  tx_explorers:                      # Links to the actual exploit transactions
    - "https://etherscan.io/tx/0x..."
  papers: []                         # Academic papers (if any)

historical_scope:
  chains:                            # Which chains were involved
    - "ethereum"
  fork_blocks:                       # Block BEFORE the first exploit tx on each chain
    # This is now the SOURCE OF TRUTH for fork blocks. Crucible auto-merges
    # these values into the chain config at runtime via `crucible experiment run`.
    # You no longer need to manually set `fork_block` in testbed.yaml or
    # testbed.local.yaml — the evidence.yaml drives the fork block automatically.
    ethereum: 15259100
  exploit_transactions:              # The exact exploit transaction hash(es)
    - "0x..."
  affected_contracts:                # Contract addresses (VERIFIED on block explorer!)
    EthCrossChainManager: "0x838bf9E9..."
    EthCrossChainData: "0xcF2afe10..."
  attacker_addresses:                # Known attacker wallets
    - "0x..."
  vulnerability_function: >          # The vulnerable function name
    verifyHeaderAndExecuteTx / putCurEpochConPubKeyBytes

reproduction_scope:
  fidelity_level: L2                 # See fidelity levels below
  exact_tx_replay: false             # True if replaying the exact historical tx
  modeled_components:                # What you CAN reproduce
    - "Ethereum-side keeper key replacement via EthCrossChainManager.call"
  excluded_components:               # What you CANNOT reproduce (off-chain, external)
    - "BSC-side asset withdrawal mechanics"
    - "Polygon-side asset withdrawal mechanics"

acceptance:
  required_assertions:               # What must be true for reproduction to be valid
    - name: vulnerability_exists
      description: "The vulnerable state exists at fork block"
      expected: true
    - name: exploit_succeeds
      description: "The exploit transaction does not revert"
      expected: true

    <name>.py — The Attack Plugin (Python)¶

    Purpose: Contains the actual exploit reproduction logic — connecting to the forked chain, forging calldata, sending transactions, and verifying results.

    Why it exists: YAML is great for declaring state, but exploits need code. The Python plugin gives you full access to: - Forge arbitrary calldata using ABI encoding - Impersonate accounts (run transactions as if you control them) - Read and write storage slots - Emit structured logs with metadata - Define pass/fail verification logic

    from crucible.sdk import AttackPlugin, AttackResult, ExperimentContext, VerificationResult
from crucible.rpc import JsonRpcClient

class YourAttack(AttackPlugin):
    def metadata(self) -> dict:
        """Describe the attack for the experiment manifest."""
        return {
            "name": "your_attack",
            "version": "1.0.0",
            "type": "attack",
            "fidelity_level": "L2",
        }

    def execute(self, context: ExperimentContext, phase: int | None = None) -> AttackResult:
        """Execute the exploit reproduction logic."""
        # 1. Get the RPC URL for the forked chain
        rpc_url = context.chains.runtime_for("ethereum").rpc_url
        client = JsonRpcClient(rpc_url)

        # 2. Verify preconditions
        # ...

        # 3. Execute exploit logic
        # ...

        # 4. Log progress — these appear in the experiment report
        context.log.step("Exploit executed successfully",
                         phase="attack", action="exploit", status="success")

        # 5. Return result
        return AttackResult(success=True, phase_results=[...])

    def verify(self, context: ExperimentContext) -> VerificationResult:
        """Independent verification that the exploit worked."""
        # Read storage slots, check balances, verify events
        # Return pass/fail with detailed assertions
        return VerificationResult(passed=True, assertions=[...], errors=[])

    scenario.yaml — The Environment Definition¶

    See docs/SCENARIO_AUTHORING.md for a complete guide. At minimum:

    name: my_attack_pre_exploit
description: >-
  Fork Ethereum at block XXXXXX for [attack name] reproduction.
chains:
  - ethereum
phases:
  - name: setup
    steps:
      - type: log
        message: "Starting reproduction at fork block XXXXXX"
  - name: attack
    steps:
      - type: log
        message: "Attack plugin executed by experiment runner"

    experiments/<name>_replay.yaml — The Wiring¶

    Purpose: Tells the experiment runner which scenario, attack plugin, and topology to use.

    name: my_attack_replay              # Used for log directory naming
scenario: my_attack_pre_exploit      # Points to scenarios/<name>/
attack: my_attack                    # Points to attacks/<name>/
topology: single                     # Points to topologies/<name>.yaml
run:
  traces: true                       # Capture transaction traces
  storage_diffs: true               # Capture storage slot changes
  gas_profiling: true               # Capture gas usage
tags: [my_attack, l2]              # Tags for filtering in crucible history

    Getting Started Faster With crucible attack extract¶

    💡 Scaffold from a transaction hash: crucible attack extract <tx_hash> --name <name> --chain <chain> auto-generates the plugin scaffold, evidence.yaml, and scenario from an on-chain transaction. It fetches the ABI, decodes the function call, and pre-fills contract addresses, fork blocks, and the exploit date from the block timestamp.

    [warn] Reliability note: This is a scaffolding tool, not a fully automated exploit generator. Expect the following limitations: - ABI decoding is best-effort (~60% success rate) — complex calldata (nested tuples, proxy patterns, fallback functions) may not decode correctly - The generated plugin.py contains TODO stubs — never working exploit logic. You must implement the actual execute() method yourself. - ABI fetch can fail for unverified contracts on Sourcify/Etherscan - Evidence metadata is incomplete — postmortem URLs, fidelity level, and affected bridge name must be filled in manually

    Use attack extract to save 15–20 minutes on file scaffolding, but always review and customize every generated file. See the step-by-step guide below for how to fill in the gaps.

    Step 1: Research On-Chain Data¶

    Before writing any code, gather data from block explorers (Etherscan, Polygonscan, BscScan) and postmortems (rekt.news, Immunefi, Certik, Halborn).

    What To Collect¶

    Data Point Why You Need It Where To Find It Example
    Fork block Sets the chain state to moments before the exploit Block of the first exploit tx MINUS 1 15259100 (block before 15259101)
    Exploit tx hash(es) Used in evidence.yaml and to verify calldata Explorer tx page 0x5ea16a...
    Vulnerable contract(s) Target of your attack plugin Explorer address page (must show [ok] "Verified") 0xB9233675...
    Attacker wallet Source of the exploit transactions "From" field on the exploit tx 0xA87fB85...
    Postmortem links Source citation for evidence.yaml rekt.news, Immunefi, project blogs https://...
    Funds lost Impact metric for evidence.yaml rekt.news or on-chain analysis $611,000,000
    Vulnerable function The function that was exploited Transaction input data decoded via explorer verifyHeaderAndExecuteTx

    Critical Rule: Verify Contract Addresses On Etherscan¶

    This is the #1 source of bugs in attack packs. During initial validation, 3 out of 6 attack packs had wrong contract addresses — they were EOAs (regular user wallets), not contracts. The addresses looked plausible because they had similar prefixes.

    How to verify: 1. Paste the address into Etherscan 2. Check the top of the page — it should say "Contract" (not "Address (EOA)") 3. Look for a blue checkmark [ok] next to the name (verified source code) 4. Check that the contract has more than 0 transactions

    How To Find The Correct Fork Block¶

    1. Find the exploit transaction on Etherscan — paste the known exploit tx hash into the Etherscan search bar. If you only have the attacker's wallet address, go to their transaction history page and look for the transaction that calls the vulnerable function.
    2. Note the block number (e.g., 12996694)
    3. Subtract 1 → that's your fork block (e.g., 12996693)
    4. Double-check: at the fork block, the vulnerable contract should exist and the exploit should NOT have happened yet

    Step 2: Update evidence.yaml¶

    Fill in all the data from Step 1. See the template above for the complete structure.

    Key fields to get right: - fork_blocks — must be block BEFORE the exploit. This is now the source of truth for fork blocks. The auto-merge feature reads fork_blocks from evidence.yaml and applies it to the chain config at runtime. You no longer need to manually set fork_block in testbed.yaml or testbed.local.yaml. - affected_contracts — must be verified on the block explorer - exploit_transactions — must be the ACTUAL exploit tx (not an attacker's message or unrelated tx) - modeled_components — be honest about what you can and cannot reproduce - excluded_components — document everything you're NOT reproducing (off-chain, non-EVM, etc.)

    Step 3: Write The Attack Plugin¶

    The Standard Pattern (From Working Plugins)¶

    1. Get the RPC URL: context.chains.runtime_for("ethereum").rpc_url
    2. Create a client: JsonRpcClient(rpc_url)
    3. Verify preconditions: Check that the vulnerable contract exists and has the expected state
    4. Execute the exploit: Forge calldata, impersonate accounts, send transactions
    5. Log everything: Use context.log.step() for structured logging — these appear in the experiment report
    6. Verify success: Check storage slots, balances, events
    7. Return results: AttackResult from execute(), VerificationResult from verify()

    Step 6: Validate & Test¶

    # 1. Validate the plugin structure
docker compose run --rm orchestrator plugin validate attacks/my_attack

# 2. Confirm it's discovered
docker compose run --rm orchestrator plugin list --type attack

# 3. Fork and run (auto-merges fork_block from evidence.yaml)
# No testbed.local.yaml editing needed!
docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/my_attack_replay.yaml --no-baseline
"

# 4. Check the results
docker compose run --rm orchestrator logs list

# 5. Run twice for determinism proof
# (auto-merge sees fork_block matches, skips re-forking)
docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/my_attack_replay.yaml --no-baseline
"

# 6. Compare the two runs
# `diff` checks gas, storage, events, balances
docker compose run --rm orchestrator diff <run_a> <run_b>

    Common Pitfalls (And How To Avoid Them)¶

    [fail] Wrong Contract Address (EOA instead of Contract)¶

    • Error: get_code() returns "0x", eth_call returns empty
    • Why it happens: The address looks like a contract address but is actually a regular wallet
    • How to avoid: Always paste the address into Etherscan and check it says "Contract"

    [fail] Wrong Fork Block¶

    • Error: Vulnerable state is not present (storage slots are wrong, contract doesn't exist yet)
    • Why it happens: Fork block is set to the exploit block (or after), or too early (before contract deployment)
    • How to avoid: Use block BEFORE the first exploit tx. Verify by checking a known storage value at the fork block.

    [fail] Wrong Exploit Transaction¶

    • Error: The transaction in evidence.yaml doesn't actually exploit the vulnerability
    • Why it happens: The attacker's wallet made many transactions — you picked the wrong one
    • How to avoid: Check the transaction's function signature. The exploit tx should call the vulnerable function.

    [fail] Python sha3_256() Instead Of Ethereum keccak256¶

    • Error: ABI encoding mismatches, wrong event signature hashes
    • Why it happens: Python's hashlib.sha3_256() is NIST SHA-3 (FIPS 202), NOT Ethereum's keccak256
    • How to avoid: Use Web3.keccak(bytes) or from Crypto.Hash import keccak

    [fail] LogManager.message Duplication¶

    • Error: LogManager.step() got multiple values for argument 'message'
    • Why it happens: Passing message= as both positional AND keyword argument
    • How to avoid: Only pass the message as the first positional argument

    [fail] Bridge Deployment — Double 0x Prefix In Bytecode¶

    • Error: RPC error for eth_sendTransaction: invalid value: string "0x0x6080..."
    • Why it happens: Foundry's compiled bytecode.object already includes the 0x hex prefix. Prepending another "0x" + bytecode produces "0x0x6080..." which Anvil rejects.
    • How to avoid: Use "data": bytecode without the "0x" + prefix. Foundry's JSON artifact always has 0x built in.
    • If it happens: Check your bridge plugin's deploy() or _deploy_contract() method. The fix is removing the extra "0x" + prefix.

    [fail] Bridge Deployment — anvil_setBalance Fails On Polygon/BSC Archive¶

    • Error: anvil_setBalance failed: historical state ... is not available
    • Why it happens: Some archive nodes (especially Polygon and BSC) don't serve full historical state at older fork blocks. When anvil_setBalance is called on a new (non-existent) address, Anvil tries to look it up in the remote archive and the archive returns "state not available".
    • How to avoid: Wrap per-chain deployment in a try/except so that if one chain's archive can't serve state, it's skipped gracefully with a warning rather than crashing the whole experiment. See bridges/nomad_bridge/nomad_bridge.py for a reference implementation.
    • If it happens: The chain with the problematic archive is skipped. The attack can still run on other chains. Consider using a different RPC provider with better archive depth, or use a more recent fork block.

    [fail] Auto-Merge Kills Chain But Doesn't Restart It¶

    • Error: Connection refused on the chain port after auto-merge updates the fork block
    • Why it happens: The auto-merge feature correctly reads fork_blocks from evidence.yaml and updates the chain config. It terminates the old Anvil process and deletes the stale meta file. But if the chain isn't restarted, the experiment tries to connect to a dead process.
    • How to avoid: This is a framework fix — already handled in crucible/experiment.py. The chain is now automatically re-forked at the updated block after auto-merge. If you see this error, update your Crucible version.
    • If it happens: Run --no-auto-merge to bypass auto-merge entirely, then manually manage fork blocks.

    [fail] Port TIME_WAIT After Killing Old Anvil¶

    • Error: New Anvil fails to bind because the port is in TIME_WAIT state
    • Why it happens: After SIGTERM/SIGKILL, the OS keeps the port in TIME_WAIT for 30-120 seconds. If the auto-merge tries to start a new Anvil immediately on the same port, it fails.
    • How to avoid: Framework fix — Crucible now polls the port every 500ms for up to 10 seconds after killing the old process before starting the new one.
    • If it happens: Increase the polling window or wait 30 seconds before re-forking.

    [fail] Checkpoints Snapshot Dead Chains¶

    • Error: RPC call failed for anvil_dumpState at http://127.0.0.1:8547: Connection refused
    • Why it happens: The checkpoint manager iterates over all .meta.json files in states/. If a stale meta file exists (from a previous topology with different chains), it tries to snapshot a chain that's no longer running.
    • How to avoid: Framework fix — checkpoint now filters to only snapshot chains in the active topology. Always clean stale meta files with rm -f states/*.meta.json before switching topologies.

    [fail] Bridge evidence.yaml Uses Placeholder Contract Address¶

    • Error: Evidence validation failed: Contract "deployed by bridge plugin" has no code on any running chain
    • Why it happens: The evidence pre-flight validation checks that all affected_contracts declared in evidence.yaml have code on the running chains. If you use a placeholder string like "deployed by bridge plugin" instead of a real address, the validation code tries to call get_code() on that string, which fails.
    • How to avoid: For dynamically deployed contracts (bridge plugins), set affected_contracts: [] (empty list) since there are no static contract addresses to verify.

    [fail] L4 Fidelity Without Transaction Traces¶

    • Error: Fidelity violation: No transaction actions were found in run.jsonl
    • Why it happens: L4 fidelity requires transaction traces. If the bridge deployment failed on one chain, there may not be enough trace data. Or the experiment is a demo/simulation, not a real exploit reproduction.
    • How to avoid: Run demo experiments with --no-fidelity to skip fidelity checks. For production experiments, ensure all chains deploy correctly and transactions are captured.

    [fail] set_balance On Precompile Address¶

    • Error: anvil_setBalance: failed to get account for 0x0000000000000000000000000000000000000001
    • Why it happens: Address 0x...0001 is the ECDSA-recover precompile. Anvil has special handling for precompile addresses and may reject set_balance on them, especially on forked chains.
    • How to avoid: Use a random EOA address (e.g., 0xDeaDDeaDDeaDDeaDDeaDDeaDDeaDDeaDDeaDDeaD) instead of a precompile address for the deployer. Or use an address that exists in the forked state (like a real contract address).

    From Research To Code: A Practical Reference¶

    This section is for a researcher who has on-chain data (fork block, contract addresses, tx hashes) and needs to write the Python attack plugin. It explains the mindset shift from Solidity-based exploit replication, maps Crucible APIs to exploit operations, and gives a clear learning path.

    Crucible API Map¶

    This table maps what you want to do → the Crucible API to use.

    Getting Connected¶

    You Want To... Use This Example
    Get the RPC URL for a chain context.chains.runtime_for("ethereum").rpc_url rpc_url = context.chains.runtime_for("ethereum").rpc_url
    Create a JSON-RPC client JsonRpcClient(rpc_url) client = JsonRpcClient(rpc_url)

    Reading Chain State¶

    You Want To... Use This Example
    Check if a contract exists client.get_code(address) code = client.get_code("0x...") → returns "0x" if no code
    Read a storage slot client.get_storage_at(address, slot) val = client.get_storage_at("0x...", "0x1")
    Check an account's balance client.get_balance(address) bal = client.get_balance("0x...")
    Call a contract function (read-only) client.call(method, params) client.call("eth_call", [{"to": addr, "data": "0x..."}, "latest"])
    Get the current block number client.call("eth_blockNumber", []) block = int(client.call("eth_blockNumber", []), 16)

    From Exploit Tx To Python Code: A Worked Example¶

    Here's how to translate an actual exploit transaction from Etherscan into Crucible Python code.

    Step 1: Find the exploit tx on Etherscan. Copy these details:

    Tx: 0x5ea16a...
From: 0xa87fB85... (attacker)
To: 0x838bf9E... (EthCrossChainManager)  ← target contract
Function: verifyHeaderAndExecuteTx       ← the vulnerable function
Input Data: 0x41973cd9...                 ← 4-byte selector + ABI-encoded args
Block: 12996694

    Step 2: Decompose what the tx does:

    The exploit calls verifyHeaderAndExecuteTx() on EthCrossChainManager.
That function internally calls putCurEpochConPubKeyBytes() on EthCrossChainData
via a hash collision (selector 0x41973cd9 matches both functions).
This replaces the bridge keeper key.

    Step 3: Decompose what the transaction does — use Etherscan's "Click to see More" on Input Data to see the decoded function arguments. This tells you the parameter structure.

    Step 4: Translate each action to Crucible code:

    # 1. Connect to the forked chain
client = JsonRpcClient(rpc_url)

# 2. Verify pre-condition: target contract exists
code = client.get_code("0x838bf9E...")  # EthCrossChainManager
if code == "0x":
    return AttackResult(success=False, ...)

# 3. Impersonate the privileged account
client.impersonate_account("0x838bf9E...")  # EthCrossChainManager has onlyManager role

# 4. Build the calldata — DON'T copy the raw hex from Etherscan!
#    The hex embeds specific addresses and values. You need to
#    RECONSTRUCT it programmatically so it works on YOUR fork.
#    Use Etherscan's decoded view to understand the parameter structure,
#    then build it with proper ABI encoding (see poly_network_privkey.py
#    for a complete example of dynamic calldata construction).
calldata = build_put_keeper_calldata(ATTACKER_ADDRESS)

# 5. Send the transaction (same function, on the forked chain)
tx_hash = client.send_transaction({
    "from": "0x838bf9E...",
    "to": "0xcF2afe10...",  # EthCrossChainData
    "data": calldata,
    "gas": hex(300_000),
})

# 6. Log success
context.log.step("keeper key replaced", phase="attack", action="exploit", status="success")

# 7. Return
return AttackResult(success=True, ...)

    Key rule: Never copy-paste calldata hex from Etherscan. The calldata contains addresses, amounts, and values specific to the real exploit's chain state. Your fork has a different chain state, so you must reconstruct the calldata with the correct parameters for your forked environment. The Etherscan decoded view tells you the structure — you implement the structure.

    The pattern is always the same: connect → verify preconditions → impersonate → build calldata → send tx → log → return.

    Reference Implementation Pattern Summary¶

    Pure Python Checklist¶

    from crucible.sdk import AttackPlugin, AttackResult, ExperimentContext, VerificationResult
from crucible.rpc import JsonRpcClient

class MyAttack(AttackPlugin):
    def metadata(self) -> dict:
        return {"name": "my_attack", "version": "1.0.0", "type": "attack", "fidelity_level": "L2"}

    def execute(self, context: ExperimentContext, phase: int | None = None) -> AttackResult:
        # ── 1. Get RPC ──
        rpc_url = context.chains.runtime_for("ethereum").rpc_url
        client = JsonRpcClient(rpc_url)

        # ── 2. Verify preconditions ──
        code = client.get_code("0xContract...")
        if code == "0x":
            return AttackResult(success=False, phase_results=[{"step": "check_code", "passed": False}])

        # ── 3. ABI-encode calldata and send tx ──
        client.impersonate_account("0xAttacker...")
        tx_hash = client.send_transaction({
            "from": "0xAttacker...",
            "to": "0xTarget...",
            "data": "0x...",  # ABI-encoded calldata
        })

        # ── 4. Log ──
        context.log.step("Exploit sent", phase="attack", action="send_tx", status="success")

        # ── 5. Return ──
        return AttackResult(success=True, phase_results=[{"step": "exploit", "passed": True}])

    def verify(self, context: ExperimentContext) -> VerificationResult:
        # ── 6. Verify ──
        client = JsonRpcClient(context.chains.runtime_for("ethereum").rpc_url)
        storage = client.get_storage_at("0xTarget...", "0x1")
        passed = storage != "0x0"
        return VerificationResult(
            passed=passed,
            assertions=[{"name": "state_changed", "passed": passed, "actual": storage}],
            errors=[] if passed else ["Storage slot did not change"]
        )

    L3/L4 Reproduction Workflow¶

    Fidelity Enforcement¶

    Fidelity enforcement is active by default — it validates runtime behavior against the fidelity level declared in evidence.yaml. Use --no-fidelity to skip:

    crucible experiment run experiments/nomad_replay.yaml
    Level What's Enforced Required In Evidence
    L3 All L2 + exact_tx_replay=true, exploit_transactions non-empty, tx actions found in artifacts exploit_transactions: ["0x..."], exact_tx_replay: true
    L4 All L3 + multi-chain topology + bridge plugin + excluded_components declared Same as L3 + L4 topology with bridge plugin

    L3: Exact Transaction Replay¶

    L3 replays the exact historical exploit transaction on a forked chain. This requires: 1. Setting exact_tx_replay: true in evidence.yaml 2. Reconstructing the exact calldata from the original transaction (use crucible attack extract) 3. Verifying gas, events, and storage match the original

    # evidence.yaml
reproduction_scope:
  fidelity_level: L3
  exact_tx_replay: true
historical_scope:
  exploit_transactions:
    - "0xebf283515720d2d3a33f431f9d659ccf919163273e5f299166f2878d655f4625"

    # Run with fidelity enforcement (enabled by default)
crucible experiment run experiments/my_l3_replay.yaml

    L4: Multi-Chain With Bridge Plugin¶

    L4 requires multiple chains + a bridge plugin in the topology. Use one of the 3 bridge topology files:

    Topology Plugin Chains

    topologies/poly_network_bridge.yaml poly_network_bridge ethereum + polygon + bsc
    topologies/wormhole_bridge.yaml wormhole_bridge ethereum + polygon 
    # experiments/my_l4_experiment.yaml
name: my_l4_experiment
