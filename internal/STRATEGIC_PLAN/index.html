    Crucible Strategic Plan — Philosophy, Review Findings & Improvement Roadmap¶

    Date: June 7, 2026
    Purpose: A single document that captures the project's core philosophy, summarizes findings from an expert architecture review, and lays out our prioritized plan to close gaps and improve the researcher experience.
    Audience: The team, collaborators, and anyone who wants to understand where Crucible is going and why.

    1. Core Philosophy — The "Attacker's Seat" Principle¶

    This document does not describe a pivot. Crucible was designed from day one around this model: the testbed provides Chain + Bridge + Artifact Collector, and the researcher sits in the attacker's seat — forking a real chain, running attacks like they normally would, and getting complete artifacts automatically. The expert review confirmed the architecture is sound; the work ahead is about closing correctness gaps and reducing friction, not re-architecting.

    2. Architecture¶

    Crucible's architecture has three layers:

    flowchart TD
    subgraph Crucible["CRUCIBLE"]
        A["Chain Manager<br>forks mainnet"]
        B["Bridge Manager<br>connects chains"]
        C["Artifact Collector<br>(RPC proxy +<br>structured logs)"]

        D["Attack Plugin<br>(researcher writes this)<br>'sits in attacker's seat'"]

        A --- D
        B --- D
        C --- D
    end

    The testbed does: chain forking, bridge deployment, RPC proxying, transaction tracing, artifact collection, integrity hashing, publication bundling.

    The researcher does: write an attack plugin (or use an existing one), run it, read the artifacts. That's it.

    2.1 Two-Phase Workflow¶

    Phase What the researcher does What Crucible does How often
    Setup Configure testbed.yaml + experiment.yaml + run crucible baseline Sets up the environment — starts Anvil forks, deploys bridges, prepares baselines One-time per attack/project
    Attack Run crucible experiment run experiments/my_attack.yaml, then write attack plugin code Runs the attack, auto-captures every transaction's gas/events/storage/balance, saves structured artifacts Repeatedly

    The setup phase is explicit by design — it serves as documentation of exactly what was forked, when, and on which chains. These YAML files are checked into git, reviewed, and versioned. They're the researcher's lab bench: set it up once, document it, then run experiments on it. The automation is reserved for what happens during the attack: the RPC proxy auto-captures every transaction, the logging system auto-organizes artifacts, and the integrity hashing auto-verifies them.

    2.2 What Makes a Good Attack Plugin¶

    An attack plugin is just a Python class with a run() method that gets a live chain, deployed bridge, and auto-instrumenting RPC proxy:

    class NomadReplicaFraud(AttackPlugin):
    def run(self, context):
        client = JsonRpcClient(context.chains["ethereum"].rpc_url)
        # The researcher does what they'd do on a real chain
        # Every transaction is auto-traced by the RPC proxy
        client.send_transaction(to=replica, data=forged_call)

    No logging boilerplate. No artifact management. No chain lifecycle. The testbed handles all of that. This is the automation that matters — not the setup, but the attack execution.

    2.3 Fidelity Boundary — What Crucible Guarantees (and What It Doesn't)¶

    Every research instrument needs an explicit threat model. A reviewer will ask: "What can go wrong between your fork and the real chain, and how do you know your reproduction is faithful?"

    What Crucible guarantees (EVM-level fidelity):

    Property Guarantee Mechanism
    State fidelity Fork starts with identical storage, balance, and code as the real chain at block N Anvil fetches from archive node
    Execution fidelity EVM opcodes execute identically Anvil uses the same EVM as Ethereum clients
    Gas accounting Gas is metered with identical opcode costs Same EVM implementation
    Event emission Events are emitted from the same code paths Deterministic execution
    Artifact integrity Artifacts are hashed and provenance-tracked SHA-256 + run_manifest.json

    What Crucible does NOT guarantee (protocol-level fidelity):

    Property Limitation Impact
    Mempool dynamics Anvil auto-mines with FIFO ordering Attacks depending on MEV, front-running, or block inclusion order are not reproducible
    Block timing timestamp: fork_block freezes time Attacks depending on block.timestamp changes across blocks are not reproducible
    Cross-chain relay timing Bridge plugins are synchronous mocks Real bridge exploits may depend on relay latency, guardian quorum timing, or message ordering
    Consensus behavior Anvil is a single-node devnet Reorgs, uncle blocks, and finality delays are not reproducible
    External oracle state Oracle values reflect the fork block Oracles that were manipulated during the exploit window may show different values

    This is not a weakness — it's a scope boundary. Crucible reproduces the EVM execution of bridge exploits. It does not reproduce mempool ordering, cross-chain relay timing, consensus behavior, or external oracle dynamics. The fidelity level (L0–L4) describes how much of the original attack's EVM behavior is replayed, not the completeness of the end-to-end attack path.

    Publication bundles should state this boundary explicitly. A USENIX reviewer will respect scoped claims. They will not respect unqualified claims of "faithful reproduction."

    2.4 Why Crucible, Not Just Foundry¶

    Foundry (Anvil + Forge) already does most of what's needed to reproduce an exploit:

    // DeFiHackLabs pattern — works, has been cited in papers
contract NomadExploit is Test {
    function testExploit() public {
        vm.createSelectFork("mainnet", 15259100);
        replica.process(fraudulent_message);
        assertGt(token.balanceOf(attacker), 0);
    }
}
// Run: forge test --match-test testExploit -vvvv

    What Foundry already handles: forking at any block, running exploit transactions, showing traces and events (-vvvv), gas reports, snapshot/restore, deterministic execution.

    What Crucible adds — the three things Foundry can't do:

    Capability Why Foundry can't How Crucible does it
    Structured, machine-readable artifact capture forge test -vvvv outputs unstructured text to stdout. You can't hash it, diff it, or programmatically verify it. RPC proxy intercepts all transactions and emits typed JSONL streams (events, storage diffs, gas, balances) with SHA-256 integrity hashes.
    Multi-chain coordination Forge can only fork one chain per test. Bridge exploits that span Ethereum + BSC need two simultaneous forks. Topology system manages N Anvil instances, each with its own instrumentation proxy, coordinated by a single experiment runner.
    AI agent in the attack loop Forge is a Solidity-native tool. No LLM integration. AgentPlugin + LLMAgentPlugin + 9 privileged tools allow an LLM to explore and exploit a forked chain autonomously.

    Everything else — config management, plugin discovery, scenario YAML, baseline management, 17 CLI subcommands — is orchestration. It exists because Foundry has no experiment management layer, and academic reproducibility needs one (fork block documentation, evidence metadata, provenance tracking, export bundles). But the orchestration is not the core value. The core value is: transparent instrumentation that produces publication-grade artifacts without the researcher writing logging code.

    If the orchestration ever becomes a barrier to the researcher experience, it should be made optional (convention over configuration), not removed. The simple path must stay simple: run an attack, get artifacts.

    3. Expert Review Summary¶

    An independent architecture review was conducted (June 2026) covering ~21k LOC Python, 8 attack packs, the Streamlit dashboard, the LLM agent system, and 29 documentation files.

    3.1 Overall Verdict¶

    7.5–8 / 10. Excellent bones; a handful of correctness/credibility items stand between you and "press the button, trust the output."

    The review confirmed the architecture is solid — the plugin system, topology abstraction, artifact pipeline, and provenance machinery are all genuinely well-designed. The gaps are about closing trust and verification loops, not adding features.

    3.2 What We Agree With (no debate)¶

    Finding Our Assessment
    Soft-reset state leakage — anvil_loadState merges instead of wiping. After a soft reset, newly-created accounts from the previous run persist. Real correctness gap. Verified in code. Fix: change default to medium (kill+relaunch).
    Silent degradation in artifact path — except Exception: logger.debug(...) then continue means incomplete artifacts can look like a successful run. Verified in code. Fix: add artifacts_complete flag to manifest.
    Evidence.yaml has no run-time validation — fork block and contract addresses are declared but never checked against the live fork. Verified in code. Fix: pre-flight bytecode + block height check.
    No CI-runnable E2E attack test — the one reproduction test requires an archive node, so CI proves nothing about whether attacks still reproduce. Verified in code. Fix: RPC recording/replay fixture system.
    LLM agent has no dollar-cost budget — token caps exist but nothing limits actual API spend. Verified in code. Fix: add llm_budget parameter.
    29 docs with no golden path — comprehensive but unguided. No single "run these 5 commands" tutorial. Confirmed. Fix: 10-minute TUTORIAL.md.
    Checkpoint save/restore lacks integrity validation — missing/corrupt snapshot silently skipped. Reported, consistent with code patterns. Fix: hash+verify.
    Port/process lifecycle not verified — TOCTOU race, failed kills hidden, partial cleanup. Reported. Fix: retry+verify pattern.

    3.3 What We Nuanced or Disagreed With¶

    Finding Our Position
    Hermetic E2E with checked-in state fixture Agree it's needed, disagree with "check in 200MB state dump." Better approach: RPC response recording/replay at the framework level (see §4.2).
    Prompt injection from experiment YAML Low risk — experiment files are committed and reviewed, not uploaded by untrusted users. Not worth prioritizing.
    Cross-chain correlation IDs should be deterministic Design trade-off, not a bug. Counter-based IDs are more debuggable during development. Current approach is fine.
    Schema validation at log-write time Too heavy on the hot path. Better: validate at bundle time (export/publication) with zero runtime cost.

    3.4 Best Ideas from the Review¶

    Idea Priority Why
    crucible verify <bundle> — re-run an archived bundle and hash-compare artifacts P1 Closes the reproducibility loop. This turns "we logged it" into "anyone can re-derive it."
    Per-attack fidelity table in BENCHMARKS.md P3 Cheap, high credibility. Shows why each attack can't go higher than its current fidelity level.
    --from-tx attack scaffolder P3 Pulls tx data from a real exploit and pre-fills plugin.yaml + evidence.yaml. Dramatically lowers authoring cost.
    Determinism harness — crucible doctor --determinism that runs same experiment twice and diffs artifact hashes P3 Makes non-determinism visible instead of suspected.

    4. The Researcher Experience — Current vs. Target¶

    Researcher Workflow¶

    # 1. Setup (one-time per attack/project)
git clone <repo>
cp .env.example .env                             # fill in RPC URL
pip install -e .
crucible doctor                                  # verify everything is ready

# Write testbed.yaml (chains, fork blocks, logging)
# Write experiments/my_attack.yaml (name, scenario, attack, topology)
# Write scenarios/my_scenario/scenario.yaml (phase definitions, if needed)

crucible baseline                                # capture initial state

# 2. Attack (the real work, repeated)
crucible experiment run experiments/my_attack.yaml
# ... researcher writes attack plugin code ...
# Artifacts auto-saved to logs/my_attack/DS-2026-00042/

# 3. Inspect results
cat logs/my_attack/DS-2026-00042/report.md       # human-readable summary
ls logs/my_attack/DS-2026-00042/chain_ethereum/  # raw artifact streams

    The setup phase is not toil — it's documentation. Every YAML file is a git-tracked, reviewable artifact that captures exactly what was done. The automation is reserved for what happens during the attack: the RPC proxy, artifact collection, and integrity hashing. This separation of concerns is intentional and is one of Crucible's strengths.

    5. Mitigation Plan — What We Need to Fix¶

    P0 — Trust & Correctness (before any external release)¶

    These ensure the environment is honest and artifacts are trustworthy. Without these, a reviewer or careful co-author can say "how do you know this is faithful?" and Crucible can't give a crisp answer.

    # Item What Why Effort
    1 artifacts_complete flag After a run, scan run.jsonl for auto-trace failures and compare Anvil's actual transaction count against the number of traced transactions (to catch any traffic that bypassed the proxy). Include complete: true/false + trace_failures + tx_count_mismatch in run_manifest.json. Add a publication gate that rejects incomplete runs. Silent degradation: a run can produce incomplete artifacts and still report "success." The RPC proxy only captures transactions that flow through it — a plugin that bypasses the proxy produces invisible gaps. 2 hrs
    2 Default reset → medium Change reset_cmd.py default from soft to medium (kill + relaunch Anvil). Document soft mode for explicitly-stateless quick iterations only. anvil_loadState merges, doesn't wipe. Soft reset leaves state from previous runs behind, breaking determinism guarantees. 10 min
    3 Evidence pre-flight validation At experiment start, call eth_getCode(affected_contract) and eth_getBlockByNumber(fork_block) against the live fork. Fail fast if bytecode is empty or block doesn't match. Converts "the researcher claims it forked the right state" into "the testbed proves it." This is the highest-leverage rigor upgrade. 4 hrs
    4 Hermetic CI test via RPC recording Build a framework-level RPC recording/replay system. The existing RpcProxy already intercepts all traffic — add record mode (serialize request/response pairs) and replay mode (mock RPC server from fixture). Any attack can be recorded once and replayed in CI without an archive node. Without this, CI proves nothing about whether attacks still reproduce. The one real reproduction test is always skipped. 2-3 days

    P1 — Researcher Experience (the "attacker's seat" feeling)¶

    These make working with the testbed feel seamless — the setup is explicit, but the attack loop is frictionless.

    # Item What Why Effort
    5 Optional run alias Add --name flag to experiment runs so researchers can label them (e.g. --name "nomad-v2-repro"). Display in history, dashboard, and manifest. Run IDs are opaque (DS-2026-00001). A human-readable label makes finding runs later trivial. No infra change, just a new field. 2 hrs
    6 crucible verify <bundle> Re-run an archived bundle from scratch and hash-compare all artifacts against the original. Report pass/fail per stream. Closes the reproducibility loop. This is what turns Crucible from "a tool that logs things" into "a tool that proves reproducibility." The core diffing infrastructure already exists in diff.py. 1-2 days
    7 10-minute TUTORIAL.md "Reproduce the Nomad hack in 10 minutes" — 8 copy/paste commands with expected output, artifact locations, and a "what just happened" section. 29 docs with no golden path. A single tutorial is the highest-ROI docs change possible. 3 hrs

    P2 — Robustness (catches edge cases, prevents waste)¶

    # Item What Why Effort
    8 LLM spend cap llm_budget parameter (USD or tokens) on agent experiments. Hard-stop the agent loop when budget exceeded. Record in manifest. Running parameter sweeps with frontier models can burn $50+ in a single experiment with no warning. 2 hrs
    9 Checkpoint integrity Validate JSON after write, SHA-256 hash it, verify hash on restore. Fail fast on corrupt/missing checkpoint instead of silently continuing. Phase checkpoints are the safety net for long-running experiments. A corrupt checkpoint currently causes silent non-determinism. 4 hrs
    10 Port/process lifecycle Verify ports are free after stop_all() before declaring cleanup done. Add retry/backoff around chain start. Log failed kills at WARNING (not swallowed). Track per-target success in clean --all. Orphan Anvil processes holding ports are the most common source of cryptic "port in use" errors. 4 hrs
    11 Consolidate redundant docs 4 roadmap-ish docs (IMPLEMENTATION_PLAN, IMPLEMENTATION_STATUS, ROADMAP_TO_100, FUTURE_ARCHITECTURE_PLAN) and 2 handoff docs create confusion. Consolidate into docs/internal/ and keep only user-facing docs in the root docs directory. 29 docs works against the "low learning curve" goal. Clean separation of user docs vs internal planning docs makes navigation easier. 2 hrs

    P3 — Polish & Strategic (nice to have, high credibility when done)¶

    # Item What Why Effort
    12 Per-attack fidelity table in BENCHMARKS.md Show fidelity level (L0–L4) per attack with the explicit reason it can't go higher. Pre-empts the "is this real?" question from reviewers. The current honesty is a credibility asset — make it visible. 30 min
    13 Run tiers (scratch / saved / published) Auto-clean scratch runs after N days. Preserve saved/published runs indefinitely. crucible publish <run_id> marks a run as immutable and optionally backs it up to a configurable remote. Currently all runs are equal. A publication run and a scratch test run get the same retention policy. Tiering solves this cleanly. 1 day
    14 Bundle-time JSONL schema validation At export time, validate every JSONL line has the expected fields for its stream type. Catch field-type drift across code versions. Same coverage as per-write validation, zero runtime cost. 4 hrs
    15 --from-tx attack scaffolder crucible new-attack --from-tx 0x... --chain eth — pull the tx, fork block, touched contracts, and pre-fill plugin.yaml / evidence.yaml. Dramatically lowers the cost of authoring a new attack from a blog post or DeFiHackLabs link. On-mission. 2-3 days

    7. Key Design Decisions¶

    Why RPC Recording/Replay Instead of Checked-In State Fixtures¶

    Approach Fixture Size CI Speed Fragility
    Anvil state dump (checked-in) 50–200 MB per attack Fast (no fork) High — Foundry version changes can break loading
    RPC response recording (checked-in) 10–200 KB per attack Fast (no network) Low — just JSON request/response pairs
    Live archive node (current) None Slow (network I/O) Medium — depends on RPC provider availability

    RPC recording is the clear winner: small fixtures, fast execution, hermetic, low maintenance. The infrastructure already exists (the RpcProxy intercepts all traffic). We just need to add serialize/deserialize modes.

    Why the YAML Configs Stay¶

    The experiment YAML, topology configs, and scenario definitions are not going away. They serve real needs:

    • Multi-chain topologies — configuring two chains and a bridge is complex. A YAML file documents it reproducibly.
    • Parameter sweeps — experiments/benchmark_matrix.yaml runs the same attack at 10 different fork blocks. This requires structured config.
    • Publication mode — asserting evidence completeness, reproducibility gates, and provenance checks need explicit configuration.
    • Team collaboration — YAML files are checked into git, reviewed, and versioned. CLI flags are ephemeral.

    The fix is: make them optional for the simple case, not required for all cases.

    Why the Report Layer Exists¶

    report.md is a post-processing aggregation of the raw JSONL artifact streams. It exists because:

    • Human readability — open any markdown viewer and see summary counts, phase breakdowns, gas profiles, storage changes, and event frequencies at a glance.
    • Shareability — attach to publication bundles, include in PRs, send to non-technical stakeholders.
    • Quick inspection — "Did the attack run? How many events did it emit? Any failures?" answered in 3 seconds.

    It's not the source of truth (the JSONL streams are). It's a convenience layer on top.

    8. Open Questions for the Team¶

    1. Run retention policy — What's the default? Keep all runs indefinitely? Auto-clean after 30 days? Tiered by importance?
    2. Remote artifact storage — Do we want crucible push to S3/IPFS/archive? In scope for Phase 2 or Phase 3?
    3. Dashboard — The current dashboard is a view layer over the JSONL/logs. Does it need deeper integration (e.g., running experiments from the UI), or is "read-only view" the right scope?
    4. Community/OSS — Do we want to accept external attack plugins? If so, what's the review process? (Impacts P0 items like evidence validation — external plugins need stricter gates.)

    9. Summary: Where We Excel, Where We Lack, What We'll Do¶

    Where We Excel¶

    Area Why
    Architecture Plugin system, topology abstraction, and scenario engine are genuinely composable. The separation of chains/bridges/artifacts/attacks is clean and correct.
    Artifact pipeline 6 typed JSONL streams, deterministic serialization, SHA-256 integrity manifests, provenance tracking. Above the bar for most academic artifact submissions.
    Attack fidelity honesty Each attack's fidelity level is documented accurately. No over-claiming. The testbed is honest about what it can and can't reproduce (EVM-only, Solana sysvar root cause, etc.). This is a credibility asset.
    Cross-platform Windows, macOS, and Linux all supported. shutil.which(), CREATE_NEW_PROCESS_GROUP on nt, msvcrt vs fcntl file locking — this is genuinely cross-platform.
    Testing 521 unit tests, integration tests for CLI/experiment/baseline/publication, dashboard e2e tests. Good coverage for a research tool.
    Docker deployment Fully containerized. docker compose up runs everything. docs/DEPLOYMENT.md covers SSH clone → Docker deployment.

    Where We Lack¶

    Area The Gap
    Trust verification The testbed claims forks are at the right block — but never proves it at run time. Evidence.yaml declares contracts and blocks but never checks them against the live fork. Fixing this is the single highest-leverage rigor upgrade (item 3).
    Silent degradation A run can lose trace data (receipt not found, trace failed) and still report outcome: "success". The researcher has no machine-readable way to know artifacts are incomplete (item 1).
    Reset correctness Soft reset (the default) uses anvil_loadState which merges, not wipes. State from previous runs can leak into the next run. This breaks determinism guarantees for parameter sweeps and A/B comparisons (item 2).
    CI attack testing The one real reproduction test requires an archive node and is always skipped in CI. CI proves nothing about whether attacks still reproduce after refactors (item 4).
    Fidelity definitions Fidelity levels (L0–L4) describe what metadata is present in evidence.yaml, not what behavior was demonstrated. An L3 attack that lists exploit_transactions but doesn't replay them is still classified L3. No runtime check that declared fidelity matches actual execution (item 16).
    Bridge simulation model BridgePlugin.send_message() is a researcher-written mock, not a replayed protocol. At L4 fidelity, a reviewer will ask "how do you know the mock is faithful?" — and there's no built-in answer. Bridge plugins need protocol specification references and behavioral assertions (item 17).
    LLM spend safety Agent experiments have iteration caps but no dollar-cost budget. A parameter sweep with frontier models can burn significant API spend with no warning (item 8).
    Process lifecycle Port race conditions, swallowed kill errors, and partial cleanup failures are the most common source of cryptic errors (item 10).
    Onboarding 29 docs with no golden path. No single "run these commands" tutorial that takes a new user from zero to reproduced attack (item 7).
