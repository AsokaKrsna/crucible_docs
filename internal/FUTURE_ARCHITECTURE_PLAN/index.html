    Crucible Future Architecture Plan¶

    Toward a Modular, Researcher-First Blockchain Security Testbed¶

    Status: Planning | Date: June 2026 | Audience: Research team & contributors

    1. Vision¶

    Crucible should feel like this:

    The researcher sits in the attacker's seat, writes or runs exploit logic, and Crucible automatically forks the right state, instruments every action, records publication-grade artifacts, and leaves the researcher with a complete dataset ready for analysis.

    The system must remain: - EVM-only — no scope creep into non-EVM chains - Modular and plug-and-play — new capabilities arrive as plugins, not core rewrites - Easy to extend — researchers should not need to be framework engineers - USENIX-grade — strong enough for artifact evaluation and reproducibility requirements

    2. Product Principles¶

    2.1 Researcher-First UX¶

    The framework should not make researchers think about internal plumbing.

    Target workflow: 

    1. crucible init
2. crucible doctor
3. crucible attack init my_attack --template historical-single
4. Write exploit logic in attack.py or Exploit.s.sol
5. crucible experiment run experiments/my_attack_replay.yaml
6. crucible export <run_id>

    2.2 Instrumentation by Default¶

    All formal experiment runs must be instrumented automatically. The researcher should not need to opt into logging or tracing.

    2.3 Plugins for Capabilities¶

    New capabilities arrive as plugins — no core rewrites: - attack — exploit reproduction plugins (current) - bridge — bridge protocol simulation (current) - agent — LLM-driven autonomous agents (new) - analyzer — post-run analysis extensions (future)

    2.4 Strong Defaults, Optional Overrides¶

    If a researcher does nothing special, Crucible still: - Collects all artifacts (gas, events, storage diffs, balance diffs) - Produces a human-readable report - Hashes outputs for integrity verification - Preserves full provenance (git, versions, fingerprints) - Supports reruns and canonical comparison

    3. Architecture Direction¶

    3.1 New Plugin Type: AgentPlugin¶

    Purpose: Allow an LLM-based or scripted autonomous agent to operate the testbed as an attacker, research assistant, or fuzz-style explorer.

    New SDK interface (crucible/sdk.py):

    class AgentPlugin(CruciblePlugin):
    """Interface for AI/autonomous agent modules."""

    def metadata(self) -> dict[str, Any]: ...

    def setup(self, context: ExperimentContext, tools: AgentToolbox) -> None:
        """Called once before run(). Agent can inspect environment."""

    def run(self, context: ExperimentContext, tools: AgentToolbox) -> AgentRunResult:
        """Main agent loop — observe, reason, act, repeat."""

    def verify(self, context: ExperimentContext) -> VerificationResult:
        """Post-run assertions."""

    def cleanup(self, context: ExperimentContext) -> None:
        """Teardown."""

    New result type (crucible/sdk.py):

    @dataclass
class AgentRunResult:
    success: bool
    actions: list[dict[str, Any]] = field(default_factory=list)
    observations: list[dict[str, Any]] = field(default_factory=list)
    error: str | None = None

    3.2 Agent Tool Layer¶

    New file: crucible/agent_tools.py

    Exposes safe, structured tools to AI agents:

    Tool Description
    list_chains() Available chains in current topology
    get_balance(chain, account) ETH or token balance
    get_storage(chain, address, slot) Storage slot value
    get_code(chain, address) Contract bytecode
    call(chain, to, data) eth_call (read-only)
    impersonate(chain, account) anvil_impersonateAccount
    send_tx(chain, tx) eth_sendTransaction
    advance_time(chain, seconds) evm_increaseTime + evm_mine
    assert_balance(...) Check balance condition

    Design rule: Every tool call must: 1. Execute the underlying action 2. Auto-emit a semantic log entry 3. Attach structured metadata for analysis

    3.3 Agent Execution in Experiment Runner¶

    Files to change: crucible/experiment.py, crucible/config.py, crucible/plugin.py

    The experiment runner gains an optional agent path alongside attack:

    experiment.run()
  → load scenario
  → start instrumentation proxies (always)
  → if experiment.attack: load AttackPlugin → attack.run(context)
  → if experiment.agent:  load AgentPlugin → agent.run(context, tools)
  → drain instrumentation
  → write manifest + report + integrity
  → always

    ExperimentConfig gains optional agent field:

    class ExperimentConfig(BaseModel):
    name: str
    scenario: str
    attack: str | None = None
    agent: str | None = None      # NEW
    topology: str = "single"
    ...

    3.4 Semantic Auto-Logging¶

    New file: crucible/action_log.py

    Introduce an ActionRecorder that wraps all high-level actions with automatic semantic logging:

    Action Auto-logged fields
    impersonate chain, account, timestamp
    send_tx chain, from, to, data_preview, value, tx_hash, role
    advance_time chain, seconds, new_timestamp
    assert_balance chain, account, min_value, actual_value
    forge_script script, chain, sender, exit_code

    Files to change: crucible/scenario_steps.py, crucible/experiment.py

    Every registered step handler auto-emits step_declared → step_started → step_completed / step_failed. This removes the need for manual context.log.step() in most scenarios.

    3.5 Instrumentation Config Block¶

    File to change: crucible/config.py

    Add first-class instrumentation settings:

    class InstrumentationConfig(BaseModel):
    enabled: bool = True
    auto_start: bool = True
    collect_events: bool = True
    collect_storage_diffs: bool = True
    collect_balance_diffs: bool = True
    collect_gas_profile: bool = True
    trace_timeout_seconds: int = 10
    max_workers: int = 10

    3.6 Attack Scaffolding — crucible attack init¶

    New file: crucible/attack_templates.py

    Command:

    crucible attack init my_attack --template <type>

    Template types:

    Template Best for Generates
    historical-single One-chain exploit (DeFiHackLabs style) plugin.yaml, evidence.yaml, attack.py, scenario.yaml, experiment.yaml
    historical-bridge Dual-chain bridge exploit Same + bridge config
    tx-replay Extract from tx hash Uses existing attack extract + scaffold
    foundry-poc Solidity-first workflow Exploit.s.sol + forge config
    ai-agent LLM-driven agent agent plugin.yaml + agent.py

    Output: 

    attacks/<name>/
  plugin.yaml
  evidence.yaml
  attack.py
scenarios/<name>_pre_exploit/
  scenario.yaml
experiments/<name>_replay.yaml

    3.7 Declarative Assertions¶

    New file: crucible/assertions.py

    Move standard postconditions out of Python where possible:

    # attacks/<name>/assertions.yaml
assertions:
  - type: balance_increased
    chain: ethereum
    account: "0xAttacker"
    token: "0xWETH"
    min_delta: 1000000000000000000
  - type: slot_changed
    chain: ethereum
    contract: "0xBridge"
    slot: "0x01"
    expected_after: "0x01"

    This makes packs more declarative and easier for non-framework engineers.

    6. Success Criteria¶

    Crucible meets the goal when all of the following are true:

    1. [ok] A researcher can create a new attack pack with one scaffold command
    2. [ok] A researcher can write only exploit logic and still get a full artifact bundle
    3. [ok] All experiment runs are instrumented by default
    4. [ok] An AI agent can operate through a first-class plugin type
    5. [ok] Logs and artifacts are coherent enough for notebook analysis and paper figures
    6. [ok] Docs are reduced to one canonical source per topic
    7. [ok] Adding a new plugin type or implementation does not require invasive core edits
    8. [ok] Remaining benchmark packs are upgraded from scaffold to evidence-backed reproductions

    7. Non-Goals¶

    Crucible is not trying to be: - A mempool simulator - A non-EVM research platform (Solana, Cosmos, etc.) - A validator-node orchestration framework - A zk-proof execution environment (v1)

    This focus is a strength, not a limitation.
