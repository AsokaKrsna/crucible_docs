    Crucible — Closing Plan¶

    Mission: Take the testbed from current state (~80%) to project-complete. Current: 662 tests pass, 27 lint warnings (all E402), 6/6 attacks validated, platform layer solid. Target: Docker-validated, CI-hermetic, zero known correctness gaps.

    How This Plan Works¶

    This plan is organized into 4 tracks that can run in parallel. Each track has clear deliverables and exit criteria. Tracks are ordered by impact — do Track 1 first, then choose the highest-value remaining track.

    2.3 Polygon Bridge Topology Timeout [ok]¶

    Current state: [ok] Fork timeout increased from 180s to 300s (5 min) with periodic progress messages every 30s. --fork-timeout CLI flag added. Completed.

    Root cause: Polygon at fork block 16,332,982 has significant historical state. Anvil's cold-cache fork requires downloading this state from the archive RPC, which can take 3-5+ minutes with free archive RPCs (Ankr, LlamaNodes). The previous 180s timeout was insufficient for cold-cache Polygon forks. Docker Anvil healthchecks also had a 120s effective window (start_period 45s + retries 15 × interval 5s = 120s) that was too tight.

    What was implemented: - Increased default timeout from 180s to 300s (5 min) in ChainManager._await_ready() - Added fork_timeout constructor parameter to ChainManager — cleanly configurable, not a hacky attribute set - Added --fork-timeout CLI flag to crucible fork — users can set --fork-timeout 600 for slow RPCs - Progress messages every 30s — print() with flush=True so the user sees live updates on the CLI (not just log files) - Actionable error messages — on timeout, points to logs/anvil/<name>.log, suggests faster RPC providers, and recommends --fork-timeout increase - Fixed busy-loop bug — time.sleep(0.5) moved outside the except block so it applies on every loop iteration

    Task Effort Status
    Investigate root cause — is it slow RPC, config issue, or Anvil bug? 2h [ok] (cold-cache Polygon fork + Docker healthcheck window)
    Increase timeout and add progress messages 1h [ok] (300s default, print() every 30s)
    Add --fork-timeout CLI parameter 1h [ok]
    Exit criteria: crucible fork --topology bridge succeeds within 180s for warm-cache, 300s+ for cold-cache [ok] (default 300s, configurable via --fork-timeout)

    How To Use This Plan¶

    1. Start with Track 1.2 (Horizon) — it has no blockers and is quick (1-2 days). This gives momentum.
    2. In parallel, unblock Track 1.1 (Poly Network) — secure BSC/Polygon archive RPC access. This is the critical path.
    3. Tackle Track 2.2 (Docker) in the background — it's mostly waiting for builds.
    4. Fill in Tracks 3 and 4 as time allows between attack runs.
    5. Final step: Run full test suite, verify all 6 attacks, run crucible doctor, commit.

    This document is a living plan. Cross items off as they're completed and update the date.
