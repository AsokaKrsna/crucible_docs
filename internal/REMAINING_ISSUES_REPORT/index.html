    Crucible — Thorough Re-Check After Commit 74d0b56¶

    Date: 2026-06-13 Reviewed: HEAD = 74d0b56, covering the full fix series 2ad6702 → e0e1a39 → b0916fc → 74d0b56. Method: Static review. Two parallel deep passes — (A) auditing the security-fix commits for regressions/bypasses, (B) re-checking the deferred Medium/Low findings — plus direct verification of the headline items. No code executed.

    V-HIGH-11 is fixed (verified in both docs/ and website/), so every item from the prior three reports is now closed.

    However, this deeper pass found that two of the "fixed" CRITICALs don't fully achieve their stated security goal, plus one new regression introduced by the C3 fix. Details below. These are new findings, not previously reported.

    Part 2 — NEW findings from the deep pass¶

    🔴 N1 (NEW REGRESSION) — Batch JSON-RPC requests crash the proxy¶

    File: crucible/instrumentation.py:296-301 

    try:
    req = json.loads(body)
    method = req.get("method", "")      # body may be a JSON array → list has no .get
except (json.JSONDecodeError, ValueError):   # AttributeError is NOT caught
    self._send_error(400, "invalid JSON-RPC request")
    return
    A batch JSON-RPC request is a top-level JSON array (valid per spec; emitted by web3.py/ethers when batching). json.loads returns a list; list.get raises AttributeError, which is not in the caught tuple → the exception escapes do_POST. BaseHTTPRequestHandler doesn't handle handler exceptions, so the connection is dropped with a server-side traceback and no response. - This is a regression introduced by the C3 fix. The pre-fix code only parsed the method opportunistically inside a broad except Exception, so batch requests forwarded fine. - Impact: any client sending a batch (or a bare []) breaks; remote-triggerable crash/DoS of the instrumentation proxy. - Fix: detect isinstance(req, list) and either reject with a clean 400 or validate each element's method before forwarding.

    🟠 N2 — C3 allowlist permits the exact privileged methods its own comment says it blocks¶

    File: crucible/instrumentation.py _ALLOWED_METHODS The header comment claims privileged anvil_* methods are blocked to stop "a compromised attack plugin … calling anvil_setBalance, anvil_impersonateAccount." But the set explicitly allowlists anvil_setBalance, anvil_setStorageAt, anvil_setCode, anvil_setNonce, anvil_impersonateAccount, anvil_setChainId, evm_revert, evm_setNextBlockTimestamp, etc. - Net: C3 genuinely closes the SSRF + DoS holes (open relay, unbounded body) — that's real and good. But it does not enforce least-privilege; a plugin reaching the proxy can still mint balance, impersonate, overwrite code, or roll back instrumentation. - These cheatcodes are needed by the reset/impersonation lifecycle, so this may be acceptable — but then the comment is misleading and should be corrected, and ideally the lifecycle's privileged calls should use a separate internal path not exposed to plugins. - Severity: MEDIUM (by-design trade-off, but the stated guarantee isn't met).

    🟠 N3 — C4 external hash is write-only; no verifier consumes it¶

    File: crucible/export.py:217-230 (writer) vs verify_bundle:278+, verify_run, commands/bundle.py _write_external_hash() writes datasets/<run_id>.sha256, but no code path ever reads it. verify_bundle still recomputes file hashes and compares them against the in-bundle artifact_bundle_manifest.json (self-attesting). Grep confirms .sha256 appears only in the writer. - Net: the anchor exists on disk but the integrity check still trusts the bundle's own manifest, so an attacker who edits an artifact + recomputes the manifest still passes verify. The tamper-evidence gap C4 targeted is not actually closed in the verify path. - Also, the .sha256 lands in the same datasets/ dir next to the bundle (co-located), so it's not a trust boundary unless distributed out-of-band. - Fix: have verify_bundle read <run_id>.sha256, recompute the manifest's hash, and compare before trusting the manifest's artifacts map. For publication-grade, sign it or anchor via Zenodo DOI. - Severity: MEDIUM if "tamper-evident" is a hard requirement; LOW if "corruption-detectable + optional external anchor" suffices.

    🟢 N4 — validate_name regex allows a leading -¶

    File: crucible/utils.py _VALID_NAME_RE = ^[A-Za-z0-9_.\-]+$ Docstring claims to block "leading-dash (arg injection)" but the regex accepts -rf, --fork-url, etc. Current call sites use names as path components (where ..///\ are blocked, so safe), so not exploitable today — but latent if any name is ever passed as a bare CLI token. Also . alone passes (though .. is blocked). - Fix: reject a leading - and a name equal to ..

    🟢 N5 — Chain-name → path surface still unvalidated¶

    File: crucible/chain_manager.py:428 (meta_path), :75 (anvil --cache-path); config.py:179 validates topology name but not the chains: list[str] entries meta_path(chain_name) builds states_dir / f"{chain_name}.meta.json" with no validate_name. Mitigated by start_topology requiring chain_name in self.config.chains, but config chain keys themselves are never validated — a key like ../../foo in testbed.yaml would write outside states/. Requires a malicious/edited config, so low risk, but it's the one H1 surface left uncovered. - Fix: validate_name on chain keys at config load (or in meta_path).

    Verified-correct (no issue)¶

    • C2 forge_script & python_script: .resolve() is done before relative_to(workspace.resolve()) on the chosen path → symlink-to-/etc escapes are blocked. ✅
    • C3 ordering: Content-Length cap enforced before rfile.read(); allowlist checked before _forward; non-integer/missing length rejected; method match is case-sensitive (correct for JSON-RPC). ✅ (the gaps are N1/N2, not ordering)
    • H1 core rules: validate_name correctly rejects .., /, \, null byte, empty/whitespace, >128 chars, non-[A-Za-z0-9_.-]. Absolute paths blocked. ✅ (except N4 leading-dash)
    • H4 fix: the synthetic chain_id no-op is gone everywhere; _call_llm raises on exhausted retries. ✅
    • No new regressions found in the other touched files (cache, checkpoint, plugin, config, experiment, scenario_steps) beyond the above.

    Recommended fix order¶

    1. N1 — batch-request crash (new regression, remote-triggerable). Quick: handle isinstance(req, list).
    2. N3 — make verify_bundle actually consume <run_id>.sha256, else C4 gives no real tamper-evidence.
    3. N2 — correct the misleading allowlist comment and/or split privileged lifecycle calls off the plugin-facing proxy.
    4. D1 — add import urllib.error (one line, removes latent crash).
    5. D4 / D5 — cap external response.read(); fix rpc.py:214 block > 0 → >= 0.
    6. N4, N5, D2, D3, D6 — hardening / robustness fast-follows.

    Bottom line¶

    All previously-reported issues are closed, and the genuinely dangerous original blockers (command injection, script/plugin exec, open-relay SSRF, body-DoS, path traversal, secret leak) are resolved. The remaining items are narrower:

    • N1 is a real new regression worth fixing before release (breaks batch clients).
    • N2/N3 mean C3 and C4 are "improved but not complete" against their stated security goals — a judgment call about how strong the proxy-privilege and artifact-integrity guarantees need to be for your release bar.
    • The rest (D1–D8, N4, N5) are robustness/defense-in-depth fast-follows, none release-blocking.

    With N1 fixed and a conscious decision on N2/N3, this is a credible release candidate.
