    Response to Team Feedback on Architecture Review¶

    Date: June 7, 2026

    Where the Team Is Right (and Why)¶

    1. Soft-reset leakage is real — verified in code¶

    I checked the implementation. The team's concern is precisely correct.

    baseline.py:105-108 — restore() calls chain_manager.restore() which calls JsonRpcClient.load_state() which calls anvil_loadState:

    # chain_manager.py:222-227
def restore(self, chain_name: str, snapshot_path: Path) -> Any:
    state = json.loads(snapshot_path.read_text(encoding="utf-8"))
    return JsonRpcClient(runtime.rpc_url).load_state(state)

# rpc.py:96-97
def load_state(self, state: Any) -> Any:
    return self.call("anvil_loadState", [state])

    Anvil's anvil_loadState merges the state blob into the current chain state — it does not reset the chain. If state has been modified between the baseline capture and the restore (e.g., from a previous experiment run), any storage slots or accounts not covered by the snapshot blob will retain their current values.

    This means: - Researcher A (medium reset: kill → restart → loadState) gets a clean fork + snapshot → deterministic state - Researcher B (soft reset: loadState only) gets snapshot merged onto dirty state → potentially non-deterministic

    The fix: Change the default reset mode from "soft" to "medium" in reset_cmd.py:18, and add a loud warning in the docs. Alternatively, the ExperimentRunner.run() should use medium reset between matrix runs.

    2. Silent degradation — confirmed, no artifacts_complete flag exists¶

    I verified: the run_manifest.json (built by provenance.py:240-250) contains no completeness indicator. The manifest records outcome: "success" even if the instrumentation proxy silently failed to trace some transactions.

    The specific code path that degrades silently:

    # instrumentation.py:203-213
def _do_trace(self, tx_hash: str) -> None:
    receipt = self._wait_for_receipt(tx_hash)
    if not receipt:
        log.step(
            "auto-instrumentation: receipt not found for tx",
            phase="attack", action="auto_trace", status="failure",  # ← logged
            tx_hash=tx_hash[:18],
        )
        return  # ← but no flag set, no counter incremented

    The failure is logged to run.jsonl (line 208-211, status="failure"), but: - The manifest doesn't check it - The publication gates don't check it - A researcher exporting the bundle has no machine-readable way to know artifacts are incomplete

    The team's recommendation is exactly right. A artifacts_complete: true|false field on the manifest — computed by scanning run.jsonl for auto_trace entries with status="failure" — is cheap and high-impact.

    3. Evidence.yaml pre-flight validation — feasible and high-value¶

    Currently, evidence.yaml validation (in evidence.py:36-97) checks only structure: are the required sections present, is the fidelity level valid, are the lists non-empty?

    It does not check: - Is fork_block actually the block the team claims? - Does the contract at affected_contracts[0] actually have code at that fork block? - Is the exploit_transaction hash valid?

    This could be added as a pre-flight check in ExperimentRunner.run(), right after _check_publication_gates():

    # Proposed: after line 156 in experiment.py
if publication_mode or self.config.publication.enabled:
    self._validate_evidence_against_chain(experiment)

    The check would: 1. Connect to the chain (after forking) 2. Call eth_getCode(affected_contract, fork_block) — must return non-empty 3. Call eth_getBlockByNumber(fork_block) — must exist

    This converts "the researcher claims it forked the right state" into "the testbed proves it." Exactly the kind of rigor that USENIX reviewers care about.

    4. Tutorial gap — confirmed¶

    29 docs, no TUTORIAL.md or "quickstart" path. The closest is USER_GUIDE.md (8KB, 12 sections) and ATTACK_PACK_GUIDE.md (34KB, very comprehensive but also very long). Neither says "run these 5 commands and you've reproduced a hack."

    5. LLM budget gap — confirmed¶

    agent_loop.py has no spend cap. The LLMAgentPlugin.run() loop (lines 186-250) iterates until is_done() returns True or max_steps is hit. But max_steps controls tool calls, not API spend. A frontier model could burn $50 in a single experiment with no warning.

    Where I Agree With the Team's Pushbacks¶

    1. Hermetic E2E with checked-in state fixture IS heavy¶

    The team is right. An Anvil state dump can be 50-200MB. Checking that into git is impractical.

    Their alternative — RPC response recording and replay — is better. Record the RPC calls during a real fork, save them as a fixture, and replay them in CI without a network connection. This gives: - Hermetic tests (no archive node needed) - Small fixtures (only the RPC responses that were actually used, not the entire chain state) - Fast execution (no real forking)

    This is the pattern used by Foundry's own test suite and by vcrpy in the Python world.

    2. Prompt injection from experiment YAML — low risk, agree¶

    Experiment YAML files are committed and reviewed. The attack vector requires a malicious file in the repo. Not worth prioritizing over correctness items.

    3. Cross-chain correlation IDs — design trade-off, not a bug¶

    The current counter-based approach (xchain-0001, xchain-0002, ...) is: - [ok] Human-readable during debugging - [ok] Deterministic within a single run (counter resets to 0 at run start) - [ok] Unique within a run (monotonic)

    Deterministic content-addressed IDs (hash of source_chain + dest_chain + tx_hash + payload) would be: - [ok] Reproducible across re-runs - [fail] Harder to read in logs

    Both are valid. The current approach is fine.

    4. Schema validation at log-write time — too heavy, agree¶

    Every emit() call would need a jsonschema.validate() call per stream. The instrumentation proxy processes transactions on background threads — adding schema validation there would slow down the critical path.

    The lighter approach is better: validate at bundle-time (crucible export or crucible bundle verify). Check that every JSONL line has the expected fields for its stream type. Same coverage, zero runtime cost.

    Where the Team Adds Value Beyond the Review¶

    1. crucible verify as the killer feature — AGREE, should be P1¶

    This closes the reproducibility loop. The flow would be:

    Researcher A: crucible experiment run → export → publish bundle
Researcher B: crucible verify <bundle.tar.gz>
              → unpack → re-run → hash-compare all artifacts
              → "PASS: 6/6 streams match original hashes"

    This is what turns Crucible from "a tool that logs things" into "a tool that proves reproducibility." Implementation:

    1. crucible verify unpacks the bundle
    2. Reads run_manifest.json for the experiment config, topology, fork block
    3. Re-runs the experiment from scratch
    4. Compares the new log_integrity.json hashes against the bundle's hashes
    5. Reports pass/fail per stream

    The core diffing infrastructure already exists in diff.py — this is mostly orchestration.

    2. Per-attack fidelity table in BENCHMARKS.md — cheap and high-impact¶

    Current BENCHMARKS.md says "2 at L2 + 4 scaffolds." A table like this would be much stronger:

    Attack Fidelity Deterministic Chains Why Not Higher
    Nomad L3 [ok] Ethereum Full tx replay on fork
    Wormhole L2 [ok] Ethereum eth_call probe, no state change
    Poly Network L3 [ok] Ethereum-only BSC archive needed for L4
    Horizon L1 [fail] — Scaffold only
    BNB Token Hub L1 [fail] — BSC archive needed
    Multichain L0 N/A — Narrative only

    3. .env setup in tutorial — yes¶

    The .env.example exists but no doc says "get a free Alchemy/Infura key, put it here, run this." This is the #1 onboarding friction point.

    Concrete Implementation: artifacts_complete Flag¶

    The cheapest high-impact fix. In experiment.py, after stop_instrumentation():

    # In the finally block of run(), after line 212
self.chain_manager.stop_instrumentation()

# Check for any failed auto-trace entries in run.jsonl
trace_failures = sum(
    1 for entry in log_mgr._root._load_jsonl("run.jsonl")
    if entry.get("action") == "auto_trace" and entry.get("status") == "failure"
)
artifacts_complete = trace_failures == 0

    Then include it in the manifest:

    # In _write_manifest, add to the return dict
return {
    ...
    "outcome": outcome,
    "artifacts_complete": artifacts_complete,
    "trace_failures": trace_failures,
    ...
}

    And in the publication gates, reject if artifacts_complete is False:

    # In publication.py PublicationGate
@staticmethod
def check_artifacts_complete(manifest: dict) -> GateResult:
    if not manifest.get("artifacts_complete", True):
        failures = manifest.get("trace_failures", 0)
        return GateResult(
            passed=False,
            failures=[f"{failures} transactions failed auto-instrumentation — artifacts are incomplete"],
        )
    return GateResult(passed=True)

    Concrete Implementation: Evidence Pre-Flight Check¶

    # New method on ExperimentRunner
def _validate_evidence_against_chain(self, experiment: ExperimentConfig) -> None:
    """Verify evidence.yaml claims against live chain state."""
    evidence = load_evidence(self.workspace / "attacks" / experiment.attack)
    historical = evidence.get("historical_scope", {})
    contracts = historical.get("affected_contracts", [])

    for contract_entry in contracts:
        address = contract_entry.get("address", "")
        chain = contract_entry.get("chain", "ethereum")
        if not address:
            continue

        runtime = self.chain_manager.runtime_for(chain)
        client = JsonRpcClient(runtime.rpc_url)
        code = client.get_code(address)

        if not code or code == "0x":
            raise CrucibleError(
                f"Evidence claims {address} on {chain} is a contract, "
                f"but it has no code at the fork block. "
                f"This usually means the address is wrong (EOA vs contract)."
            )

    This would have caught 4 of the 13 bugs found during the initial A-Z validation — the EOA-vs-contract address errors that caused the most debugging time.
