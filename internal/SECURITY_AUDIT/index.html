    Security Audit & Logging Architecture Assessment¶

    1. Mainnet Isolation — RPC Leakage Risk¶

    Audit Result: [ok] Fully Isolated — No Write Operations Reach Mainnet¶

    Layer Outbound Calls Writes to Mainnet? Risk
    Anvil fork eth_getBlockByNumber, eth_getStorageAt, eth_getCode (archive RPC) [fail] Read-only None — Anvil never forwards eth_sendTransaction upstream
    RPC Proxy Forwards to 127.0.0.1:8545 only [fail] Local only None — upstream_url is always the local Anvil
    Attack Packs Use runtime_for().rpc_url → always 127.0.0.1:N [fail] Local only None
    Scenario Steps context.chains.runtime_for(chain).rpc_url → local Anvil [fail] Local only None
    LogManager Writes JSONL files to local disk only [fail] Local only None — no network calls
    crucible doctor --rpc Pings configured archive RPC [fail] Read-only Checks connectivity, no writes
    crucible attack extract Fetches ABI from Sourcify / Etherscan APIs [fail] Read-only Reads contract metadata, no writes
    Notification hooks Optional webhook POST Configurable Disabled by default

    Key Isolation Guarantees¶

    1. Anvil's --fork-url is read-only. Anvil uses the archive RPC only to fetch historical state at the fork block. It never submits transactions upstream. This is a Foundry-level guarantee.

    2. All transaction traffic goes to 127.0.0.1. Every attack pack, forge script, and external tool that goes through the testbed connects to the local Anvil fork. The RPC proxy only forwards to http://127.0.0.1:<port>.

    3. No private keys on mainnet. The --auto-impersonate flag means all transactions use impersonated accounts — no real private keys are loaded. State-changing ops (anvil_setStorageAt, anvil_setBalance) are local-only anvil RPC methods that don't exist on mainnet nodes.

    4. LogManager is purely local. The per-chain LogEmitter instances write JSONL files to the local logs/ directory. No log data is ever transmitted over the network.

    5. crucible attack extract is read-only. The only external API calls are to Sourcify and Etherscan for ABI lookup — no transactions, no state writes. The extracted data is written locally.

    Residual Risk (Low)¶

    Risk Mitigation
    A forge script with --rpc-url pointing to mainnet instead of local Anvil The experiment runner replaces --rpc-url with the local Anvil URL in forge_script scenario steps
    A notification webhook sending experiment data to an external URL Disabled by default; researcher must explicitly configure it
    ETH_RPC_URL leaking via .env into version control .env is in .gitignore
    Sourcify/Etherscan API calls from crucible attack extract Read-only; no keys needed for Sourcify; Etherscan key is optional

    2. Logging Architecture — Is It Independent?¶

    Answer: [ok] Yes — Zero Hardcoded Data, Zero Per-Experiment Code¶

    The logging system operates at two independent layers:

    Layer 1: LogManager (Orchestrator) + LogEmitter (File Writer)¶

    Created automatically by ExperimentRunner.run():

    log_mgr = LogManager(run_dir, run_id, experiment.scenario, experiment.attack, experiment.name)
    • LogManager creates one LogEmitter per chain (via for_chain(name)) — each writes to a separate chain_<name>/ subdirectory
    • Root-level run.jsonl captures the experiment lifecycle (shared, not per-chain)
    • Bridge events go to a bridge/ subdirectory
    • Every entry is tagged with timestamp, run_id, scenario, attack — metadata passed at construction, not hardcoded
    • No hardcoded addresses, no chain-specific logic, no attack-specific branches

    Layer 2: Instrumentation Proxy (Automatic Artifact Capture)¶

    The RPC proxy auto-captures artifacts from any transaction on the fork:

    Transaction → RPC Proxy → Forward to Anvil → Background trace
                                            → log.gas()          — gas_profile.jsonl
                                            → log.event()        — events.jsonl
                                            → log.storage_diff() — storage_diffs.jsonl
                                            → log.balance_diff() — balance_diffs.jsonl
    • The proxy has zero knowledge of what attack is running
    • It captures every eth_sendTransaction and eth_sendRawTransaction regardless of sender
    • Same logic applies to Nomad, Poly Network, Horizon, or any future attack

    What the Researcher MUST Do vs. What's Automatic¶

    Task Automatic? Details
    Log stream creation [ok] Yes LogManager created by ExperimentRunner, creates per-chain LogEmitter instances
    Gas/Events/Storage/Balance capture [ok] Yes RPC proxy captures from every tx
    report.md generation [ok] Yes write_report() called in finally block
    Integrity hashing [ok] Yes write_integrity() called in finally block
    Use proxy URL (NOT hardcoded localhost) [ok] Required pattern Must use runtime_for().rpc_url instead of hardcoded http://127.0.0.1:8545
    Custom step logging (optional) ⏭️ Optional context.log.step() for experiment-phase annotations
    Attacker account setup [fail] Manual Impersonate + fund (done by attack pack)
    Transaction construction [fail] Manual The actual exploit logic (researcher's work)

    The One Rule¶

    Always use runtime_for().rpc_url — never hardcode http://127.0.0.1:8545:

    # [ok] Correct — gets proxy URL (auto-artifacts)
client = JsonRpcClient(context.chains.runtime_for("ethereum").rpc_url)

# [fail] Wrong — bypasses proxy (no artifacts)
client = JsonRpcClient("http://127.0.0.1:8545")

    Other than that, the researcher writes the exploit and gets artifacts automatically — no per-attack configuration, no logging code, no hardcoded data.
