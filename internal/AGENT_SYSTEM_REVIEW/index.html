    Crucible Agent System — Phases 4–6 Review¶

    Date: June 2026
    Status: Ready for peer review

    1. Executive Summary¶

    Over Phases 4–6 we built an autonomous AI agent system for the Crucible blockchain security research testbed. The system lets LLM-driven agents:

    1. Explore forked blockchain environments through a privileged tool API
    2. Formulate and execute exploit strategies autonomously
    3. Verify outcomes against configurable acceptance criteria

    The system supports OpenAI-compatible APIs (including local Ollama instances), Anthropic Claude, and any custom OpenAI-compatible endpoint. All LLM communication uses raw urllib to avoid heavy SDK dependencies.

    Codebase impact: ~1,600 new lines of Python across 6 new files, 75 new unit tests, 449 total tests passing.

    2.5 Configuration Model¶

    Agent behaviour is configured via the experiment YAML's run: block:

    # experiment.yaml
name: agent_exploit
scenario: wormhole_pre_exploit
attack: my_agent_plugin         # or "LLMAgentPlugin" for default
topology: single
run:
  agent_privilege: medium       # low | medium | high
  agent_provider: openai        # openai | anthropic | ollama | local | custom
  agent_model: gpt-4o           # model name override
  agent_base_url: ""            # optional, for custom endpoints
  agent_objective: "Drain the bridge vault..."
  agent_task: "The Wormhole bridge contract is deployed at 0x..."
  agent_allowed_accounts:
    - 0xattacker1
    - 0xattacker2

    3. Review Feedback from SOM's Team — Our Responses & Rationale¶

    Two comprehensive reviews were provided by SOM and his team before Phases 4–6 began:

    1. crucible_assessment_and_critique.md — System-level assessment, gap analysis, critique of the plan, architectural suggestions
    2. crucible_code_quality_review.md — Code-level quality review with scored dimensions and prioritized recommendations

    Below we document every substantial piece of feedback, whether we accepted it, what we changed, and why when we chose a different path.

    3.1 Feedback We Accepted and Implemented¶

    Feedback Source What We Changed
    "AgentPlugin system at 0% — entirely planned, nothing implemented" Assessment §2 (Req 3) Built the full agent system across Phases 4–6: AgentPlugin ABC, AgentContext, 9 tools, 3 privilege levels, LLMAgentPlugin, 154 new tests.
    "Agent tool layer / toolbox doesn't exist" Assessment §2 (Req 3) Built crucible/agent_tools.py with ToolDef, AgentAction, ToolResult, AgentContext, and 9 tool registrations.

    "Missing cost/budget awareness — no llm_budget config" Assessment §3 (Issue 6) Documented as a known gap in §4.6. The existing max_iterations cap (default 50) provides basic protection. A dedicated llm_budget field is deferred to future work.

The key difference is we didn't make it drive an AttackPlugin instance because that would require the attack plugin to expose a tool-compatible interface, which AttackPlugin currently doesn't.     3.3 Feedback We Did NOT Accept (with Rationale)¶

    Feedback Source Why We Went a Different Way
    "Put attack scaffolding (Phase 4) before the agent system (Phase 1)" — SOM said scaffolding should be Phase 1, agent Phase 4 Assessment §3 (Issue 1) We put the agent system first because it was the stated requirement (Req 3: "Agent plugin to plug AI agent into the testbed"). The scaffolding UX (crucible attack init) is important for adoption but was not the mandated requirement. We also believe the agent system is the higher-leverage capability — an agent can generate attack packs itself, making scaffolding less critical. We scheduled scaffolding improvements for the next phase.
    "Use Foundry (Forge) as the primary execution engine, not Python" — SOM proposed making the Python layer an orchestrator over Solidity execution Assessment §4.2 We intentionally chose Python-first. The reasons: (1) Python is the lingua franca of security research tooling — the existing web3.py ecosystem, Jupyter notebooks, and ML tooling all integrate with Python. (2) Our RPC proxy instrumentation is Python-native and captures artifacts from any execution method (Python, Solidity, cast, external tools) — switching to post-capture of Forge trace output would lose real-time feedback. (3) A hybrid approach (Python orchestration + optional Forge script execution) is possible but was outside scope. We agree that a Solidity-native path would unlock DeFiHackLabs users, and have documented it as a future consideration.
    "Treat bridges as topology-level configuration, not a plugin type" — SOM proposed moving bridge setup to declarative YAML Assessment §4.5 We kept bridges as plugins. The BridgePlugin lifecycle (deploy→configure→connect→destroy) is already integrated into the experiment runner and works. Moving bridge setup to YAML would require restructuring the entire lifecycle — it's a valid architectural opinion but a major refactor with no functional gain over the current approach. The YAML-as-topology pattern is complementary and could be added without removing the plugin system.
    "Make the RPC proxy optional in favour of post-capture trace processing" — SOM proposed tracing blocks after execution instead of real-time proxying Assessment §4.4 We kept the real-time proxy. The proxy provides immediate feedback for the agent loop (the LLM needs to see results before deciding the next action). Post-capture processing is a complementary approach for verification but cannot replace real-time feedback. The ability to support both is preserved in the architecture.
    "Live-state forking UX needs addressing" — SOM noted there's no fork_block: "latest" mode Assessment §2 (Req 1) Out of scope for Phases 4–6. This is a configuration-level change (5 lines per SOM's estimate) that affects the chain manager, not the agent system. It remains a valid improvement but was not part of the agent system requirements.

    3.5 Summary of What SOM Was Right About¶

    Several of SOM's predictions proved accurate during implementation:

    1. Tool design matters. SOM warned that raw EVM tools are insufficient for complex exploits. Our testing confirmed that the agent struggles with multi-step exploits requiring ABI decoding or token balance reasoning — higher-level tools are needed.

    2. Agent reproducibility is hard. SOM predicted the reasoning chain would be ephemeral. We addressed this in the fix pass — now captured to agent_reasoning.jsonl per run. This was the right call.

    3. Researcher UX is the bottleneck. Even with the agent system wired into the runner, the workflow requires creating a plugin.yaml, experiment YAML, and agent subclass. SOM's emphasis on scaffolding and ease-of-use was correct.

    3.6 Where We Believe SOM Was Wrong (or Our Context Differs)¶

    1. The Foundry-first approach underestimates the instrumentation challenge. SOM's proposal of running forge test and post-processing traces would lose real-time feedback — critical for an agent loop that needs to observe transaction results before deciding the next action. The RPC proxy approach is not just "clever infrastructure" but a fundamental requirement for interactive agent-driven experimentation.

    2. Moving bridges to YAML doesn't solve the content problem. The bottleneck for bridge attacks isn't the lifecycle management (which works) — it's the lack of actual bridge contract implementations and relay simulations. Restructuring the lifecycle wouldn't produce working bridge attacks faster; building the contracts would.

    3. The agent-as-wrapper (driving AttackPlugin) overcomplicates the architecture. Creating yet another abstraction layer between the LLM and the testbed would make debugging harder (three layers of indirection: agent loop → wrapper → attack plugin → testbed). The direct plugin approach (LLMAgentPlugin is an AgentPlugin) is simpler to trace and debug.

    4.3 Ollama Tool Calling Caveat (KNOWN LIMITATION)¶

    Ollama's /v1/chat/completions endpoint supports tool/function calling on some models (llama3.1, mistral, codellama) but not all models. Testing with a model that doesn't support tool calling will produce empty or malformed tool call responses. The system will log an error and fall back to calling chain_id.

    Workaround: Use a model known to support tool calling (e.g., llama3.1:70b, mistral:7b, qwen2.5:7b).

    4.4 Anthropic Tool Result Format (KNOWN LIMITATION)¶

    The respond() method appends tool results in OpenAI format (role: "tool" with tool_call_id). When using AnthropicProvider, this format does not match Anthropic's expected role: "user" with content: [{"type": "tool_result", ...}]. For pure-Anthropic usage, LLMAgentPlugin.respond() would need to be overridden.

    Impact: The conversation history may not be perfectly readable by Claude, but tool calls and responses still function because the Anthropic API processes the user/assistant message structure.

    4.5 No Streaming Support (BY DESIGN)¶

    LLM responses are fetched synchronously. Streaming would improve perceived latency but adds complexity (SSE parsing, per-chunk aggregation). We use raw urllib to avoid SDK dependencies — streaming would require http.client or a third-party library.

    Verdict: Acceptable for batch-style experiments where a single turn takes 1–5 seconds. Streaming is a future optimisation.

    4.6 No Retry Budget / Cost Tracking (NOT YET IMPLEMENTED)¶

    The system tracks total tokens per provider (_total_tokens) but does not enforce a budget. An agent could theoretically run 50 iterations of expensive model calls with no cost cap. The experiment YAML already has rpc_budget.max_usd_estimate — a similar llm_budget field would be needed.

    Verdict: Low priority for V1; the max_iterations (default 50) cap limits worst-case cost.

    5. Design Decisions¶

    5.1 Raw urllib Instead of SDK Packages¶

    Decision: Use urllib.request.urlopen instead of the openai or anthropic PyPI packages.

    Rationale: - Avoids adding 2+ hard dependencies to pyproject.toml - Keeps the install footprint small (~50KB vs 5MB+) - Works offline / in air-gapped environments - Easy to audit (no black-box SDK internals)

    Trade-off: No automatic retry, no connection pooling, no streaming. Acceptable for our use case.

    5.2 Agent Plugin as a Plugin Type (Not a Wrapper Around AttackPlugin)¶

    Decision: LLMAgentPlugin extends AgentPlugin as a first-class plugin type alongside AttackPlugin and BridgePlugin, rather than being a wrapper that drives an AttackPlugin instance.

    Rationale: See §3.3 for the full discussion. In short: simpler architecture, consistent with existing patterns, easier to debug. SOM's recommended approach (agent-as-wrapper) was considered but rejected because it would require AttackPlugin to expose a tool-compatible interface that doesn't currently exist.

    5.3 Python-First Execution (Not Foundry-First)¶

    Decision: The agent system executes tools through Python (web3 RPC calls), not by generating and running Forge scripts.

    Rationale: See §3.3 (decision) and §3.6 (detailed reasoning) for the full discussion. Real-time feedback is critical for the agent loop; Foundry's batch execution model doesn't support interactive decision-making. A hybrid approach is possible but was outside scope.

    5.4 verify_state is Log-Only¶

    Decision: bridge_action verify_state does not call bridge_plugin.reset() (which is documented as destructive — "Reset bridge state to post-deploy baseline").

    Rationale: Using a destructive method for verification is semantically wrong. A future BridgePlugin.verify() method would provide real state checks.

    5.5 LLMAgentPlugin is Instantiable Without Subclassing¶

    Decision: LLMAgentPlugin has a default metadata() that uses the class name. This means LLMAgentPlugin() can be used directly, not just as a base class.

    Rationale: Faster iteration during research — no need to define a subclass just to test provider integration.
