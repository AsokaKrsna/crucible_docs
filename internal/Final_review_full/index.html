    Part III: Correctness Findings (Verified Against Code)¶

    3.1 Soft-Reset State Leakage [warn] CRITICAL¶

    anvil_loadState merges state — it does not wipe. The restore chain is:

    baseline.restore() → chain_manager.restore() → JsonRpcClient.load_state() → anvil_loadState

    If two experiment runs use soft reset between them, storage slots and accounts created by Run 1 that aren't in the baseline snapshot will persist into Run 2. This breaks determinism guarantees for parameter sweeps and A/B comparisons.

    Fix: Change default from "soft" to "medium" in reset_cmd.py:18 (10 minutes).

    3.2 Silent Artifact Degradation [warn] CRITICAL¶

    The run_manifest.json contains no completeness indicator. The manifest records outcome: "success" even if the instrumentation proxy silently failed to trace transactions.

    The specific failure path (instrumentation.py:203-213): 

    def _do_trace(self, tx_hash):
    receipt = self._wait_for_receipt(tx_hash)
    if not receipt:
        log.step(..., status="failure")  # logged to run.jsonl
        return  # but no flag set, no counter incremented

    The failure IS logged to run.jsonl, but: - The manifest doesn't check it - The publication gates don't check it - A researcher exporting the bundle has no way to know artifacts are incomplete

    Fix: Add artifacts_complete: true/false + trace_failures count to run_manifest.json. Additionally, compare Anvil's actual transaction count against traced count to catch proxy-bypass gaps.

    3.3 Evidence.yaml Has No Runtime Validation¶

    evidence.py:36-97 validates structure only (sections present, fidelity level valid, lists non-empty). It does NOT check: - Is fork_block actually the block claimed? - Does the contract at affected_contracts[0] actually have code at that fork block? - Is the exploit_transaction hash valid?

    Fix: Pre-flight check after chain fork: eth_getCode(contract) + eth_getBlockByNumber(fork_block). This would have caught 4 of the 13 bugs found during A-Z validation (EOA-vs-contract address errors).

    3.4 No CI-Runnable E2E Attack Test¶

    The one real reproduction test requires an archive node and is always skipped. CI proves nothing about whether attacks still reproduce after refactors.

    Fix: RPC response recording/replay (10-200KB fixtures vs 50-200MB state dumps). The RpcProxy already intercepts all traffic — add serialize/deserialize modes.

    3.5 LLM Agent Has No Dollar-Cost Budget¶

    agent_loop.py iterates until is_done() or max_steps. max_steps controls tool calls, not API spend. A frontier model can burn $50+ in a single experiment.

    Fix: Add llm_budget parameter (USD or tokens) with hard-stop.

    Part IV: Architectural Critique¶

    4.1 The Threat Model Is Implicit — It Needs to Be Explicit¶

    A USENIX reviewer will ask: "What can go wrong between your fork and the real chain?"

    What Crucible guarantees (EVM-level fidelity):

    Property Guarantee
    State fidelity Identical storage/balance/code at block N
    Execution fidelity Same EVM opcode execution
    Gas accounting Same opcode costs
    Event emission Same code paths
    Artifact integrity SHA-256 hashed, provenance-tracked

    What Crucible does NOT guarantee (protocol-level fidelity):

    Property Limitation
    Mempool dynamics Anvil auto-mines FIFO — no MEV, no front-running
    Block timing Timestamp frozen at fork block
    Cross-chain relay timing Bridge plugins are synchronous mocks
    Consensus behavior Single-node devnet — no reorgs, no finality delays
    External oracle state Oracle values reflect fork block, not exploit window

    This is not a weakness — it's a scope boundary. The paper must say: "We reproduce the EVM execution of N bridge exploits." Not: "We faithfully reproduce bridge exploits." The qualifier matters.

    4.2 The Instrumentation Proxy Has an Observability Boundary¶

    The proxy captures eth_sendTransaction and eth_sendRawTransaction that flow through it. It CANNOT capture: - Plugin opening a direct connection to Anvil (bypassing proxy) - Forge script with hardcoded RPC URL - Internal Anvil state changes via cheatcodes

    The transparent URL substitution (runtime_for() returns proxy URL) makes the common case work, but there's no enforcement.

    Fix: Post-run consistency check: 

    expected_tx_count = latest_block.txs - fork_block.txs
actual_traced     = count(gas_profile.jsonl entries)
if mismatch → artifacts_complete = false

    4.3 The Bridge Model Is a Mock, Not a Simulation¶

    BridgePlugin.send_message() default implementation: 

    def send_message(self, context, source_chain, dest_chain, data, correlation_id=None):
    context.log.step(f"bridge send_message {source_chain} -> {dest_chain}", ...)
    return {"source_tx_hash": "0x", "correlation_id": correlation_id or ""}

    This is a no-op stub. The bridge behavior is whatever the researcher codes. For L2/L3 (single-chain), this doesn't matter — the exploit happens on one chain. For L4 (multi-chain), a reviewer will ask: "How do you know the mock is faithful?"

    Fix (documentation, not code): L4 evidence.yaml must require: (1) protocol specification reference, (2) behavioral assertions, (3) excluded components with impact analysis.

    4.4 Fidelity Levels Describe Metadata, Not Behavior¶

    The current system: - L0: narrative only - L1: conceptual (no additional checks) - L2: must have chains + affected_contracts in evidence.yaml - L3: must have exploit_transactions in evidence.yaml - L4: must have excluded_components

    Problem: An L3 attack that lists exploit_transactions but doesn't replay them is still classified L3. No runtime check that declared fidelity matches actual execution.

    Fix: For L3, compare reproduction results against real chain receipts (gas, event count, balance change).

    4.5 Cross-Chain Timing Is Undefined¶

    cross_chain.jsonl records correlation IDs but not timing. In real bridge exploits, relay delay and finality are part of the attack. In Crucible, both chains are instant-mine — send_message() is synchronous.

    Fix (documentation): State explicitly: "Cross-chain timing is not simulated. Exploits depending on relay latency or finality gaps are not reproducible at L4 without a custom BridgePlugin."

    4.6 The Diff Engine Proves Structural Equality, Not Semantic Equivalence¶

    diff.py:compare_runs() compares storage diffs, events, gas, and balances structurally. Two runs of the "same exploit" that drain 100 ETH via different transaction counts will fail the diff even though they're semantically equivalent.

    This is fine for its intended purpose (proving determinism of identical runs). But crucible verify should use hash comparison (from log_integrity.json), not diff comparison.

    Part V: Why Crucible, Not Just Foundry¶

    This is the most important architectural question. Foundry (Anvil + Forge) already does most of what's needed:

    // DeFiHackLabs pattern — 300+ exploits, cited in papers
contract NomadExploit is Test {
    function testExploit() public {
        vm.createSelectFork("mainnet", 15259100);
        replica.process(fraudulent_message);
        assertGt(token.balanceOf(attacker), 0);
    }
}

    What Foundry handles: fork at any block, run exploit transactions, traces (-vvvv), gas reports, snapshot/restore, determinism.

    What Crucible genuinely adds — three things Foundry can't do:

    # Capability Why Foundry can't Crucible's approach
    1 Structured, machine-readable artifacts forge test -vvvv → unstructured text. Can't hash, diff, or verify it. RPC proxy → typed JSONL + SHA-256 integrity
    2 Multi-chain coordination Forge forks one chain per test N Anvil instances, each with instrumentation proxy
    3 AI agent in the attack loop Forge is Solidity-only AgentPlugin + LLM loop + 9 privileged tools

    Everything else — config management, plugin YAML manifests, scenario DSL, baseline management, topology profiles, 17 CLI subcommands — is orchestration. It exists because Foundry has no experiment management layer, and academic reproducibility needs one. But it's not the core value.

    Where the Orchestration IS Justified¶

    • Multi-chain topologies — Managing 3 Anvil instances needs structured config
    • Parameter sweeps — Running 10 fork blocks needs experiment YAML
    • Publication mode — Provenance + gates need explicit configuration
    • Team collaboration — YAML files are git-tracked and reviewable

    Where the Orchestration Is Overweight¶

    Layer Issue Leaner alternative
    Plugin manifest (plugin.yaml + PluginRegistry) Dynamic importlib loading with YAML discovery Convention: attacks/<name>/<name>.py with class Attack
    Scenario engine (8 step types) Mini-DSL for setup phases Let researchers write setup in Python
    3-tier reset system Exists because Crucible manages Anvil externally Foundry's vm.snapshot() is one line
    Topology YAML profiles 5-line YAML files CLI flags: --chains eth,bsc --bridge wormhole

    Recommendation: Keep what exists (it works, 526 tests pass), but make the simple path simple. Convention over configuration for the common case. YAML and plugins stay for power users.

    8.1 artifacts_complete Flag¶

    # In experiment.py, finally block after stop_instrumentation():
trace_failures = sum(
    1 for entry in log_mgr._root._load_jsonl("run.jsonl")
    if entry.get("action") == "auto_trace" and entry.get("status") == "failure"

artifacts_complete = trace_failures == 0

    # In provenance.py build_run_manifest(), add to return dict:
return {
    ...
    "outcome": outcome,
    "artifacts_complete": artifacts_complete,
    "trace_failures": trace_failures,
    ...
}

    # In publication.py, new gate:
@staticmethod
def check_artifacts_complete(manifest: dict) -> GateResult:
    if not manifest.get("artifacts_complete", True):
        failures = manifest.get("trace_failures", 0)
        return GateResult(
            passed=False,
            failures=[f"{failures} transactions failed auto-instrumentation"],
        )
    return GateResult(passed=True)

    8.2 Evidence Pre-Flight Check¶

    # New method on ExperimentRunner
def _validate_evidence_against_chain(self, experiment: ExperimentConfig) -> None:
    evidence = load_evidence(self.workspace / "attacks" / experiment.attack)
    historical = evidence.get("historical_scope", {})

    for contract_entry in historical.get("affected_contracts", []):
        address = contract_entry.get("address", "")
        chain = contract_entry.get("chain", "ethereum")
        if not address:
            continue

        runtime = self.chain_manager.runtime_for(chain)
        client = JsonRpcClient(runtime.rpc_url)
        code = client.get_code(address)

        if not code or code == "0x":
            raise CrucibleError(
                f"Evidence claims {address} on {chain} is a contract, "
                f"but it has no code at the fork block. "
                f"This usually means the address is wrong (EOA vs contract)."
            )

    The Core Value Proposition¶

    Crucible adds three things Foundry can't do: (1) structured, hashable, diffable artifacts captured transparently, (2) multi-chain coordination, (3) AI agent integration. Everything else is orchestration — useful for reproducibility, but not the differentiator.

    What Would Make a USENIX Reviewer Accept This¶

    1. Explicit fidelity boundary — "We reproduce EVM execution, not protocol behavior"
    2. artifacts_complete flag — Proves artifacts are complete, not assumed
    3. Evidence pre-flight validation — The testbed proves state matches claims
    4. At least one hermetic CI test — Demonstrates reproducibility without trusted infrastructure
    5. crucible verify — Closes the loop: anyone can re-derive and verify

    What Would Make a USENIX Reviewer Reject This¶

    1. Claiming "faithful reproduction" without qualifying what "faithful" means
    2. Publishing artifacts from a run where traces silently failed
    3. Bridge simulation at L4 without documenting what the mock does and doesn't simulate
    4. No way for a reviewer to independently verify the reproduction

    The architecture is sound. The claims need to be as precise as the code.
