    Crucible — Website & Documentation Quality Assessment¶

    Prepared for: Engineering team Date: 2026-06-15 Scope: Documentation (docs/, README.md), the MkDocs website (website/), and their consistency with the codebase at the current working tree (v0.4.0). Method: Static analysis only — no code executed, no site served. Documented commands/flags/APIs/claims were verified against the actual source (crucible/), the attack packs (attacks/), and the repo's tracked-file state. Findings from automated sweeps were re-verified by hand before inclusion; corrections are noted.

    Verdict: NOT release-ready — but the problem is packaging, not content¶

    This is the key distinction for planning:

    • Documentation content (prose accuracy vs. code): GOOD. The CLI reference, tutorial, user guide, scenario-authoring, plugin-development, fidelity, and LLM-agent docs were verified command-by-command and API-by-API against the source and are accurate or only minor-drift. Someone following the content will mostly succeed. This is genuinely strong and rare.
    • Release packaging (README links, website config, confidentiality, branding): NOT READY. The front door (README) has ~27 broken documentation links, the public website is configured to publish 20 internal documents (including a security audit and USENIX submission notes), and the site still carries placeholder branding. These are blockers for a public/academic release.

    Bottom line: the writing is shippable; the release hygiene is not. None of the blockers are hard to fix — they're configuration, link paths, and file-placement decisions — but until they are fixed, do not publish the site or tag a public release, and in particular do not run mkdocs gh-deploy, which would expose the internal docs.

    Severity-ranked findings¶

    🔴 BLOCKER¶

    W1 — The public website is configured to publish the entire docs/internal/ tree (20 documents). website/mkdocs.yml (docs_dir: ../docs) lists every internal doc in its nav under "Developer Internals", "Planning & Strategy", and "Reviews" (lines 112–138). All 20 docs/internal/*.md are git-tracked, so mkdocs build/gh-deploy will render and publish them. Exposed material includes: - internal/SECURITY_AUDIT.md — threat model, isolation analysis, and a residual-risk table enumerating leak/exfiltration vectors (adversary-useful, self-incriminating). - internal/USENIX_SUBMISSION_NOTES.md — double-blind de-anonymization hazard. A page literally titled "USENIX Submission Notes" on the project's public site links the public artifact to the (anonymous) submission, and leaks evaluator-facing strategy and "known issues for evaluators." - internal/DOCS_GAP_ANALYSIS.md — admits "61% doc coverage, 20 missing features, CLI 67%, core modules 49%." - internal/STRATEGIC_PLAN.md, ROADMAP_TO_100.md, IMPLEMENTATION_STATUS.md — internal roadmaps/status. - internal/Final_review_full.md, feedback_response.md, thearchitectural_and_code_review.md, DEVELOPER_HANDOFF_SUMMARY.md — internal peer-review threads with reviewer scores and bug lists.

    This is the single most serious item. Publishing the security audit and USENIX submission notes is both a confidentiality and a (potential) double-blind-integrity problem. Fix: remove all internal/* entries from mkdocs.yml nav (and from docs/index.md, see W3); keep internal docs out of docs_dir, e.g. move them to a non-published location or add an exclude. Decide deliberately which (if any) technical internals — TECHNICAL_OVERVIEW, DESIGN_DECISIONS, BRIDGE_PLUGIN_LIMITATIONS, DOCKER_IMPLEMENTATION — to keep, after review.

    W2 — README documentation links are almost entirely broken (≈27 of ~39). Every link in the README "Documentation" tables points to website/docs/…, but there is no website/docs/ directory (the site's docs_dir is ../docs; the docs live in docs/). Verified: website/docs/** matches zero files. Additionally, three targets use pre-rename filenames that no longer exist: GETTING_STARTED.md (now USER_GUIDE.md), ARCHITECTURE_AND_DESIGN.md (now ARCHITECTURE.md), SCOPE_AND_CAPABILITIES.md (now CAPABILITIES.md). A reader clicking almost any doc link from the README (GitHub front page) hits a 404. Fix: rewrite website/docs/… → docs/… and correct the three renamed targets. docs/index.md already uses the correct paths and can be used as the reference.

    🟠 HIGH¶

    W3 — The public site landing page (docs/index.md) itself links to internal docs. Independent of the nav, docs/index.md:102–111 links to internal/DEVELOPER_HANDOFF.md, IMPLEMENTATION_STATUS.md, ROADMAP_TO_100.md, STRATEGIC_PLAN.md, SECURITY_AUDIT.md, AGENT_SYSTEM_REVIEW.md, etc. So even with the nav fixed, the home page routes readers to confidential material. (index.md is the site Home.) Fix: strip the "Internal & Reference" internal links from docs/index.md.

    W4 — Nine internal review reports are git-tracked at the repo root and will ship in any source/Zenodo archive. CODE_REVIEW_REPORT.md, DOCS_VERIFICATION_REPORT.md, REMAINING_ISSUES_REPORT.md, SCIENTIFIC_INTEGRITY_AUDIT.md, FINAL_RELEASE_SCAN.md, FINAL_PROJECT_REPORT.md, FINAL_PRE_RELEASE_REPORT.md, FINDINGS_AND_IMPROVEMENT_PLAN.md, REVIEW_FINDINGS.md — all tracked. These are candid internal assessments (the full vulnerability history, including this audit series). They should not be in a public artifact. Fix: move to an untracked scratch/-style location or delete before tagging. (This assessment file itself is in the same category — relocate it too once actioned.)

    W5 — Central scientific claim (fidelity tiering) in README contradicts the artifacts. README.md:5 states "3 at L2 reproducibility (Nomad, Wormhole, Poly Network), 2 executable scaffolds (Horizon, BNB Bridge), 1 narrative (Multichain)." But the actual evidence.yaml declarations are: Nomad L2, Wormhole L2, Poly Network L2, Horizon L2, BNB L2, Multichain L1. So five packs declare L2, not three — Horizon and BNB are described as "scaffolds" in the README yet declare fidelity_level: L2 (with exact_tx_replay: false) in their evidence. A USENIX artifact evaluator cross-referencing the README against evidence.yaml will see a direct contradiction in the headline reproducibility claim. Fix: reconcile the two — either downgrade Horizon/BNB evidence.yaml to a scaffold-appropriate level, or restate the README tiering to match the declared levels and explain the "scaffold = L2 without exact_tx_replay" nuance.

    W6 — Site branding is placeholder/internal. website/mkdocs.yml: site_description: "Internal documentation for the Crucible…" (the word Internal on a public site, line 7); repo_url/social link = https://github.com/your-org/crucible (unfilled placeholder, lines 9 & 59) → the "edit" and GitHub links on every page are broken. Fix: set a public-facing description and the real repo URL.

    🟡 MEDIUM (content accuracy — fix before release, low effort)¶

    W7 — README.md:15 claims "808 unit tests"; the real count is different. Actual def test_ functions under tests/unit/ = 757; pytest collects ~805 items (parametrization); a prior internal report said 805. Three different numbers, none of which is 808. Also "no external deps" is slightly misleading: the bridge Solidity tests require Foundry/forge (they fail without it). Fix: state the real collected count and qualify the foundry-dependent subset.

    W8 — README.md:37 says "agent plugins planned" — but they already exist. The codebase ships AgentPlugin (sdk.py:174), the agent loop (agent_loop.py), LLM providers (llm_providers.py), agent attack packs (attacks/llm_exploit_agent, attacks/demo_agent), and a full docs/RUNNING_LLM_AGENTS.md. "Planned" is stale and contradicts the rest of the docs. Fix: change to reflect that agent plugins are implemented.

    W9 — docs/CUSTOM_CONTRACT_TESTING.md falsely claims "10 pre-funded accounts (10,000 ETH each)". Anvil is launched with --accounts 0 --balance 0 (chain_manager.py:92–93), so no default dev accounts exist. The doc's worked example survives only because the deploy script funds the victim via vm.deal() with gas-price 0; the stated "account #5 attacker with 10,000 ETH" is false and will mislead users writing their own scripts. (CUSTOM_CONTRACT_TESTING.md:62–64, 503–506) Fix: correct the environment description to match --accounts 0 --balance 0.

    W10 — docs/SCENARIO_AUTHORING.md verify_tracing example uses the wrong field. Doc shows chain: ethereum (singular); the handler reads chains (plural list) plus min_events/min_gas_entries/min_storage_diffs/min_balance_diffs/check_consistency (scenario_steps.py:386–391). A copy-pasted example silently does nothing useful. Fix: correct the field name and document the real tuning fields.

    W11 — docs/PLUGIN_DEVELOPMENT.md:108 shows crucible attack init my_exploit --type attack — --type does not exist. attack init flags are --chain/--fidelity/--author/--with-scenario/--from-tx/--rpc-url (commands/attack.py). The command as documented errors out. Fix: remove --type.

    🟢 LOW (polish)¶

    • W12 — README "6 benchmark + 2 demo packs" doesn't map cleanly to the 11 dirs in attacks/. Beyond the 6 benchmarks and 2 demos there are l4_full_validation, nomad_bridge_demo, llm_exploit_agent (validation/agent fixtures). Either fold them into the count narrative or note them as fixtures.
    • W13 — Authoring docs disagree with each other on evidence.yaml field shapes: attacker_addresses (plural) vs attacker_address (singular); required_assertions as list-of-dicts vs list-of-strings. The validator tolerates both, but the docs should be self-consistent.
    • W14 — docs/CLI_REFERENCE.md:175 tags bundle verify as [deprecated], but the command is fully functional with no deprecation in code. Remove the stale tag or implement the deprecation.
    • W15 — docs/TUTORIAL.md has a hand-written doctor sample output that doesn't match the real Rich table, a duplicated table header (~line 423), and PowerShell code fences on Unix commands. Cosmetic.
    • W16 — commands/attack.py:153 has a stale help example referencing a nonexistent --bsc-rpc-url flag (code comment, surfaces in --help).
    • W17 — Missing standard release files: no CONTRIBUTING.md, no CHANGELOG.md. (LICENSE ✓ MIT and CITATION.cff ✓ are present — good for an academic artifact.)

    What was checked and is genuinely good¶

    • CLI documentation is faithfully in sync with the code. CLI_REFERENCE.md, TUTORIAL.md, USER_GUIDE.md, SIMPLE_OPERATIONS.md, OPERATIONS_RUNBOOK.md were verified against all 21 command modules: no documented command/flag is missing or fictional, and every example file (experiments/*.yaml, topologies/*, plugin paths) referenced exists on disk. Verdict: ACCURATE.
    • Plugin SDK, scenario steps, fidelity levels, and LLM-agent docs match the code. All 11 @register_step handlers are documented; SDK class names (CruciblePlugin/AttackPlugin/BridgePlugin/AgentPlugin), method signatures, and AttackResult/VerificationResult fields match sdk.py; fidelity L0–L4 checks and error strings match enforcer.py verbatim; LLM provider names/models/env vars/tool-privilege counts match llm_providers.py/agent_tools.py. (Only W9/W10/W11 drift within these.)
    • "7 JSONL streams" is accurate (6 in STREAMS + agent_reasoning.jsonl). "$1.9B+" is reasonable (sum of the six historical losses is ≈$2B). SHA-256 integrity / publication gates / determinism pinning claims match the code reviewed in prior rounds.
    • docs/index.md uses the correct current doc filenames (unlike the README) — it was updated when files were renamed; the README was not.

    Corrected (claims that did NOT hold up on verification)¶

    • crucible.db is NOT git-tracked (an automated sweep claimed it was). git ls-files crucible.db → empty. Not a finding.
    • website/site/ is correctly gitignored (website/.gitignore → site/) and untracked — it will not ship via git. It is a stale local build (built from the pre-rename, pre-internal/-expansion structure: orphan GETTING_STARTED/, ARCHITECTURE_AND_DESIGN/, SCOPE_AND_CAPABILITIES/ pages, only 4 of 20 internal pages rendered). Harmless to git, but it confirms the deploy is not reproducible from current source — a fresh build is required (and W1 activates on that build unless the nav is fixed first).

    Release-readiness checklist (documentation/website)¶

    Must fix before any public release / site deploy: 1. [ ] W1 — Remove internal/* from mkdocs.yml nav; keep internal docs out of the published docs_dir. Do not gh-deploy until done. 2. [ ] W3 — Remove internal links from docs/index.md. 3. [ ] W2 — Fix the ~27 broken README links (website/docs/… → docs/…; fix 3 renamed targets). 4. [ ] W4 — Relocate/delete the 9 root review reports (and this file) before tagging. 5. [ ] W5 — Reconcile the README L2 tiering with the evidence.yaml fidelity levels. 6. [ ] W6 — Real repo_url + public site_description.

    Should fix (low effort, credibility): 7. [ ] W7 (test count + "no external deps"), W8 ("agent plugins planned" stale), W9 (10-accounts false), W10 (verify_tracing field), W11 (attack init --type).

    Nice to have: W12–W17 (count narrative, doc self-consistency, stale [deprecated] tag, tutorial cosmetics, --bsc-rpc-url help, CONTRIBUTING/CHANGELOG).

    Why this verdict¶

    For a normal OSS tool, the broken README links and placeholder repo URL alone would gate a release. For a USENIX-grade academic artifact, W1 (publishing the security audit and USENIX submission notes) and W5 (headline reproducibility tiering contradicting the artifacts) are disqualifying until fixed — these are exactly what artifact evaluators and double-blind chairs scrutinize. The encouraging part is that the underlying documentation is accurate and well-maintained against the code; the remaining work is bounded, mechanical, and mostly about what to publish and fixing link paths — not rewriting content. Estimate: a focused half-day clears the blockers.

    Static review only. No code was executed and no files other than this report were modified. Line numbers and counts reflect the working tree at review time.
