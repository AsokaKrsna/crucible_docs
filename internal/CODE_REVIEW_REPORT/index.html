    Crucible — Code Review Report¶

    Date: 2026-06-13 Reviewed version: 0.4.0 (pyproject.toml) Scope: Static code review of the crucible/ Python package (~13,300 LOC across 60 modules). Method: Read-only static analysis. No code was executed. Documentation was not verified against the codebase (per request). Findings below were located by direct reading and cross-checked against the source.

    Headline verdict: NOT ready for release. Multiple arbitrary-code-execution / injection paths and an integrity model that does not actually provide tamper-resistance. Details and minimum release bar at the end.

    1. Summary¶

    Severity Count
    🔴 Critical 5
    🟠 High 6
    🟡 Medium 8
    🟢 Low / latent 6

    Theme: The project is architecturally mature — clean module separation, a thin CLI entry point, consistent yaml.safe_load, explicit HTTP timeouts, and a substantial test suite (49 test files, ~776 test functions). However, the security posture is inconsistent with the project's purpose. For a tool whose core value proposition is reproducible, tamper-evident security-research artifacts, the presence of command injection, unsandboxed script/plugin execution, an open RPC proxy, and a self-attesting (forgeable) integrity manifest is disqualifying for a production release.

    2. Critical Findings¶

    C1 — Command injection in notification hooks¶

    File: crucible/utils.py:32-34 Severity: CRITICAL

    A notification hook command string from config (testbed.yaml / testbed.local.yaml, which expands env vars) is .format()-substituted and executed with shell=True. Any shell metacharacter in config or env executes arbitrarily.

    cmd = hook.get("command", "").format(outcome=outcome, run_id=run_id)
try:
    subprocess.run(cmd, shell=True, timeout=30, capture_output=True)

    Fix: Use list-form argv with shell=False, or at minimum shlex.split and validate. Do not interpolate untrusted values into a shell string.

    C2 — Arbitrary code execution via scenario python_script / forge_script steps¶

    File: crucible/scenario_steps.py:272-322 (and 249-269) Severity: CRITICAL

    The python_script step resolves a YAML-supplied script path relative to the workspace and exec_modules it. There is no check that the resolved path stays inside the workspace, so script: ../../../etc/something.py escapes and executes any .py on disk. The forge_script step has the analogous problem shelling out to forge script <script>.

    script_path = (workspace / script_rel).resolve()   # no containment re-check
...
spec.loader.exec_module(module)

    Fix: After resolve(), assert the path is_relative_to(workspace). Validate script against an allowlisted directory.

    C3 — Open RPC proxy: SSRF + unbounded request body (DoS)¶

    File: crucible/instrumentation.py:213-268 Severity: CRITICAL

    The instrumentation HTTPServer forwards any POST body verbatim to the upstream RPC with no method allowlist, no auth, and no body-size cap. It will relay privileged calls (anvil_setBalance, anvil_dumpState, arbitrary eth_sendTransaction). The bind host is caller-controllable; if ever set to 0.0.0.0 or run on a shared network, any actor gains unauthenticated control of the fork and a pivot to the upstream archive RPC.

    content_length = int(self.headers.get("Content-Length", 0))  # unvalidated, no cap
body = self.rfile.read(content_length)
response = self._forward(body)

    Additional problems on this path: - int(...) on a malformed Content-Length header raises ValueError that escapes the handler. - send_response(200) is sent unconditionally even when _forward returned an error body — upstream failures masked as HTTP 200.

    Fix: Bind to loopback only and reject non-loopback host; add a JSON-RPC method allowlist; cap Content-Length; guard the integer parse; propagate upstream status.

    C4 — Integrity manifest is self-attesting (forgeable)¶

    File: crucible/export.py:36-55, 254-281 Severity: CRITICAL

    The SHA-256 "integrity manifest" (artifact_bundle_manifest.json) is stored inside the same bundle it attests to, and verify_bundle recomputes file hashes and compares them against the hashes listed in that same manifest. There is no signature, MAC, or external root of trust. Anyone editing a file can recompute its hash and overwrite the manifest entry. This detects accidental corruption only — not tampering — despite being framed as publication-grade integrity.

    "artifacts": {rel: sha256_file(path) for rel, path in sorted(files.items())}
...
manifest = json.loads(manifest_path.read_text(...))   # trusts the bundle's own manifest
expected = manifest.get("artifacts", {})

    Fix: Sign the manifest (e.g. detached signature / minisign / GPG), or publish the manifest hash out-of-band (Zenodo DOI, git tag). At minimum record the manifest's own hash somewhere external to the bundle.

    C5 — Code execution on plugin discovery¶

    File: crucible/plugin.py:104, 147-155 Severity: CRITICAL

    discover() → validate() calls spec.loader.exec_module(module) on every plugin found by globbing plugins/*/plugin.yaml. Merely listing/discovering plugins executes their top-level module code — before any trust decision. The same exec_module runs again in load(), doubling the blast radius. A malicious plugin directory dropped into the workspace runs code as soon as discovery occurs.

    module = importlib.util.module_from_spec(spec)
sys.modules[module_name] = module
spec.loader.exec_module(module)   # arbitrary code runs on discovery

    Fix: Separate metadata parsing (read plugin.yaml only) from code import. Only import a plugin when the user explicitly opts to run it, ideally behind a trust/allowlist gate.

    3. High Findings¶

    H1 — Path traversal from name → path across the codebase¶

    Files: crucible/scenario.py:33, crucible/config.py:181, crucible/experiment.py:150,195,232,239, crucible/checkpoint.py:21-23,69,166, crucible/cache.py:207-234 Severity: HIGH

    scenario, attack, topology, run_id, and chain_name are raw strings interpolated directly into filesystem paths with no ../absolute-path rejection. Path("/a") / "/etc/x" yields /etc/x. The cache.invalidate() path feeds an unsanitized chain_name into shutil.rmtree, making it a destructive primitive.

    path = self.workspace / "scenarios" / scenario_name / "scenario.yaml"   # scenario.py:33
path = root / "topologies" / f"{name}.yaml"                             # config.py:181
attack_dir = self.workspace / "attacks" / experiment.attack            # experiment.py:150

    Fix: Validate all name inputs against ^[A-Za-z0-9_-]+$ before constructing paths; assert resolved paths stay within their intended root.

    H2 — Secret-bearing RPC URL persisted to disk and embedded in command¶

    File: crucible/chain_manager.py:55,157 Severity: HIGH

    --fork-url <rpc_url> (often containing an Alchemy/Infura API key) is added to the anvil argv, and the full command array — including that URL — is written verbatim into the unencrypted chain meta file and printed.

    cmd.extend(["--fork-url", chain.rpc_url])
...
"command": command,   # written to meta_path as JSON, key included

    Fix: Redact the key (or the whole fork URL) when persisting/printing. Store the RPC reference by provider name, not full URL with embedded secret.

    H3 — Unvalidated tx_hash / address into RPC, generated files, and cache paths¶

    Files: crucible/extract.py:99,194-209, crucible/abi.py:80-105 Severity: HIGH

    tx_hash and address flow from the CLI straight into eth_getTransactionByHash, into generated evidence.yaml / README.md templates (template/YAML injection), and into the ABI cache filename (cache_dir / f"{address.lower()}.json" — a path component). No format validation.

    cache_path = cache_dir / f"{address.lower()}.json"   # address unvalidated -> path component

    Fix: Validate ^0x[0-9a-fA-F]{64}$ (tx) and ^0x[0-9a-fA-F]{40}$ (address) before any use.

    H4 — Agent silently fabricates success after LLM failure¶

    File: crucible/agent_loop.py:554-563 Severity: HIGH

    After retries are exhausted, _call_llm returns a synthetic AgentAction(tool="chain_id", params={}) instead of propagating the failure. The broad except Exception hides network/auth errors and the fake action keeps the agent loop executing on-chain tools as if nothing failed.

    except Exception as exc:
    last_error = str(exc)
    ...
# All attempts failed — log error and return a no-op
return AgentAction(tool="chain_id", params={})

    Fix: Propagate the failure or terminate the run with a clear error; never synthesize a tool call.

    H5 — anvil process leak / zombie children¶

    File: crucible/chain_manager.py:139-171 Severity: HIGH

    start_chain spawns anvil, writes the meta file, then calls _await_ready, which raises on timeout. The spawned process is not terminated on that exception — it keeps running and holds the port. More broadly, Popen objects are never wait()ed/reaped, producing zombies when anvil exits.

    Fix: Wrap _await_ready in try/except that kills the child on failure; track and reap child processes.

    H6 — No hard cap on LLM cost / call count¶

    Files: crucible/agent_loop.py:256,471-492, crucible/llm_providers.py:232,349 Severity: HIGH

    llm_budget_max_tokens defaults to None → no cost ceiling. Even when set, the budget relies on provider self-reported usage.total_tokens; many OpenAI-compatible / local endpoints omit usage, so the counter stays 0 and the budget never trips. The iteration cap only counts executed tools, so an LLM that emits tokens without tool calls is not reliably bounded.

    Fix: Add an unconditional LLM-call counter and wall-clock limit independent of provider-reported usage.

    4. Medium Findings¶

    # File:Line Issue
    M1 instrumentation.py:440-446 ThreadPoolExecutor.shutdown(timeout=...) is not a valid kwarg in any CPython; the call always raises TypeError and falls back to shutdown(wait=True) with no timeout → drain() can hang forever despite its 30s contract.
    M2 chain_manager.py:202 (+ rpc.py:89-91) Readiness requires block > 0; a blank (non-fork) anvil sits at block 0 and never becomes "ready," so _await_ready always times out for blank-mode chains.
    M3 enforcer.py:91-107 Fragile nested-loop/break logic in fidelity trace detection; the bridge check runs every stream iteration and a stream present only in a chain_* dir relies on a bottom-of-loop break. Needs a single membership scan and a test for chain-only traces.
    M4 publication.py:271-285 raise_on_failure classifies gate failures by substring ("evidence" in f.lower() / "provenance" in f.lower()). Mixed or keyword-colliding messages are misrouted or land in neither branch. Use structured per-check result types.
    M5 extract.py:257-267 ABI fallback abi = [v for v in data.values() if isinstance(v, list)] yields a list-of-lists, which passes the isinstance(abi, list) check and is returned as a valid ABI, silently producing garbage entries downstream.
    M6 export.py:174-195 ExportBundle.create returns the bundle object built before a failed re-verify rmtree/copytree, and an integrity failure on an existing export is silently overwritten — caller gets no signal that verification failed.
    M7 extract.py (multiple), provenance.py:39,55,71, commands/attack.py:46-48 Pervasive broad except Exception (one bare pass) swallows genuine errors: _decode_args returns [] on any failure (indistinguishable from "no args"); _register_abi resets the map to {} on read error (data loss); config load failures during --from-tx discarded silently.
    M8 scenario_steps.py:391-415, logging.py:278,489 Per-line json.loads over JSONL with no try/except; one corrupt/truncated line aborts report/verification generation. Also forge_script extra_args (M-adjacent) is taken from YAML and concatenated into argv unchecked (arg injection; TypeError if non-list).

    5. Low / Latent Findings¶

    # File:Line Issue
    L1 llm_providers.py:199,322 except urllib.error.HTTPError referenced but only import urllib.request present. Works today only because CPython's urllib.request imports urllib.error transitively — add the explicit import to remove the latent NameError risk.
    L2 rpc.py:78, abi.py:113,133 response.read() with no size cap; a hostile/misbehaving RPC or explorer can OOM the process.
    L3 hashing.py:17-19 json.dumps(..., default=str) makes the "deterministic config fingerprint" non-injective — distinct configs that stringify alike collide. Acceptable for fingerprinting, not for integrity.
    L4 agent_loop.py:186-204 parse_llm_response honors only tool_calls[0] (drops the rest) and treats any non-{"status":"done"} prose as "done," silently terminating a run on unparseable output.
    L5 env.py:13,31-39 expand_env regex [A-Z0-9_]+ misses lowercase ${var}; load_dotenv uses os.environ.setdefault, so a pre-set env var silently overrides .env.
    L6 agent_loop.py:601-613,629-642 Conversation truncation isn't role-aware and can orphan a leading role:"tool" message (no preceding tool_calls), causing a provider 400 on the next call.

    6. Positive Observations¶

    • No unsafe deserialization: YAML loading is consistently yaml.safe_load (config.py:158, scenario.py:36, baseline.py:128, plugin.py:165); no yaml.load/Loader=, no pickle, no eval/exec on untrusted data (the exec_module cases are plugin/script loading, flagged separately).
    • HTTP timeouts present on all provider calls (llm_providers.py:197,320, default 60s, configurable).
    • No hardcoded API keys; keys sourced from env / .env.
    • Clean architecture: thin CLI entry point (cli.py is 11 lines delegating to commands/), well-separated modules, readable code.
    • Substantial test suite: 49 test files, ~776 test functions, with markers separating tests that require foundry/archive-node/docker.

    7. Release Verdict & Minimum Bar¶

    Verdict: NOT ready for a 0.4.0 "production readiness" release.

    The architecture and feature surface are impressive, but there are five distinct arbitrary-code-execution / injection paths (C1, C2, C5) plus an integrity model that does not provide integrity (C4) and an open RPC proxy (C3). For a security-research testbed specifically, shipping shell=True command injection, unsandboxed script/plugin execution, and a forgeable artifact manifest contradicts the tool's own purpose — these are the exact issue classes the testbed exists to study.

    Minimum bar to tag a release:

    1. Close C1–C5:
    2. Replace shell=True hook with argv + shell=False.
    3. Sandbox / containment-check python_script & forge_script paths.
    4. Proxy: bind loopback only, method allowlist, body-size cap, guarded parse.
    5. Sign the integrity manifest or anchor its hash out-of-band.
    6. Separate plugin metadata parsing from code import; gate import behind explicit trust.
    7. Add ^[A-Za-z0-9_-]+$ validation on all name→path inputs (H1) and assert resolved paths stay within their roots.
    8. Redact secrets from meta files / printed commands (H2); validate tx_hash/address format (H3).
    9. Remove the "fake success" agent fallback (H4) and add a hard LLM call/cost ceiling (H6).

    Medium/Low items are acceptable as fast-follows. With the criticals closed and High items addressed, this becomes a credible 0.5.0.

    8. Suggested Remediation Order¶

    1. Quick, contained, high-impact: H1 path validation helper (reused everywhere), C1 hook fix, H3 hash/address validation, H2 secret redaction. (~1 focused session.)
    2. Design-touching: C3 proxy hardening, C2/C5 script & plugin trust model.
    3. Integrity: C4 manifest signing / external anchor — requires deciding on a signing story (minisign/GPG/DOI).
    4. Agent robustness: H4, H6, L4, L6.
    5. Correctness cleanup: M1–M8, then L2–L5.
