    Crucible — Documentation Verification Report¶

    Date: 2026-06-13 Reviewed version: pyproject.toml = 0.4.0 / crucible/__init__.py __version__ = 0.3.0 (mismatch — see V-LOW-1) Scope: Verification of the 51 docs under docs/ (+ README.md) against the actual source code. Every falsifiable claim — CLI commands, flags, config keys, env vars, Python API imports, scenario step types, plugin signatures, attack/topology/scenario names, fork blocks, and counts — was checked against ground truth in crucible/. Method: Static analysis only. No code executed. Findings cross-checked by direct reads of the cited source lines.

    Headline: The docs are broadly structurally accurate, but contain 12 HIGH-severity claims that will make a copy-pasted command or code example fail outright, plus a phantom CLI flag repeated across 3 docs, several wrong on-chain constants, and contradictory test counts. For a project whose value proposition is reproducibility, copy-paste-fails in the tutorial-grade docs are the most damaging class of issue.

    Severity legend¶

    • HIGH — documented command/flag/import/signature/step does not exist or is wrong → the user's command or example fails.
    • MEDIUM — misleading, outdated, or internally contradictory; won't always hard-fail but causes wrong results or confusion.
    • LOW — cosmetic / metadata drift.

    🔴 HIGH — Copy-paste will fail¶

    V-HIGH-1 — --enforce-fidelity flag does not exist (3 docs)¶

    Docs: CLI_REFERENCE.md:91,113,133 · USER_GUIDE.md:219,327 · OPERATIONS_RUNBOOK.md:126 Code: crucible/commands/experiment.py:24 defines only --no-fidelity (fidelity is on by default). There is no --enforce-fidelity option anywhere in crucible/commands/. The string survives only in a stale docstring at enforcer.py:8. Impact: Every documented crucible experiment run … --enforce-fidelity command errors out. CLI_REFERENCE's entire "Fidelity Enforcement" section (L111–134) is built around a non-existent flag. Fix: Replace with "fidelity runs by default; use --no-fidelity to skip."

    V-HIGH-2 — client.call(...)["result"] crashes (3 docs, many call sites)¶

    Docs: ATTACK_PACK_GUIDE.md:485 · CUSTOM_CONTRACT_TESTING.md:289,327,348,597 Code: crucible/rpc.py:87 — JsonRpcClient.call() returns body.get("result"), i.e. the result itself, not an envelope. So: - int(client.call("eth_getBalance", [...])["result"], 16) → TypeError: string indices must be integers - client.call("eth_getTransactionReceipt", [tx])["result"]["contractAddress"] → KeyError (the returned dict already is the receipt) Impact: The CUSTOM_CONTRACT_TESTING guide is a "copy this execute()/verify() and run it" tutorial — every copy crashes. Fix: int(client.call("eth_getBalance", [...]), 16) and client.call(...)["contractAddress"], or use typed helpers (client.get_balance(...), client.block_number()).

    V-HIGH-3 — bridge.send_message(...) example omits required context arg¶

    Doc: ATTACK_PACK_GUIDE.md:740-744 Code: crucible/sdk.py:147 — send_message(self, context, source_chain, dest_chain, data, correlation_id=None). context is the required first arg; real plugins call bridge.send_message(context=context, …) (attacks/l4_full_validation/l4_full_validation.py:163). Impact: Doc example raises TypeError: missing 1 required positional argument: 'context'.

    V-HIGH-4 — Double 0x prefix bug in deploy example¶

    Doc: CUSTOM_CONTRACT_TESTING.md:589 Code/contract reality: bytecode = artifacts["bytecode"]["object"] already includes 0x; "0x" + bytecode yields 0x0x6080…, which Anvil rejects. This is the exact bug ATTACK_PACK_GUIDE.md:414-418 warns against — the two docs contradict each other.

    V-HIGH-5 — bridge_action lists non-existent actions¶

    Doc: SCENARIO_AUTHORING.md:230 (comment: "e.g. send_message, deploy, configure") Code: crucible/scenario_steps.py:510-578 dispatches only send_message, relay_messages, check_delivery, verify_state. deploy/configure are BridgePlugin lifecycle methods, not actions → ConfigError: unsupported bridge_action: deploy.

    V-HIGH-6 — plugin scenario step documented as functional, but is a no-op¶

    Doc: SCENARIO_AUTHORING.md:295-305 (describes method:/phase: dispatch to invoke plugin logic mid-scenario) Code: crucible/scenario_steps.py:152-162 — the handler only logs status="skipped" and ignores method/phase. The documented behavior does not exist.

    🟠 MEDIUM — Misleading / contradictory / wrong constants¶

    V-MED-1 — "801 unit tests" is unsupported, and test counts contradict across docs¶

    • README.md:15 "801 unit tests"; RELEASE_CHECKLIST.md:16,28,247 "801"; RELEASE_CHECKLIST.md:128 "669"; MAINTENANCE.md:195,441 "798".
    • Ground truth: def test_ in tests/unit/ = 751 source functions + 15 @parametrize blocks (collected count not run-verified here). At minimum the four documents disagree; the literal "801" is unsupported by the source count. Treat all hard-coded counts as stale.

    V-MED-2 — BNB Token Hub fork block wrong (2 docs)¶

    OPERATIONS_RUNBOOK.md:44 and TROUBLESHOOTING.md:511 list 28,125,000. Actual is 22,166,659 (scenarios/bnb_token_hub_pre_exploit/scenario.yaml:8, testbed.yaml:29, and correctly stated in RPC_PROVIDER_GUIDE.md:246).

    V-MED-3 — "6 benchmark exploits" naming vs filesystem¶

    README.md:5 names "Wormhole, Nomad, Poly Network, Horizon, Multichain, BNB Bridge." The framing (6 named exploits) is defensible, but no pack is literally named "Multichain"/"Horizon"/"BNB Bridge" — attacks/ holds 11 dirs (multichain_router_permit, horizon_multisig_drain, bnb_token_hub_proof, plus demo/agent/validation packs). Clarify the mapping so a reader doesn't grep for a non-existent attacks/multichain.

    V-MED-4 — client.call described as an eth_call helper¶

    ATTACK_PACK_GUIDE.md:484 — call() is a generic JSON-RPC dispatcher, not an eth_call wrapper. The example works (method arg is "eth_call") but the description conflates the two.

    V-MED-5 — verify_tracing step example uses chain: instead of chains:¶

    SCENARIO_AUTHORING.md:218-220 shows chain: ethereum. Handler reads chains: (a list, scenario_steps.py); singular chain: is ignored and falls back to "all chains." Won't error, but the example key is wrong.

    V-MED-6 — compare_runs pulled from undocumented module¶

    ANALYSIS_NOTEBOOKS.md:49 imports from crucible.diff import compare_runs (valid — crucible/diff.py:86) but it sits outside the crucible.analysis API the doc is about; the doc also omits load_agent_reasoning (the 9th analysis function).

    V-MED-7 — priority: rpc-provider key is silently ignored¶

    RPC_PROVIDER_GUIDE.md:270-280, TROUBLESHOOTING.md:314-379 show priority: N under rpc_providers. RpcProviderConfig (config.py:42-46) has only name, url, archive, rate_limit_per_second. Because TestbedConfig uses pydantic's default extra="ignore", the key is accepted but does nothing — there is no priority-ordered fallback. (testbed.yaml itself also uses priority, so it's a repo-wide inconsistency.)
