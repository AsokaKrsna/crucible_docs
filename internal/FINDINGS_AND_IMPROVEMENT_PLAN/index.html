    Crucible — Comprehensive Findings & Improvement Plan¶

    Date: June 9, 2026 Audit scope: Full codebase review — architecture, modularity, correctness, test coverage, documentation, UX Test suite: 754 unit tests passing, 0 failures, 0 skipped

    Executive Summary¶

    Crucible is a well-architected EVM bridge security research testbed. The 4-layer architecture (Foundation → Runtime → Orchestration → CLI) is cleanly separated. The instrumentation proxy is the standout feature — making artifact capture automatic and universal. The plugin SDK, scenario engine, fidelity levels, and publication gates provide the academic rigor needed for USENIX-grade research.

    The codebase is production-quality in its core paths but has specific gaps in completeness and polish that would elevate it from "functional" to "exceptional."

    🎯 Goal Alignment Assessment¶

    How well does Crucible meet the user's stated goals?

    Goal Score Assessment
    "Easy like defihacklabs" 6/10 DefiHackLabs is single-click. Crucible requires RPC config → init → fork → experiment run. The CLI is scriptable but has a steeper initial setup. The --from-tx auto-extract feature bridges this gap partially.
    "Researcher just needs to sit in attacker seat" 8/10 Once configured, crucible experiment run experiments/something.yaml is indeed one command. The "sit in attacker seat" experience is good for attack plugins. Less so for agent plugins (need API keys).
    "Don't worry about artifact collection" 9/10 This is Crucible's strongest feature. The instrumentation proxy captures everything automatically. Artifact integrity hashing closes the loop. The agent_reasoning.jsonl gap is the only exception.
    "One-click reset for multi-phase experiments" 8/10 3-tier reset (soft/medium/hard) is well-designed. Soft reset (<5s via anvil_loadState) is the "one-click" experience. Hard reset (<5 min for cold cache) could use more progress feedback.
    "Granular/modular lego-like architecture" 8/10 Plugin system, step registry, and topology-driven composition are very modular. The ExperimentRunner god-object is the main anti-modularity concern.
    "Multi-chain flexibility" 7/10 Topology system supports single/dual/multi-chain declaratively. But only 1 bridge plugin exists (demo_bridge). L4 multi-chain fidelity is declared but not validated with a real bridge plugin.
    "CLI-first with optional UI wrapper" 9/10 Every action is available via CLI. The Streamlit dashboard is a complementary viewer. This is well executed.
    "Automation for repetitive tasks" 7/10 Cleanup, reset, and baseline are automated. Matrix experiments cover parameter sweeps. Missing: automated baseline monitoring, cron-based cleanup, GitHub Actions CI/CD.
    "Comprehensive structured logging for research purposes" 9/10 7 JSONL streams with per-chain isolation, SHA-256 integrity hashing, and cross-chain correlation IDs. The agent_reasoning.jsonl integrity gap is the only weakness.
    "USENIX-grade research rigor" 8/10 Fidelity levels are enforced programmatically. Publication gates (7 checks) enforce completeness. Provenance tracks everything. Missing: strict schema validation at write time (not just at export).
    "Low learning curve" 5/10 Tutorial is good but the tool has 20+ CLI commands, 4 config file types, 3 plugin types, 2 output formats, and 2 deployment modes. A new researcher faces a significant surface area.

    Overall Goal Alignment Score: 7.6/10

    The testbed is strongest where it matters most (artifact collection, modularity, multi-chain) but the initial setup and learning curve need attention for the "easy like defihacklabs" aspiration.

    🔍 Findings & Gaps¶

    1. Critical Gaps¶

    1a. ExperimentRunner has too many responsibilities¶

    File: crucible/experiment.py — 638 lines

    The ExperimentRunner class handles: - Config loading - Plugin loading (with agent/attack preference logic) - Foundry version checking - Cleanup enforcement - Evidence validation - Publication gates - Baseline verification - Run orchestration (bridge deploy → scenario → instrumentation → plugin → fidelity → manifest) - Manifest writing - Cross-chain message tracking - Export - Matrix running - Run lookup

    This is a god-object pattern. The run() method alone spans ~200 lines with try/except/finally containing multiple phases. Making changes here is risky — the tests (22 tests) provide good coverage but don't exercise every path.

    Recommendation: Extract into separate concerns: - ExperimentRunner.run() orchestrator → keep thin, delegate to phase managers - New PhaseRunner class for each phase - Move _load_plugin() to PluginRegistry - Move _validate_evidence_against_chain() to evidence.py (partially done already)

    1b. Agent reasoning logs bypass integrity hashing¶

    File: crucible/agent_loop.py, crucible/logging.py

    The 7th artifact stream (agent_reasoning.jsonl) is written directly by LLMAgentPlugin via _log_reasoning_step(), NOT through the LogEmitter system. This means: - It's NOT included in log_integrity.json hashing - It's NOT tracked in STREAM_SCHEMAS in schema.py - A corrupted agent reasoning log would go undetected

    Recommendation: Route agent reasoning through LogEmitter.emit() with a dedicated stream, or at minimum include it in write_integrity() and STREAM_SCHEMAS.

    1c. No integration tests for the agent system¶

    Files: All agent modules — agent_loop.py (480 lines), agent_tools.py (298 lines), llm_providers.py (399 lines)

    The agent system has unit tests (94 in test_llm_providers.py + agent tests) but zero integration tests that verify: - An actual LLM provider integration end-to-end - Agent tools actually interact with a running fork correctly - Agent reasoning logs are properly written and retrievable

    This matters because the agent system is the second major use case (after attack plugins).

    2. Architectural Concerns¶

    2a. The resume() path is untested¶

    File: crucible/experiment.py — resume() method (~100 lines) Tests: Zero

    The resume feature (restarting interrupted experiments from checkpoints) has no unit tests. It's a complex method that involves: - Loading checkpoint manifests - Restoring chain state from snapshots - Skipping completed phases - Writing updated manifests with resume metadata

    A regression here would silently produce incorrect results.

    2b. Bridge plugin coverage is thin¶

    Files: crucible/bridge_lifecycle.py, bridges/demo_bridge/

    Only one bridge plugin (demo_bridge) exists. The BridgePlugin ABC has 7 methods but bridge_lifecycle.py only exercises deploy and destroy. The send_message, configure, connect, and reset methods are never tested in integration.

    Recommendation: Implement at least one real bridge plugin (e.g., Nomad's xApp or Wormhole's core bridge) to validate the ABC design before publishing. This is important because the multi-chain L4 fidelity claim depends on bridge plugins working correctly.

    2c. No hermetic end-to-end test with real fork¶

    File: tests/e2e/test_hermetic_nomad.py

    The E2E test exists but uses a JSONL fixture (recorded RPC calls via ReplayServer from crucible/recording.py) instead of actually forking from an archive node. This is fine for CI but means the test doesn't validate the full pipeline: - Config loading → chain forking → scenario execution → instrumentation → artifact capture

    A true hermetic E2E test would: 1. Start an Anvil fork at a known block (using a cached state fixture) 2. Run an experiment end-to-end 3. Verify all 7 artifact streams exist with expected content 4. Assert manifest fields are correct 5. Export and verify the bundle

    Note: crucible/recording.py (ReplayServer + record_fixture) is well-implemented and provides good infrastructure for this. The gap is in using it for a full pipeline test rather than just replaying RPC calls.

    3. Polishing Opportunities¶

    3a. CLI output still has silent gaps¶

    Addressed partially (progress spinners in library modules), but: - crucible resume has no progress indicators during chain state restoration - crucible export does not validate the bundle until after creation — user waits for archive creation before finding errors - crucible baseline verify doesn't show per-chain progress when checking multiple chains

    3b. Docker compose health checks missing¶

    File: docker-compose.yml

    The docker-compose configuration has depends_on but no healthcheck directives. If an Anvil daemon starts slowly, the orchestrator will try to connect before it's ready. The _await_ready method handles this at the Python level, but Docker Compose would show no status of the delay.

    3c. No CONTRIBUTING.md or development setup guide¶

    While documentation is extensive for users, there's no concise guide for new contributors on: - Setting up a development environment - Running tests - Code style conventions (Ruff config exists but isn't documented) - How to add a new step type to the scenario engine - How to add a new CLI command

    3d. pyproject.toml had inconsistent dependency grouping — FIXED¶

    pandas and streamlit were in both [project.dependencies] and [project.optional-dependencies]. Now they only appear in main deps (which is correct since the dashboard imports them at module level). The redundant entries in optional groups (analysis, ui) have been removed.

    3e. agent_reasoning.jsonl stream not in schema¶

    File: crucible/schema.py

    The STREAM_SCHEMAS dict has 6 streams but not agent_reasoning.jsonl. This means: - The schema validator won't catch corruption in agent reasoning logs - Export bundle validation skips it - It's an invisible gap for agent-based experiments

    4. Additional Module Findings¶

    4a. crucible/analysis.py — Public API surface is adequate but undocumented¶

    File: crucible/analysis.py — 105 lines

    Provides 7 public functions for loading experiment data into pandas DataFrames: - find_run(), load_run(), list_runs() — run discovery - load_events(), load_gas(), load_storage_diffs(), load_balance_diffs(), load_cross_chain() — artifact loading

    Gaps: - No load_agent_reasoning() function (consistent with the integrity gap) - No load_balance_diffs() documented in the module docstring (it exists but is missing from the docstring's "Available functions" list) - No type annotations for returned DataFrames (columns vary, but a documented minimum schema would help) - The module's _load_jsonl_df() helper loads ALL entries — for large runs with thousands of storage diffs, this could be memory-intensive. Consider adding pagination or filter parameters.

    Recommendation: Add documented column schemas for each DataFrame, add load_agent_reasoning(), and consider lazy-loading or chunked reading for large artifact streams.

    4b. crucible/interfaces/__init__.py — Public API is thin¶

    File: crucible/interfaces/__init__.py — 4 lines

    Re-exports only 4 classes: AttackPlugin, BridgePlugin, CruciblePlugin, ExperimentContext.

    Gaps: - Missing: VerificationResult, FidelityResult, GateResult — useful for plugin authors writing custom verification - Missing: ChainRuntime, JsonRpcClient — plugins frequently need these - Missing: AgentPlugin — not re-exported despite being a first-class plugin type - Missing: LogEmitter, LogManager — for plugins that need advanced logging

    Currently, plugin authors must import from crucible.sdk, crucible.rpc, crucible.chain_manager directly. A richer interfaces/__init__.py would make the SDK surface clear and stable.

    Recommendation: Expand the public API to include all types a plugin author needs (e.g., VerificationResult, FidelityResult, ChainRuntime, JsonRpcClient, AgentPlugin, LogEmitter) and document the stable API surface.

    4c. crucible/recording.py — Well-implemented infrastructure¶

    File: crucible/recording.py — 175 lines

    Provides ReplayServer and record_fixture for hermetic CI testing. The implementation is clean: - Thread-based HTTP server with daemon thread - Fixture lookup via canonical method+params key - Helpful error message when fixture entry is not found (lists available methods) - Suppressed default logging

    No issues found. The module is well-designed and serves its purpose. It's used by tests/e2e/test_hermetic_nomad.py.

    💡 Out-of-the-Box Ideas¶

    1. Visual Diff Dashboard¶

    Currently crucible diff outputs text. A Streamlit-based visual diff comparing two runs side-by-side (storage slots, event timelines, balance flows) would dramatically improve the researcher experience. Think: "Git diff but for blockchain experiments."

    2. Attack Pack Registry / Marketplace¶

    A discoverability tool: crucible attack search "bridge" that lists known attack packs from a public registry (GitHub-based or simple YAML index). Researchers could share reproductions like packages.

    3. REPL Mode for Interactive Experimentation¶

    crucible shell — an interactive Python REPL with: - Pre-loaded context, chains, rpc objects - Tab completion for chain names, RPC methods - Auto-instrumentation of every transaction - History persistence - "Record → Export" workflow

    4. Automated Baseline Drift Detection¶

    Currently baselines capture state at a point in time. A crucible baseline monitor command that periodically checks for baseline drift would alert researchers when: - Their local modifications have drifted from the golden state - RPC provider changes have caused block differences - Cache corruption has occurred

    5. Fuzzing Engine Integration¶

    Plug in a fuzzer (like Echidna or Medusa) as a step type: 

    - type: fuzz
  contract: "0x..."
  corpus: [tx1, tx2]
  duration_seconds: 300

    This would make Crucible not just a reproduction tool but a vulnerability discovery tool.

    6. Self-Contained Experiment Snapshots¶

    Export experiments as a single portable .crucible file containing: - The experiment YAML - The chain state snapshot (compressed JSON or Anvil cache) - The attack plugin code - The scenario YAML

    Share via: crucible experiment share DS-2026-00042 → nomad_repro.crucible Replay via: crucible experiment run nomad_repro.crucible

    No RPC needed, no git clone needed. One file, one command.

    7. USENIX Artifact Packaging Shortcut¶

    A single command that outputs a usenix-artifact.zip ready for submission: 

    crucible publish --package usenix --run DS-2026-00042 --output usenix-artifact.zip

    It would auto-generate: - README.md with submission instructions - Dockerfile (if using Docker mode) - runs/ directory with run manifest + all artifacts - BIBLIOGRAPHY.md with citations from evidence.yaml - CHECKLIST.md with the AE checklist

    8. Interactive Scenario Builder¶

    crucible scenario init with an interactive wizard that: - Asks which chains to use - Asks what setup steps are needed (impersonate, send_tx, assert) - Generates the scenario YAML with proper structure - Opens it in the default editor

    This would lower the learning curve significantly — currently, scenario authoring requires knowing the YAML schema by heart.

    • test_chain_manager.py is thorough — 49 tests covering all process safety paths
    • test_monitoring.py tests are good but depended on pandas at import time (now fixed by installing pandas) [ok]

    Performance¶

    • _trace_transaction in instrumentation.py has retry + backoff — good
    • CacheManager.fingerprint() re-reads all cache state — could be slow with large caches
    • ExperimentRunner.run() scans all run.jsonl files in the finally block — this is O(files) which is fine for typical runs
    • analysis.py _load_jsonl_df() loads ALL entries into memory — for runs with millions of storage diffs, this could be an issue
