    Crucible — Final Pre-Release Scan (Independent Deep Review)¶

    Prepared for: Engineering team Date: 2026-06-15 Reviewed at: working tree on top of c7f48f5 (includes the in-flight SI-7/SI-10 follow-up fixes to crucible/logging.py and crucible/chain_manager.py). Method: Static analysis only — no code executed. Two independent adversarial sweeps (release-blocker correctness; SI-7/SI-10 fix stress-test) plus line-by-line verification of every finding against source. Findings the sweeps raised were not taken on trust — each was confirmed or rejected against the actual code, and the rejections are documented below so the team can see what was checked.

    Executive summary¶

    The security and scientific-integrity work from prior rounds holds. This final scan went looking for anything those rounds missed and surfaced one genuine release blocker plus four worth-fixing robustness issues. It also caught two false alarms from the automated sweeps, which were disproven against source.

    🔴 Release blocker¶

    B1 — A crash in fidelity enforcement leaves the run marked success with no fidelity record¶

    File: crucible/experiment.py:468-470 (with :431, :459, :491)

    except Exception as enf_exc:
    fidelity_failures = [f"Fidelity check error: {enf_exc}"]
    logger.warning("Fidelity enforcement check failed: %s", enf_exc)

    When FidelityEnforcer.check() raises any unexpected exception (malformed evidence.yaml, a KeyError/IndexError in a check helper, an I/O error reading run.jsonl, etc.): - fidelity_passed stays None (initialised at :431, never reassigned because the exception fires before :449). - outcome stays "success" — the only downgrade to "failure" is at :459, inside the if not result.passed branch, which is never reached. - At :491, if fidelity_passed is not None: is False, so no fidelity_check block is written to the manifest at all, and the captured fidelity_failures error string is silently dropped.

    Impact: a run whose fidelity verification crashed is published with outcome: "success" and a manifest that looks like fidelity was simply never requested. For a USENIX artifact this is the worst case — a claimed-reproduced result that was never actually verified, with no trace in the artifact. Same family as SI-8 (success-but-captured-nothing) and SI-11 (advisory gates).

    Fix: in the except, set fidelity_passed = False and outcome = "failure" (and an appropriate non-zero exit_code) so the crash is recorded and the manifest reflects it. Add a regression test that forces check() to raise and asserts outcome == "failure" and a persisted fidelity_check.

    🟠 Recommended before release¶

    B2 — A failed reproduction (AttackResult.success == False) can publish as outcome: success¶

    Files: crucible/sdk.py:77-81 (AttackPlugin.run), :99-101 (default verify), :34; crucible/experiment.py:388-393, :444; crucible/enforcer.py:60-63

    AttackPlugin.run() does execute() → verify() and raises only when verify_result.passed is False:

    result = self.execute(context)          # result.success may be False
verify_result = self.verify(context)
if not verify_result.passed:
    raise AssertionError(...)            # the ONLY failure signal
return result                            # result.success is never checked

    Two gaps compound: 1. result.success is never gated. execute() returns AttackResult(success: bool) — the plugin's own determination of whether the exploit reproduced. run() ignores it; experiment.py only console-prints it (:389) and passes it to the enforcer (:444), where _check_result_available checks only is None (enforcer.py:62), not .success. It is not persisted to the manifest. 2. Default verify() always passes — VerificationResult(passed=True) (sdk.py:101). A plugin that doesn't override verify() with real expected_state assertions never raises.

    Impact: an attack plugin whose execute() returns success=False (the exploit did not reproduce) but which relies on the default verify() will have its run recorded as outcome: "success", pass fidelity (L2 only checks that artifact streams exist), and be exportable — with the success=False self-report discarded entirely. A negative result is published as a positive one. This is the same false-success family as B1 and SI-8, and it's unsafe-by-default (the path of least resistance for a plugin author is the unsafe one).

    Severity: MEDIUM — depends on plugins defining real verify() assertions, but the default behavior and the silent discard of success=False make it a genuine scientific-honesty gap.

    Fix: in run(), also fail or explicitly record when result.success is False (e.g. raise unless the plugin opted into "negative result expected"); and persist attack_result.success / attack_result.error into run_manifest.json so the exploit outcome is always visible regardless of gating. Consider making the base verify() return passed=False (or abstract) so "no assertions" is not silently "passed."

    C1 — Cache invalidation ignores a customized fork_cache.path; invalidate() can be a no-op against anvil's real cache¶

    Files: crucible/cache.py:79,216 vs crucible/chain_manager.py:86; default crucible/config.py:52

    CacheManager.__init__ hardcodes self._cache_base = self.workspace / ".fork_cache", and CacheManager is constructed with only (workspace, policy) — it never receives config.fork_cache.path. Meanwhile anvil's --cache-path is self.workspace / self.config.fork_cache.path / name / str(fork_block or "latest").

    • Default config is safe: ForkCacheConfig.path defaults to ".fork_cache", so the two coincide and invalidate() (shutil.rmtree) does delete anvil's cache.
    • Customized path breaks it: if a user sets fork_cache.path to anything else in testbed.yaml, CacheManager targets .fork_cache/... while anvil reads/writes <custom>/.... Verify, invalidate, and the manifest all operate on the wrong tree → the SI-10 strict-policy invalidation silently does nothing and stale forked state is reused.
    • Edge: for fork_block == 0, anvil uses str(0 or "latest") → "latest" while CacheManager uses "0" — directory mismatch even under default config.

    Fix: pass config.fork_cache.path into CacheManager and derive _cache_base from it (or assert it equals .fork_cache). Use a consistent fork_block/"latest" key on both sides.

    M4 — Instrumentation teardown is not exception-safe; one proxy failure leaks the rest and can skip artifact finalization¶

    File: crucible/chain_manager.py:406-409 (called from experiment.py:404/:664, both inside finally)

    for name, proxy in self._proxies.items():
    proxy.drain()
    proxy.stop()
self._proxies.clear()

    No per-proxy try/except. If drain()/stop() raises for one chain, chains 2..N are never drained or stopped — their HTTPServer threads and ThreadPoolExecutors leak for the process lifetime. Worse, because this runs in the experiment finally before write_report/write_integrity/write_artifact_manifest (experiment.py:508-516), an exception here both masks the original experiment exception and skips all artifact finalization (no log_integrity.json, no bundle manifest). Most relevant to the multi-chain L4 bridge experiments the tool targets.

    Fix: wrap each proxy's drain()/stop() in its own try/except that logs and continues, so every proxy is torn down and finalization always runs.

    M5 — crucible verify <archive> extracts tarballs without a safe filter (path traversal on Python 3.11/3.12)¶

    File: crucible/commands/verify.py:43,46

    with tarfile.open(path, "r:gz") as tar:
    tar.extractall(tmp)

    extractall is called with no filter= argument. The project supports python >=3.11 (pyproject.toml); on 3.11/3.12 the default tar extraction filter is the legacy unsafe behavior, so a crafted bundle with ../ members, absolute paths, or symlinks can write outside the temp directory. crucible verify is explicitly meant to validate third-party bundles from reviewers — i.e. untrusted input. (On 3.14 the default is already 'data', so current dev machines are incidentally safe, but shipped support includes 3.11/3.12.) The .zip path via shutil.unpack_archive is safe — zipfile sanitizes traversal.

    Fix: pass filter="data" to both extractall calls (available 3.9+).

    🟡 Polish / hardening¶

    D1 — SI-10 manifest docstring overstates what readiness proves¶

    File: crucible/chain_manager.py:281-285

    The new comment says the manifest "attests that a successfully populated cache exists." But _await_ready (:333) only polls eth_blockNumber, which anvil answers as soon as its RPC server is up — before it has lazily fetched/cached forked state (anvil caches on-demand per call). The manifest reliably attests config/provenance identity, not cache completeness. Soften the wording (or add an actual populated-cache check if completeness attestation is desired).

    D2 — SI-7 byte-identical guarantee is intra-platform only¶

    File: crucible/logging.py (emit open() and canonicalize_stream's write_text)

    Log handles are opened without newline=, so Windows writes \r\n and Linux \n; sha256_file hashes raw bytes. Two identical runs on the same OS are byte-identical (the SI-7 goal — met). Two runs on different OSes are not, and a bundle produced on Windows then verified on Linux (or via the verify extract path) could mismatch on line endings. If "reproducible across machines" is a publication claim, pin newline="\n" on the emit open() and on write_text.

    ⚪ Open robustness item (deferred, non-blocking)¶

    D4 — External HTTP responses are read without a size cap¶

    Files: crucible/rpc.py:78, crucible/abi.py:115,136, crucible/llm_providers.py:199,322

    json.loads(response.read().decode("utf-8")) reads the full body with no maximum. A malicious or misbehaving archive RPC / block-explorer API / LLM endpoint could return a huge body and exhaust memory. Risk is bounded in practice because these endpoints are operator-configured (you point Crucible at your own RPC/Etherscan/OpenAI), which is why this was deferred in earlier rounds — but it's the one remaining defense-in-depth gap. Cap with a bounded response.read(MAX_BYTES) and reject oversized bodies (mirrors the proxy's existing _MAX_CONTENT_LENGTH guard).

    ✅ Raised by automated sweeps, then disproven against source¶

    These are documented so the team can see they were investigated, not skipped.

    • H2 — "a one-sided / coincidentally-empty stream can still report *_equal: true." False. In crucible/diff.py, _check_empty marks a stream incomparable only when both sides are empty, yielding verdict None — not True. A one-empty/one-full case compares [] == [x] → False. There is no path that produces a false true from an empty capture.
    • H3 — "balance-diff equality can be a false negative due to unstable sort." Neutralised by the SI-7 fix. balance_diffs.jsonl is now full-line canonically sorted on disk before diff reads it, and Python's stable sort preserves that order for ties → identical content compares equal across runs. Storage equality additionally compares the full dict including before, so no false-equal either.

    What was checked and found clean¶

    • Assertion → run outcome is sound. Scenario assert steps raise AssertionFailed (scenario_steps.py:145,240,242,515); scenario.run → _run_step → handle_step has no swallowing try/except, so a failed assertion propagates to experiment.py:396 → outcome = "failure". (The separate B2 gap is about AttackResult.success, not scenario assertions.)
    • Provenance fail-open is properly gated. git_commit == "unknown" fails check_provenance (publication.py:143-152) and check_git_provenance (:198-216); a dirty tree warns. Per the SI-11 fix these gates block in publication mode. Outside publication mode an unknown/dirty manifest is allowed by design.
    • Dataset-counter race is not reachable. next_dataset_id (ids.py) has a documented TOCTOU window, but run_matrix executes experiments sequentially (matrix.py:51, plain for loop, no thread/process pool), so duplicate IDs can't occur as shipped. Worth a file lock only if parallel matrix execution is ever added.
    • External-hash anchor is consumed (old N3 closed). verify_bundle reads <bundle>.sha256 from the parent dir and folds external_hash_valid into the verdict (export.py:278-302), so it no longer trusts only the in-bundle manifest.
    • No dangerous execution patterns. Repo-wide grep: no shell=True, eval(, exec(, pickle.load, or unsafe yaml.load. The only extractall is M5.
    • SI-7 fix (stream canonicalization) is sound: canonicalize-then-hash is idempotent (no line drop/dup; unparseable lines preserved), all trace workers are drained (drain() → executor.shutdown(wait=True), plus the SI-9 accept-flag) before canonicalization, and verify re-hashes the same canonicalized bytes. Sorting also repairs latent order-sensitivity in the gas/events diff comparison. Covered by new tests in tests/unit/test_logging.py.
    • SI-10 fix (manifest write moved after readiness) is sound: a crashed/timed-out fork now leaves no manifest, so the next run rebuilds (subject to the C1 path caveat above).
    • All 13 scientific-integrity findings (SI-1…SI-13) remain closed — see SCIENTIFIC_INTEGRITY_AUDIT.md.

    Resolved action items¶

    Items 1–6 are now fixed and verified (805/805 unit tests pass, code review clean):

    1. B1 ✅ — fidelity crash now sets fidelity_passed=False, outcome="failure", exit_code=ASSERTION_FAILED
    2. B2 ✅ — AttackResult.success==False gated (outcome→failure); persisted to manifest
    3. C1 ✅ — CacheManager accepts cache_base= parameter; all call sites pass config.fork_cache.path; fork_block==0 key fixed
    4. M4 ✅ — per-proxy try/except in stop_instrumentation; use list(self._proxies.keys()) for safe iteration
    5. M5 ✅ — filter="data" on both tar.extractall() calls
    6. D1 / D2 ✅ — docstring softened; newline="\n" pinned for cross-platform byte-identical artifacts

    D4 (HTTP read size cap) remains deferred as non-blocking — risk is bounded because all endpoints are operator-configured.

    At this point the release candidate is credible; the scientific-integrity, security, and correctness posture is solid.
