    Architectural Review of Crucible Strategic Plan¶

    Reviewer perspective: Systems architect + USENIX Security program committee Date: June 8, 2026

    1. The Threat Model Is Implicit — It Needs to Be Explicit¶

    The strategic plan says Crucible is "infrastructure, not a framework" and that the testbed "proves" fork state matches claims. But the document never states what Crucible actually guarantees and what it doesn't.

    Every research instrument needs a clear threat model. A USENIX reviewer will ask: "What can go wrong between your fork and the real chain, and how do you know your reproduction is faithful?"

    What Crucible Actually Guarantees (EVM-level fidelity)¶

    • [ok] State fidelity: The fork starts with identical storage, balance, and code as the real chain at block N (because Anvil fetches it from the archive node)
    • [ok] Execution fidelity: The EVM opcodes execute identically (Anvil uses the same EVM as Ethereum clients)
    • [ok] Gas accounting: Gas is metered identically (same opcode costs)
    • [ok] Event emission: Events are emitted from the same code paths

    What Crucible Does NOT Guarantee (protocol-level fidelity)¶

    • [fail] Mempool dynamics: Anvil auto-mines with FIFO ordering. Real attacks may depend on MEV, front-running, or specific block inclusion order
    • [fail] Block timing: timestamp: fork_block freezes time. Real attacks may depend on block.timestamp changes across blocks
    • [fail] Cross-chain relay timing: The bridge plugin is a synchronous mock. Real bridge exploits depend on relay latency, guardian quorum timing, and message ordering
    • [fail] Consensus behavior: Anvil is a single-node devnet. Attacks that depend on reorgs, uncle blocks, or finality delays are not reproducible
    • [fail] External oracle state: Chainlink/TWAP oracles at the fork block reflect historical prices, not the prices that existed during the actual exploit window

    This matters. The Wormhole exploit depended on guardian set validation. The Nomad exploit depended on a specific state of the Replica contract. The Poly Network exploit depended on cross-chain relay behavior. Crucible can reproduce the EVM-side effects of these attacks, but it cannot prove the full attack path was as described.

    Recommendation¶

    Add a section to the strategic plan (and eventually to publication bundles) called "Fidelity Boundary" that explicitly states:

    "Crucible reproduces the EVM execution of bridge exploits. It does not reproduce mempool ordering, cross-chain relay timing, consensus behavior, or external oracle dynamics. The fidelity level (L0–L4) describes how much of the original attack's EVM behavior is replayed, not the completeness of the end-to-end attack path."

    This isn't a weakness — it's scientific honesty. A USENIX reviewer will respect this. They will NOT respect a tool that claims "faithful reproduction" without qualifying what "faithful" means.

    2. The Instrumentation Proxy Has an Observability Boundary the Plan Doesn't Acknowledge¶

    The plan's core claim is: "the researcher writes zero logging code — every transaction is auto-traced." This is true if and only if all transactions flow through the proxy.

    Looking at the code (crucible/instrumentation.py#L160-L174), the proxy intercepts eth_sendTransaction and eth_sendRawTransaction. But:

    What the Proxy Captures¶

    Method Captured How
    eth_sendTransaction via proxy [ok] do_POST → _schedule_trace
    eth_sendRawTransaction via proxy [ok] Same path
    Read calls (eth_call, eth_getBalance) [fail] Forwarded, not traced
    Transactions sent by forge script --broadcast via proxy [ok] Forge uses the proxy as its RPC
    Transactions sent by cast send via proxy [ok] Same

    What the Proxy CANNOT Capture¶

    Case Why
    Plugin opens a direct connection to Anvil (bypassing proxy) Plugin has the chain config, could connect directly
    Forge script uses vm.startBroadcast() with a hardcoded RPC URL Forge talks directly to Anvil
    Plugin uses a different web3 library with its own connection pool Connection won't go through the HTTPServer proxy
    Internal Anvil state changes (auto-impersonate, cheatcodes) These happen inside Anvil, not via RPC

    The proxy's transparent URL substitution (runtime_for() returns proxy URL) makes the common case work. But there's no enforcement mechanism — a plugin CAN bypass it. The plan should acknowledge this and the artifacts_complete flag (item 1) should also check whether the transaction count in gas_profile.jsonl matches the transaction count in Anvil's block range.

    Recommendation¶

    Add a post-run consistency check:

    expected_tx_count = eth_getBlockByNumber(latest).transactions.length
                  - eth_getBlockByNumber(fork_block).transactions.length
actual_traced     = count(gas_profile.jsonl entries)
if expected_tx_count != actual_traced:
    artifacts_complete = false
    warning: "N transactions were not captured by instrumentation"

    This closes the observability gap without requiring proxy enforcement.

    3. The Bridge Simulation Model Is Architecturally Weak — and the Plan Doesn't Address It¶

    This is the most significant architectural concern for a USENIX paper on bridge security.

    Looking at the BridgePlugin interface (crucible/sdk.py#L106-L167):

    class BridgePlugin(CruciblePlugin):
    def deploy(self, context) -> dict[str, str]: ...
    def send_message(self, context, source_chain, dest_chain, data, correlation_id) -> dict: ...

    And the default send_message() implementation:

    def send_message(self, context, source_chain, dest_chain, data, correlation_id=None):
    context.log.step(f"bridge send_message {source_chain} -> {dest_chain}", ...)
    return {"source_tx_hash": "0x", "correlation_id": correlation_id or ""}

    The default bridge is a no-op stub that logs a message and returns. A real bridge plugin must be written by the researcher to simulate the actual bridge protocol. This means:

    1. The bridge behavior is whatever the researcher codes, not a replay of the real bridge
    2. There is no validation that the bridge plugin's behavior matches the real protocol
    3. The send_message() → correlation_id tracking is just bookkeeping, not protocol simulation

    For the validated attacks (Nomad, Wormhole, Poly Network), this doesn't matter because the exploits happen on a single chain — the bridge is attacked from the Ethereum side only, and the cross-chain relay is irrelevant to the EVM execution. But the strategic plan says Crucible is for "bridge security research" — so the bridge model matters.

    The Problem for a Reviewer¶

    A USENIX reviewer will ask: "You say you reproduce bridge exploits. But your bridge is a mock. How do you know the mock is faithful to the real protocol?"

    For L2/L3 fidelity (single-chain fork), this question doesn't apply — the reviewer will accept it. For L4 fidelity (multi-chain + bridge), the question is fatal unless you can show the bridge plugin is validated against the real protocol.

    Recommendation¶

    The plan should explicitly state that bridge plugins at L4 fidelity require: 1. Protocol specification reference — which bridge protocol is being simulated (Wormhole VAA format, Nomad Merkle proof format, etc.) 2. Behavioral assertions — the evidence.yaml should specify expected bridge behavior (e.g., "message is relayed within 1 block," "guardian quorum is 13/19") 3. Excluded components declaration — this already exists in L4 but should be expanded to include WHY each component is excluded and what impact the exclusion has on fidelity

    This doesn't require code changes — it requires the evidence model to be more rigorous about what the bridge mock does and doesn't simulate.

    4. The Diff Engine Proves Structural Equality, Not Semantic Equivalence¶

    diff.py's compare_runs() (crucible/diff.py#L86-L104) compares: - Storage diffs (sorted by chain, contract, slot) - Events (sorted by chain, address, event_name, args) - Gas totals - Balance diffs

    Two runs are "deterministic" if all four match. But this is structural equality, not semantic equivalence.

    Why This Matters¶

    Consider two runs of the same Nomad exploit: - Run A: attacker sends process(fraudulent_message) at fork_block + 1 - Run B: attacker sends process(fraudulent_message) at fork_block + 1

    Both produce identical storage diffs, events, gas, and balances. The diff engine says "deterministic." [ok]

    Now consider: - Run A: attacker sends process(message_A) — drains 100 ETH - Run B: attacker sends process(message_B) — drains 100 ETH, different message but same effect

    Both produce the same balance diffs (100 ETH drained) but different storage diffs (different message hash in Replica's messages mapping). The diff engine catches this. [ok]

    But: - Run A: attacker drains 100 ETH via one tx - Run B: attacker drains 50 + 50 ETH via two txs

    Different gas totals, different event counts, different storage diffs — but semantically the same exploit. The diff engine says "not deterministic." [fail]

    Recommendation¶

    The diff engine is fine for its intended purpose (proving determinism of identical runs). But the plan should not overload it for crucible verify (item 6). Verification should compare artifact hashes (from log_integrity.json), not semantic equivalence. The plan already implies this, but it should be explicit: verify = hash-compare, not diff-compare.

    5. The Fidelity Level System Needs Formal Definitions¶

    The current fidelity levels (crucible/evidence.py) are:

    Level Name Enforcement
    L0 Narrative only No code required
    L1 Conceptual No additional checks
    L2 Fork-state behavioral Must have chains + affected_contracts
    L3 Exact transaction replay Must have exploit_transactions
    L4 Multi-chain full Must have excluded_components

    The problem: These levels describe what metadata is present, not what behavior is demonstrated. An L3 attack that lists exploit_transactions in evidence.yaml but doesn't actually replay them is still classified as L3. There's no runtime check that the declared fidelity matches the actual execution.

    What a USENIX Reviewer Expects¶

    A fidelity taxonomy should define: 1. What was replayed — the exact transactions, the contract deployment, or just the vulnerability concept 2. What was compared — the attack's effect vs. the real chain's effect (storage/balance/event delta comparison against mainnet receipts) 3. What was excluded — and why the exclusion doesn't invalidate the reproduction

    The current system has #1 (in evidence.yaml) and #3 (in excluded_components). It's missing #2 — there's no comparison of reproduction results against real chain data.

    Recommendation¶

    For L3 fidelity, the evidence pre-flight check (plan item 3) should also pull the real transaction receipt from the archive node and compare: - Real gas used vs. reproduction gas used - Real event count vs. reproduction event count - Real balance change vs. reproduction balance change

    This turns "we replayed the tx" into "we replayed the tx and got the same result." That's the difference between L3-claimed and L3-proven.

    6. Cross-Chain Timing Model Is Undefined¶

    The plan lists cross_chain.jsonl as an artifact stream. But looking at the code, cross-chain correlation is just ID tracking — there's no timing model.

    In a real bridge exploit: 1. Attacker sends tx on Chain A at block N 2. Relayer picks up the message after M blocks (finality) 3. Relayer submits proof on Chain B at block N+M+K 4. The timing relationship (M, K) is part of the exploit

    In Crucible: 1. Plugin calls send_message() on Chain A — synchronous, instant 2. Plugin calls a tx on Chain B — synchronous, instant 3. cross_chain.jsonl records correlation IDs but not timing

    There's no model of relay delay, finality, or message ordering. For L4 fidelity, this matters because some bridge exploits specifically depend on timing (e.g., racing the relayer, exploiting a finality gap).

    Recommendation¶

    For the current scope (L2/L3, single-chain), this doesn't matter. But the plan should note:

    "Cross-chain timing is not simulated. Both chains are local Anvil instances with instant mining. Exploits that depend on relay latency, finality delays, or message ordering race conditions are not reproducible at L4 fidelity without a custom BridgePlugin that simulates timing."

    Overall Architectural Verdict¶

    The plan is sound for what Crucible actually does — single-chain EVM-level exploit reproduction with automatic artifact capture. The architecture is clean, the plugin model is expressive, and the artifact pipeline is above the bar for academic use.

    The gaps are all in the same direction: the plan's language sometimes implies more than the architecture delivers. Phrases like "bridge security research testbed," "the testbed proves it," and "faithful reproduction" need qualification.

    A USENIX reviewer will accept Crucible if the paper says: "We reproduce the EVM execution of N historical bridge exploits and verify determinism of the reproduction." They will reject it if the paper says: "We faithfully reproduce bridge exploits" — because "faithful" implies protocol-level, consensus-level, and timing-level fidelity that Crucible doesn't provide.

    The Fix Is Documentation, Not Code¶

    None of the above requires architecture changes. It requires:

    1. A Fidelity Boundary section in the plan and in publication bundles
    2. Formal definitions for L0–L4 that describe behavior, not just metadata
    3. Explicit bridge simulation limitations in evidence.yaml at L4
    4. A post-run tx-count consistency check for the artifacts_complete flag

    These are all small additions that dramatically increase scientific credibility. The architecture is ready. The claims just need to match it precisely.
