    Crucible — EVM Bridge Security Research Testbed¶

    Project Summary¶

    Crucible is a CLI-based research testbed for deterministic reproduction of historical EVM bridge exploits. It forks real chains at specific blocks using Foundry's Anvil, runs attack plugins against the forked state, and automatically captures publication-grade artifacts.

    Total project scope: ~15,000 lines of Python across 39 core modules, 10 attack/bridge plugins, 6 benchmark experiments, and 29 documentation files.

    What Was Built¶

    1. Chain Forking & Lifecycle Management¶

    The testbed starts and manages Anvil subprocesses for each chain in a topology. It supports 9 topology configurations (single chain, dual chain, bridge topologies, etc.) and a 3-tier reset system:

    Tier Speed Method
    Soft <5s anvil_loadState snapshot restore
    Medium <30s Kill + relaunch + restore
    Hard <5min Re-fork from archive RPC

    Each fork uses deterministic flags: fixed gas price (0), FIFO transaction ordering, pinned fork-block timestamp, and auto-mining.

    2. 6 Reproduced Bridge Exploits ($1.9B+ Total)¶

    All 6 benchmark attacks are complete at their documented fidelity levels:

    Attack Chain Damage Fidelity What It Proves
    Wormhole Guardian Bypass Ethereum $326M L2 Forged VAA accepted via completeTransfer()
    Nomad Replica Fraud Ethereum $190M L2 process() accepts forged message when committedRoot == 0
    Poly Network PrivKey Ethereum $611M L2 Hash collision replaces bridge keeper
    Horizon Multisig Drain Ethereum $100M L2 3 bridge contracts verified, multisig flow demonstrated
    BNB Token Hub Proof BSC $570M L2 IAVL proof bypass mechanism documented
    Multichain Router Permit Multi $1.4B L1 Correctly classified as MPC infrastructure compromise

    4 of 5 replayable experiments verified deterministic via two-run diff (storage, events, gas, balances all equal).

    3. Auto-Instrumentation RPC Proxy¶

    The architectural centerpiece — a transparent HTTP proxy sits between the attack plugin and Anvil, auto-capturing:

    • Gas usage (per transaction from receipt)
    • Contract events (decoded from receipt logs)
    • Storage diffs (before/after via debug_traceTransaction)
    • Balance changes (ETH/token pre/post state)

    Zero instrumentation code in attack plugins. Works transparently with Python, Forge scripts, or cast commands.

    4. Plugin SDK (3 Types)¶

    Plugin Type Purpose Count
    AttackPlugin Exploit reproduction 8 packs
    BridgePlugin Cross-chain bridge simulation 4 implementations
    AgentPlugin LLM-driven autonomous agents 2 packs

    Plus an LLM agent system with OpenAI, Anthropic, and Ollama providers, 9 agent tools across 3 privilege levels, and full reasoning chain logging.

    5. Publication Pipeline¶

    Every experiment run produces:

    • 6 JSONL artifact streams (run log, events, storage diffs, gas profile, balance diffs, cross-chain messages)
    • run_manifest.json — full provenance (git commit, Foundry version, Python version, config fingerprint)
    • log_integrity.json — SHA-256 hashes of all streams
    • report.md — human-readable summary
    • Export bundles — self-contained directories with integrity manifests for publication

    7 publication gates enforce evidence completeness, baseline verification, Foundry version pinning, and artifact completeness.

    6. Artifact Protection¶

    crucible clean --all never touches logs or datasets. Tiered retention (scratch/saved/published) and .protected markers prevent accidental deletion. Datasets survive container restarts via bind mount.

    Architecture¶

    CLI (crucible/cli.py → commands/)
  ├── Chain Manager — Anvil lifecycle, port management, health polling
  ├── Scenario Engine — YAML-based phase execution (11 step types)
  ├── Experiment Runner — formal run lifecycle, manifest writing
  ├── Plugin Registry — discovery, validation, dynamic loading
  ├── Instrumentation Proxy — transparent RPC capture
  ├── Log Manager — 6 JSONL stream writers + integrity hashing
  ├── Fidelity Enforcer — L0-L4 runtime verification
  └── Publication Gates — 7 checks before export

    Development History¶

    The project went through 3 major phases:

    Phase 1: Platform Foundation (v0.2.0)¶

    Core infrastructure: chain lifecycle, plugin system, experiment runner, basic artifact capture. ~6,000 lines.

    Phase 2: Attack Content & Validation (v0.3.0)¶

    All 6 benchmark attacks upgraded from scaffolds to working reproductions. 21 bugs found and fixed across 2 validation rounds (20 bugs + 1 enhancement). Determinism verified. Export bundles created.

    Phase 3: Production Readiness (v0.4.0)¶

    Git history cleaned, CI pipeline removed (local-only project), .venv fixed, artifact protection hardened, all 808 tests passing (up from 801 during pre-release). 2 additional bugs found and fixed during final pre-release testing. Documentation website built with MkDocs + Material theme for public GitHub Pages deployment.

    Limitations & Known Issues¶

    1. RPC dependency — Archive nodes required for historical forks. Ethereum works on Free Ankr tier; BSC/Polygon need paid provider (~$39/mo GetBlock)
    2. Non-EVM boundaries — Crucible is EVM-only. Attacks involving Solana, Cosmos, Harmony, or Substrate chains are modeled, not reproduced
    3. Poly Network determinism — Non-deterministic across 3 chains due to archive RPC variability (expected behavior)
    4. Cross-provider verification — Not done; only one RPC provider available
    5. Local environment — CI pipeline fully removed; all testing is on-device via Docker
