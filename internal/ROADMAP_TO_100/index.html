    Original Roadmap (Archived for Reference)¶

    The following phases were planned but are now deferred. They remain valid if the scope expands later.

    Phase 7: Agent Finishing + Docker Validation¶

    Goal: Agent system is fully usable; any researcher can run an agent experiment end-to-end inside Docker.

    7.1 Auto-wire LLM provider config from experiment YAML (🔥 High)¶

    The gap: LLMAgentPlugin.__init__() accepts provider_type, model, api_key as constructor args, but the experiment runner's _load_plugin() doesn't pass run_config values through. The YAML config model implies it works, but it doesn't.

    The fix: - Add a from_run_config(run_config: dict) -> Self classmethod (or constructor factory) to LLMAgentPlugin that reads agent_provider, agent_model, agent_base_url, agent_objective, agent_task from the run config - Or: move provider creation from __init__ to prepare() so it has access to context.run_config - Update _load_plugin() in experiment.py to pass run_config to the factory

    Files: crucible/agent_loop.py, crucible/experiment.py
    Tests: 3–5 new unit tests for factory method
    Effort: Small (half day)

    7.2 Create example agent attack pack (🔥 High)¶

    The gap: No concrete attacks/ example exists. Users can't see the pattern end-to-end.

    Deliverable: attacks/llm_exploit_agent/ with: - plugin.yaml — manifests type: agent - llm_exploit_agent.py — subclass of LLMAgentPlugin defaulting to Ollama - experiments/llm_exploit_agent.yaml — run config with agent_provider: ollama - scenarios/llm_exploit_agent/scenario.yaml — simple fork scenario - Tests: 5 integration-style tests (mock provider responses, verify loop terminates)

    Files: New directory attacks/llm_exploit_agent/, new experiment YAML, new scenario YAML
    Tests: 5–8
    Effort: Medium (1 day)

    7.3 Agent user guide (🟡 Medium)¶

    The gap: No documentation on how to run an agent experiment.

    Deliverable: A new agent user guide (created in a future PR) covering: - Quick-start: run the example agent plugin - YAML config reference and provider setup - Troubleshooting and custom plugin authoring

    Files: New doc
    Effort: Small (half day)

    7.4 Validate Docker end-to-end (🟡 Medium)¶

    The gap: Dockerfile + compose exist but haven't been validated on a clean environment with an actual attack run.

    Deliverable: - scripts/validate_docker.sh — runs docker compose up, verifies crucible doctor, runs demo experiment, exports result - Or: update existing AE scripts to include a Docker smoke test - Fix any Docker issues discovered during validation

    Files: scripts/validate_docker.sh (new), possibly Dockerfile fixes
    Tests: 1 integration test (test_docker_e2e, requires Docker)
    Effort: Medium (1 day)

    Phase 8: Poly Network → L2 Multi-Chain Exemplar (~85%)¶

    Goal: A full multi-chain bridge attack (3 chains: Ethereum + BSC + Polygon) running at L2 fidelity, proving the multi-chain pipeline works end-to-end.

    8.1 Verify fork blocks for all 3 chains (P0)¶

    The Poly Network exploit spanned Ethereum (block 12,996,062), BSC, and Polygon. Each needs: 1. Verify the archive node has state at the fork block 2. Confirm all affected contracts exist at that block 3. Capture the exploit transaction calldata

    Effort: 1–2 days (mostly waiting for RPC calls)

    8.2 Implement full exploit logic¶

    The Poly Network exploit works through EthCrossChainManager._executeCrossChainTx which has a keccak256 hash collision vulnerability. The attack: 1. Calls _executeCrossChainTx with a crafted cross-chain message 2. The hash of the function signature collides with putCurEpochConPubKeyBytes(bytes,bytes,uint64) 3. This replaces the keeper/public key 4. Attacker now controls the bridge and can drain all assets

    Deliverable: Full poly_network_privkey.py with actual calldata construction.

    Effort: 2–3 days (requires understanding the hash collision mechanism)

    8.3 Prove determinism (two runs, diff)¶

    Run the Poly Network attack twice and verify canonicalized state diffs are identical. Export the result.

    Effort: 0.5 day (run time + verification)

    8.4 Multi-chain topology test¶

    • Update topologies/poly_network.yaml for 3-chain topology
    • Verify the experiment runner correctly starts all 3 chains, bridges, and instruments each
    • Verify cross-chain correlation logging works

    Effort: 1 day

    Phase 9: Remaining 3 Attacks → L2 (~95%)¶

    9.1 BNB Token Hub → L2¶

    Current state: L2 scaffold with fork block (22,166,659), tx hash, and handlePackage() selector identified.
    What's needed: - BSC archive node access (hardest part — BSC archive nodes are scarce) - Verify 0x0000000000000000000000000000000000001004 (TokenHub) has code at fork block - Construct the malformed IAVL Merkle proof that bypasses verification - Demonstrate handlePackage() accepts it via eth_call - Two-run determinism verification

    Technique: The IAVL proof vulnerability allows forging cross-chain messages. The proof structure lacks height validation.

    Effort: 2–3 days

    9.2 Horizon (Harmony) → L2¶

    Current state: L2 scaffold with fork block (15,005,526), bridge address (0x2dcc...), and multisig analysis done.
    What's needed: - Verify the HorizonBridge contract has code at the fork block - demonstrate unlockTokens() function is callable and state changes

    Note: This is a multisig infrastructure compromise, not a contract bug. The reproduction demonstrates that the contracts behaved correctly under exploit conditions — which is valuable for the "defense analysis" narrative.

    Effort: 1–2 days

    9.3 Multichain → classification (not L2)¶

    What's needed: - Finalize the evidence.yaml to clearly state why L2 is impossible - Add documentation distinguishing infrastructure vs contract exploits - No code changes needed — this pack is correctly scoped

    Effort: 0.5 day

    Phase 10: Bridge Plugins + Patch Validation (100%)¶

    10.1 Real bridge plugin implementations¶

    The gap: bridges/demo_bridge/ exists but doesn't implement real bridge logic. The BridgePlugin lifecycle (deploy→configure→connect→destroy) has never been exercised with real contracts.

    Deliverable: At least one working bridge plugin (e.g., bridges/wormhole_bridge/) that: - Deploys mock Wormhole Core + TokenBridge contracts to both chains - Configures guardian set - Simulates VAA relaying via connect() - Properly cleans up via destroy()

    Why Wormhole? It's the most documented bridge with the most available Solidity source code.

    Effort: 3–4 days (Solidity deployment scripts + Python plugin)

    10.2 Patch validation workflow¶

    The gap: patches/ directory doesn't exist. The patch validation YAML spec (validation.yaml) from the implementation spec has never been built.

    Deliverable: - patches/ directory structure with at least one example patch - patches/wormhole_guardian_fix/ — a mock patch that prevents the forged VAA exploit - crucible patch validate command that: 1. Runs the experiment unpatched (records outcome) 2. Deploys the patch contract 3. Runs the experiment again 4. Asserts the attack fails and normal operations still work 5. Emits side-by-side diff report

    Effort: 3–4 days

    What USENIX AE Reviewers Will Actually Check¶

    Ranked by what the AE committee cares about most:

    1. 🥇 Run the demo experiment — docker compose run crucible experiment run experiments/demo_experiment.yaml must work on a fresh checkout
    2. 🥇 Verify determinism — Run twice, crucible diff shows identical results
    3. 🥈 Run a real attack — Nomad replay at L2 with export and bundle verification
    4. 🥈 Read the documentation — Evidence manifests, fidelity levels, architecture docs
    5. 🥉 Try the agent system — Run the example agent against the demo scenario
    6. 🥉 Inspect the artifact bundle — Verify JSONL schemas, integrity hashes, provenance

    Our focus should follow this priority order. Phase 7 (Docker validation) and Phase 8 (multi-chain L2 exemplar) directly address items 1, 2, and 3. Phase 10 (bridge plugins + patches) is lower priority for AE but important for the complete platform vision.

    Housekeeping (After Every Phase)¶

    • Update docs/internal/IMPLEMENTATION_STATUS.md with new capabilities and test count
    • Run pytest tests/unit/ and verify 0 failures
    • Run ruff check crucible/ tests/ and verify 0 warnings
    • Run pytest --cov=crucible and check coverage hasn't regressed
    • Verify CI workflows (unit.yml, integration.yml, docker-smoke.yml) pass
    • Commit and tag progress (e.g., v0.3.0-phase7)

    This roadmap is a living document. Update it as priorities shift and new gaps are discovered.
