    1. Project Overview¶

    Crucible is a CLI-first, modular EVM bridge security research testbed. It allows researchers to:

    • Fork any EVM chain at any block using Foundry/Anvil
    • Replay historical bridge exploits deterministically
    • Collect structured research artifacts (JSONL logs, state diffs, gas profiles)
    • Reset environments in seconds (3-tier reset: soft <5s → medium <30s → hard <5min)
    • Validate patches against known attack vectors
    • Publish exportable artifact bundles with integrity guarantees

    Architecture at a Glance¶

    CLI (crucible/cli.py → crucible/commands/*.py)
  │
  ├── Chain Manager — starts/stops Anvil subprocesses, manages lifecycle
  ├── Scenario Engine — loads and executes attack scenarios
  ├── Experiment Runner — orchestrates formal research runs
  ├── Plugin Registry — discovers and validates plugins
  ├── Baseline System — golden baseline capture and fingerprint verification
  ├── Structured Logger — writes 5 JSONL streams per run
  └── Dashboard (Streamlit) — browser UI for all of the above

    Key Files¶

    File Purpose
    crucible/cli.py Thin entry point, delegates to commands/
    crucible/config.py YAML config parsing with Pydantic models
    crucible/chain_manager.py Anvil process lifecycle
    crucible/rpc.py JSON-RPC client + provider fallback + rate limiting
    crucible/baseline.py Baseline capture, fingerprint, restore
    crucible/scenario.py Scenario loading and YAML-step execution
    crucible/experiment.py Formal experiment runner, matrix, resume
    crucible/logging.py JSONL emitter for 5 log streams
    crucible/plugin.py Plugin discovery and validation
    crucible/sdk.py Plugin interfaces (AttackPlugin, BridgePlugin)
    crucible/rpc.py JSON-RPC client with provider failover + rate limiting
    crucible/interfaces/__init__.py Public API re-exports (AttackPlugin, BridgePlugin, etc.)
    crucible/checkpoint.py Phase-level checkpoint/resume
    crucible/diff.py Canonicalized run comparison
    crucible/history.py SQLite experiment index

    Attack Pack Structure¶

    Each attack in attacks/<name>/ is a self-contained directory:

    attacks/<name>/
├── plugin.yaml          # Plugin manifest (name, type, class, version)
├── evidence.yaml        # Historical basis, scope, and assumptions
├── <name>.py            # Attack logic (implements AttackPlugin)
└── (optional) *.sol     # Foundry scripts

    Scenario Structure¶

    Each scenario in scenarios/<name>/:

    scenarios/<name>/
└── scenario.yaml        # Environment definition (chains, steps, assertions)

    3. Running Tests¶

    # Run all unit tests (fast — no external deps needed)
pytest tests/unit/ -q

# Run with verbose output
pytest tests/unit/ -v

# Run a specific test file
pytest tests/unit/test_config.py -v

# Run with coverage
pytest tests/unit/ --cov=crucible

# Run docs link validation (checks all internal .md links resolve)
pytest tests/unit/test_docs_links.py -v

# Run dashboard e2e tests (if Streamlit installed)
pytest tests/unit/test_dashboard_e2e.py -v

# Run integration tests (requires Foundry)
pytest tests/integration/ -m requires_foundry

    Current test count: 798 passing (all unit tests).

    4. Adding New Content¶

    Adding a New Attack (Quick Path)¶

    crucible attack extract <tx_hash> --name my_attack --chain ethereum

    This auto-generates the complete scaffold: - attacks/my_attack/plugin.yaml - attacks/my_attack/evidence.yaml - attacks/my_attack/my_attack.py - scenarios/my_attack_pre_exploit/scenario.yaml

    Adding a New Attack (Manual Path)¶

    mkdir -p attacks/my_attack
# Create plugin.yaml, my_attack.py, evidence.yaml
mkdir -p scenarios/my_attack_pre_exploit
# Create scenario.yaml

crucible plugin validate attacks/my_attack

    Adding a New Chain¶

    1. Add the chain to testbed.yaml under chains: (see ethereum for reference)
    2. Add RPC providers under rpc_providers: (at least one archive node)
    3. Create or update a topology file in topologies/ that references the new chain

    Adding a New Topology¶

    Create topologies/<name>.yaml:

    name: custom
chains: [ethereum, polygon]
bridge: null

    Adding a New Bridge Plugin¶

    1. Create bridges/<name>/ with plugin.yaml + implementation
    2. Implement BridgePlugin interface from crucible.sdk
    3. Add bridge address map to abis/<name>/

    Authoring a New Attack Pack¶

    All 6 benchmark packs are now complete at their respective fidelity levels. To author a new attack:

    1. Find exploit transaction hash on Etherscan/BscScan
    2. Set fork_block = exploit block - 1 in scenario
    3. Verify affected contract addresses are contracts (not EOAs) on Etherscan!
    4. Use crucible attack extract <tx_hash> --name <name> to auto-generate the scaffold
    5. Add concrete steps to scenario.yaml
    6. Run determinism check: run twice, crucible diff <run_a> <run_b>

    5. CLI Reference (Quick)¶

    Command Purpose
    crucible init Initialize workspace
    crucible doctor Validate environment (Python, Foundry, config, RPC)
    crucible status Show running chains
    crucible fork --topology <name> Start Anvil forks
    crucible setup --topology <name> Fork + bootstrap + capture baseline
    crucible experiment run <file.yaml> Formal research run
    crucible reset Soft reset (<5s)
    crucible diff <run_a> <run_b> Compare two runs
    crucible history List experiment history
    crucible export <run_id> Bundle a run for publication
    crucible clean Kill chains + remove temp state
    crucible dashboard Launch Streamlit UI

    For full reference: docs/CLI_REFERENCE.md

    7. Publication Workflow¶

    # 1. Run with publication gates enabled
crucible experiment run experiments/attack.yaml --publication-mode

# 2. Export the bundle
crucible export <run_id> --format tar-gz

# 3. Verify the export
crucible verify datasets/<run_id>

# 4. Archive on Zenodo
# See docs/ZENODO_ARCHIVE_GUIDE.md for details

    Each export bundle is self-contained with: - run_manifest.json (full provenance record) - run.jsonl, events.jsonl, storage_diffs.jsonl, gas_profile.jsonl - log_integrity.json (SHA-256 hashes) - REPRODUCE.md (exact commands to reproduce)

    9. Known Limitations¶

    Technical¶

    • Windows compatibility: Bash scripts in scripts/ require Git Bash or WSL. The crucible CLI works on Windows via uv run crucible.
    • Free-tier RPC limits: Archive-node RPC providers (LlamaRPC, Ankr, PublicNode) have rate limits of 10-25 req/s. Paid providers (Alchemy, Infura) are recommended for multi-chain benchmarks.
    • BSC/Polygon archive RPCs: Some multi-chain attack packs (Poly Network) require paid archive RPCs for BSC or Polygon. The Ethereum-only portions work on free tier.
    • Docker on Windows: Docker Compose networking may require additional configuration on Windows. See docs/internal/DOCKER_IMPLEMENTATION.md.

    Research Content¶

    • 6/6 benchmark packs at target fidelity: All attacks are complete at their declared fidelity levels. Horizon and BNB Token Hub were upgraded from scaffolds to L2. Multichain correctly classified as L1 (MPC infrastructure compromise, beyond EVM scope).
    • 3 bridge plugins: Nomad (ethereum + polygon), Poly Network (ethereum + polygon + bsc), Wormhole (ethereum + polygon). Each deploys real Solidity contracts with vulnerability pre-configured.
    • Poly Network multi-chain: 3-chain topology available (Ethereum + Polygon + BSC); determinism proven across all 3 chains. BSC/Polygon archive nodes at exploit blocks require paid RPCs.
    • No Foundry integration tests: The tests/integration/ directory exists as a scaffold but requires a live Anvil instance to run.

    Code¶

    • No type checking enforced: pyproject.toml has pyright config but it's not part of the regular test suite.
    • No integration tests running: Only unit tests are run in the main test suite. Integration and scenario tests require Foundry and/or RPC access.

    10. Future Work Guide¶

    Priority 1: Author New Attack Packs¶

    All 6 benchmark packs are complete. To author new attacks:

    1. Use crucible attack extract <tx_hash> to auto-generate a scaffold
    2. Verify contract addresses on Etherscan (confirmed contracts, not EOAs)
    3. Write the exploit plugin following patterns in existing packs
    4. Run determinism check — run twice, compare with crucible diff

    Priority 2: Live Anvil Fork Validation¶

    Run the 3 completed benchmark packs (Wormhole, Nomad, Poly Network) against actual Anvil forks:

    crucible setup --topology single
crucible experiment run experiments/wormhole_replay.yaml
crucible reset
crucible experiment run experiments/wormhole_replay.yaml
crucible diff DS-2026-XXXXX DS-2026-XXXXX

    Verify: storage diffs match, events match, gas matches, outcome matches.

    Priority 3: Cross-Provider Reproduction¶

    Run the full pipeline on a second machine/environment to claim the USENIX AE "Reproduced" badge:

    # On machine 2:
git clone <repo>
uv sync
cp .env.example .env  # fill in RPC URLs
crucible doctor
crucible verify datasets/<run_id>

    Priority 4: Type Checking & Linting¶

    Run type checking and linting locally as part of development: - ruff check crucible/ tests/ — linting - pyright crucible/ — type checking (optional, may not pass cleanly)

    Priority 5: Documentation Improvements¶

    • Add more case studies to docs/case_studies/ as packs are completed
    • Keep docs/internal/IMPLEMENTATION_STATUS.md updated as packs progress
    • Update README.md badge links when CI is configured
