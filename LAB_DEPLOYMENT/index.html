    Lab Deployment Guide¶

    How to host Crucible on a shared lab Linux workstation and make it accessible to all researchers. Last updated: June 2026

    1. Overview¶

    flowchart LR
    subgraph DevMachine["Your Dev Machine (Windows)"]
        A["git push origin v0.4.0"]
    end
    subgraph GitRepo["GitHub/GitLab (Remote Origin)"]
        B["Repository<br>with SSH deploy<br>key"]
    end
    subgraph LabWorkstation["Lab Linux Workstation (Ubuntu 24.04 LTS)"]
        C["git clone --branch v0.4.0<br>to ~/crucible_testbed"]

    end
    A --> B

    Your role: You develop on your Windows machine, push releases. The lab workstation runs the testbed 24/7 for all researchers to use.

    Deployment path: This guide uses ~/crucible_testbed as the deploy directory. Replace this with your actual clone path throughout.

    Important Notes Before Starting¶

    The Entrypoint Adds crucible Automatically¶

    The Docker container's entrypoint already runs uv run crucible "$@". So commands are:

    # [ok] CORRECT — entrypoint adds crucible
docker compose run --rm orchestrator experiment run ...

# [fail] WRONG — this becomes "crucible crucible experiment run"
docker compose run --rm orchestrator crucible experiment run ...

    experiment run Requires a Fork First¶

    experiment run connects to a running Anvil chain. You must fork before running. The standard pattern chains both in one container session:

    # [ok] CORRECT — always fork first, then run
docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/nomad_replay.yaml --no-baseline
"

# [fail] WRONG — this fails because no chain is running
docker compose run --rm orchestrator experiment run experiments/nomad_replay.yaml

    Why --entrypoint bash? The container's entrypoint is crucible. To run multiple commands (fork + experiment), you need a shell. --entrypoint bash gives you bash, then uv run crucible <cmd> calls the tool.

    Fresh Deployments Need --no-baseline¶

    On a fresh lab machine, no baselines exist yet. Add --no-baseline to skip the baseline comparison. Once a baseline is captured (see Operations Runbook), you can omit it.

    ANKR API Key Required¶

    Free RPC providers like LlamaRPC rate-limit Anvil's archive requests. You need an ANKR API key:

    1. Sign up at app.ankr.com
    2. Go to RPC → Ethereum → Active Endpoint
    3. Get a URL like https://rpc.ankr.com/eth/YOUR_KEY
    4. Set it as ETH_RPC_URL in .env

    2. Push to Remote Repository¶

    Your local repo currently has no remote configured. You need to create a remote repository and push the v0.4.0 tag.

    Step 1: Create a Remote Repository¶

    Create a repository on your Git provider of choice (GitHub, GitLab, self-hosted):

    Provider URL
    GitHub (recommended) github.com/new
    GitLab gitlab.com/projects/new
    Self-hosted GitLab/Gitea Your lab's internal git server

    Settings: - Repository name: crucible (or crucible-testbed) - Visibility: Private (lab research — you control access) - Initialize: Do NOT add README, .gitignore, or license (your repo already has these)

    Step 2: Link and Push¶

    # From your Windows dev machine (in the crucible directory)

# Add the remote
git remote add origin git@github.com:<your-org>/crucible.git
# or: git remote add origin git@gitlab.com:<your-org>/crucible.git

# Push the v0.4.0 tag
git push origin v0.4.0

# Push the main branch
git push origin main

# Verify
git remote -v
git log --oneline -3

    Note: Use SSH (git@github.com:...) not HTTPS. SSH key authentication is more secure for automated operations. If you need to set up an SSH key for your Windows machine first: 

    # Windows PowerShell
ssh-keygen -t ed25519 -C "your-lab-email"
# Add ~/.ssh/id_ed25519.pub to GitHub/GitLab → Settings → SSH Keys

    Step 3: Grant Lab Access¶

    Add the lab's Linux workstation SSH key as a Deploy Key with read-only access: - GitHub: Repository → Settings → Deploy Keys → Add deploy key - GitLab: Repository → Settings → Repository → Deploy Keys

    This lets the lab workstation clone without a user account.

    3. Lab Workstation Setup¶

    On the Linux workstation that will host Crucible:

    Requirements¶

    Requirement Minimum Recommended
    OS Ubuntu 22.04+ Ubuntu 24.04 LTS
    CPU 4 cores 8+ cores
    RAM 8 GB 16+ GB (Anvil forks consume 2-4 GB per chain)
    Disk 50 GB free 200 GB+ SSD (fork caches grow large)
    Docker 24.0+ 27.0+
    Python 3.11 (host, for scripts) 3.12

    Install Docker¶

    # One-time: install Docker
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh

# Add the lab user(s) to the docker group
sudo usermod -aG docker $USER

# Log out and back in (or: newgrp docker)
# Verify:
docker --version
docker compose version

    Create the Deployment Directory¶

    # Clone to ~/crucible_testbed (or any path you prefer)
mkdir -p ~/crucible_testbed

    Note: The directory is created automatically by git clone in the next step. This step is just informational.

    4. SSH Clone from Lab Workstation¶

    Step 1: Generate an SSH Deploy Key¶

    # On the lab workstation
ssh-keygen -t ed25519 -C "crucible-lab-deploy" -f ~/.ssh/crucible_deploy_key
cat ~/.ssh/crucible_deploy_key.pub

    Step 2: Add as Deploy Key¶

    Copy the output and add it to your Git provider: - GitHub: Repository → Settings → Deploy Keys → Add deploy key - Check "Allow write access" (so the lab can also push experiment artifacts later if needed) - GitLab: Repository → Settings → Repository → Deploy Keys → Add key

    Step 3: Configure SSH¶

    # On the lab workstation — create SSH config
cat >> ~/.ssh/config << 'EOF'
Host github.com gitlab.com
    IdentityFile ~/.ssh/crucible_deploy_key
    IdentitiesOnly yes
EOF

chmod 600 ~/.ssh/config

    Note: If using a self-hosted Git server, replace github.com with your server's hostname.

    Step 4: Clone¶

    # Clone into your home directory
cd ~
git clone --branch v0.4.0 --single-branch \
  git@github.com:<your-org>/crucible.git crucible_testbed

cd ~/crucible_testbed

# Verify
git log --oneline -3
git status

    Why --single-branch: Only pulls the v0.4.0 tag's branch — smaller clone, cleaner deployment. Path note: This guide assumes ~/crucible_testbed. If you cloned elsewhere, adjust paths accordingly.

    5. Docker Build & Configuration¶

    Step 1: Configure Environment Variables¶

    cd ~/crucible_testbed

# Copy the template
cp .env.example .env

# Edit .env with your lab's archive RPC URLs
nano .env

    Recommended lab configuration:

    # ── Ethereum Archive (ANKR — sign up for a free API key) ────
ETH_RPC_URL=https://rpc.ankr.com/eth/YOUR_ANKR_API_KEY

# ── Multi-Chains (GetBlock paid plan recommended for BSC/Polygon) ─
# POLYGON_RPC_URL=https://go.getblock.io/YOUR_GETBLOCK_TOKEN
# BSC_RPC_URL=https://go.getblock.io/YOUR_GETBLOCK_TOKEN
# HARMONY_RPC_URL=https://rpc.ankr.com/harmony

# ── Suppress nightly build warning from Foundry ────────────
FOUNDRY_DISABLE_NIGHTLY_WARNING=true

# ── Dashboard ────────────────────────────────────────────────
STREAMLIT_SERVER_RUNONSAVE=false

    Why ANKR and not LlamaRPC? Free providers like eth.llamarpc.com rate-limit Anvil's heavy archive requests during fork initialization. ANKR's free tier (sign up at app.ankr.com) provides a dedicated API key with better throughput.

    Step 2: Build the Docker Images¶

    cd ~/crucible_testbed

# Build both images (first build: ~5-10 minutes)
docker compose build orchestrator
docker compose build dashboard

# Verify
docker compose run --rm orchestrator doctor

    Step 3: Run Initial Validation¶

    Note on startup delay: Every docker compose run --rm orchestrator <command> first runs uv sync (~3-5 seconds) to ensure all dependencies are installed. This is expected behavior, not a performance issue.

    # Full validation — no RPC needed for these:
docker compose run --rm orchestrator init
docker compose run --rm orchestrator doctor
docker compose run --rm orchestrator plugin list
docker compose run --rm orchestrator abi validate

# Demo experiment — no RPC needed:
docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/demo_experiment.yaml --no-baseline
"

# Check results
docker compose run --rm orchestrator logs list
docker compose run --rm orchestrator history

    Step 4: Verify Anvil Can Fork¶

    Test that your ANKR API key works with Anvil:

    docker compose run --rm --entrypoint bash orchestrator -c "
  timeout 20 anvil \
    --fork-url \$ETH_RPC_URL \
    --fork-block-number 12996693 \
    --host 127.0.0.1 \
    --port 8545 \
    --chain-id 1 \
    --gas-price 0 \
    --block-base-fee-per-gas 1000000000 \
    --auto-impersonate \
    --order fifo 2>&1 || echo 'EXIT:' \$?
"

    Expected output: Anvil starts, shows "Listening on 127.0.0.1:8545" and a block number of 12996693. The EXIT: 124 at the end is just timeout killing the process — Anvil runs forever as a server.

    If you see "rate limited" or "failed to fetch network chain ID", your RPC provider is blocking archive requests. See the troubleshooting section below.

    Step 5: Run a Real Experiment¶

    # Fork the chain and run the Nomad replay in one go:
docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/nomad_replay.yaml --no-baseline \
    --name 'my-first-run'
"

    Why --no-baseline? On a fresh deployment, no baselines exist yet. The --no-baseline flag skips the baseline comparison step. Once a baseline is captured (see docs/OPERATIONS_RUNBOOK.md), you can omit it.

    6. Multi-User Access¶

    Multiple researchers will use the same workstation. Here's how to set it up properly.

    User Group Setup¶

    # Create a 'crucible' group for all researchers
sudo groupadd crucible

# Add each researcher to the group
sudo usermod -aG crucible alice
sudo usermod -aG crucible bob
sudo usermod -aG crucible carol

# Make the deploy directory group-owned
sudo chgrp -R crucible ~/crucible_testbed
sudo chmod -R g+rw ~/crucible_testbed
sudo find ~/crucible_testbed -type d -exec chmod g+s {} \;  # new files inherit group

    Recommended Workflow per Researcher¶

    Each researcher should run experiments via Docker. The entrypoint already adds crucible, so just use the subcommand:

    # Each researcher runs (note: no 'crucible' before 'experiment'):
docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/nomad_replay.yaml --no-baseline \
    --name 'alice-nomad-run1'
"

# Their artifacts go to: ~/crucible_testbed/logs/nomad_replay/DS-2026-XXXXX/
# All researchers in the 'crucible' group can read each other's logs

    Researchers can also name runs with --name to distinguish their work:

    docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/nomad_replay.yaml \
    --name 'alice-nomad-v2' --no-baseline
"

    Isolating Work with testbed.local.yaml¶

    Each researcher can create and modify testbed.local.yaml for their experiment-specific fork blocks — it's gitignored so it won't conflict:

    # testbed.local.yaml — per-user override, gitignored
chains:
  ethereum:
    fork_block: 15259100  # Alice is working on Nomad

    Coordinating Anvil Ports (Optional)¶

    If multiple researchers run crucible fork simultaneously, both will try port 8545 and get a conflict. For concurrent use, researchers can override ports in their testbed.local.yaml:

    # testbed.local.yaml — Alice uses port 9545
chains:
  ethereum:
    fork_block: 15259100
    port: 9545  # Different port to avoid conflict

    In practice, lab researchers typically run experiments one at a time — the workstation is shared, not used simultaneously. If concurrency becomes an issue, Docker Compose profiles or separate compose files can isolate environments.

    7. Shared Data & Storage¶

    Directory Layout¶

    ~/crucible_testbed/       # Deploy root (git clone)
├── crucible/              # Code (git repo)
├── docs/                  # Documentation (git repo)
├── experiments/           # Experiment YAMLs (git repo)
├── .env                   # Shared RPC credentials (gitignored)
├── testbed.local.yaml     # Per-user overrides (gitignored)
├── docker-compose.yml     # Service definitions
│
├── logs/                  # ← SHARED: All experiment artifacts
│   └── <experiment>/      #   One dir per experiment
│       └── DS-2026-xxxxx/ #   One dir per run
│
├── states/                # ← SHARED: Anvil state files
├── datasets/              # ← SHARED: Exported publication bundles
├── .fork_cache/           # ← SHARED: Anvil RPC cache (can be large!)
│
└── docker-compose.override.yml  # Lab-specific overrides (gitignored)

    Docker Volumes¶

    The docker-compose.yml uses a combination of bind mounts and named volumes:

    volumes:
  - .:/workspace              # Bind-mount the host repo (all code, cache, config)
  - states_data:/workspace/states  # Named volume for chain state
  - logs_data:/workspace/logs      # Named volume for experiment artifacts

    The named volumes (states_data, logs_data) persist across container restarts. On the host, they live in:

    # Docker named volume locations:
/var/lib/docker/volumes/crucible-testbed_states_data/
/var/lib/docker/volumes/crucible-testbed_logs_data/

    Note: The fork_cache named volume was removed because .fork_cache/ is now managed on the bind-mounted working directory (.:/workspace) directly. This avoids "Device/resource busy" conflicts between the Docker volume and the host filesystem. The .fork_cache/ directory is gitignored and lives alongside the code.

    Since the working directory is bind-mounted (.:/workspace), the logs/, states/, datasets/, and .fork_cache/ directories on the host filesystem are all accessible directly by all researchers in the crucible group.

    Disk Usage Monitoring¶

    # Quick disk check
docker compose run --rm orchestrator status --disk

# Full breakdown per directory
du -sh logs/ states/ datasets/ .fork_cache/

# Clean old scratch runs (auto-cleanable)
docker compose run --rm orchestrator clean --logs-older-than 30

# Remove all fork cache (re-downloaded on next fork)
docker compose run --rm orchestrator clean --cache

    8. Streamlit Dashboard on the Network¶

    The dashboard gives all researchers a browser UI to view experiment results.

    Step 1: Create docker-compose.override.yml¶

    # ~/crucible_testbed/docker-compose.override.yml (gitignored)
cat > docker-compose.override.yml << 'EOF'
services:
  dashboard:
    ports:
      - "0.0.0.0:8501:8501"   # Listen on all network interfaces
    restart: always
    environment:
      - STREAMLIT_SERVER_RUNONSAVE=false
EOF

    Step 2: Start the Dashboard as a Daemon¶

    # Start dashboard in background (survives terminal close)
docker compose up -d dashboard

# Verify it's running
docker compose ps
# Should show: dashboard   Up  0.0.0.0:8501->8501/tcp

    Step 3: Access from Lab Network¶

    Researchers access the dashboard at:

    http://<LAB_WORKSTATION_IP>:8501

    To find the IP: 

    # On the lab workstation
hostname -I | awk '{print $1}'
# Example output: 192.168.1.100
# → Dashboard at: http://192.168.1.100:8501

    Step 4: Security (Lab Network Only)¶

    For a lab-internal deployment (no external exposure), the dashboard is safe as-is. If the workstation has a public IP, add a firewall rule:

    # Allow only lab subnet
sudo ufw allow from 192.168.0.0/16 to any port 8501
sudo ufw deny 8501

    Starting Everything for Full Service¶

    # Start dashboard + optional daemon Anvil chains
docker compose up -d dashboard

# Optionally pre-start Anvil chains (if researchers frequently fork)
docker compose up -d anvil-ethereum

    9. Pulling Updates¶

    When you tag a new release on your dev machine, update the lab workstation:

    # On the lab workstation
cd ~/crucible_testbed

# 1. Pull the new tag
git fetch origin --tags
git checkout v0.X.Y

# 2. Rebuild Docker images
docker compose build orchestrator

# 3. Run validation
docker compose run --rm orchestrator doctor
docker compose run --rm orchestrator plugin list

# 4. Check nothing is broken
docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/demo_experiment.yaml --no-baseline
"

# 5. Restart dashboard if it was running
docker compose up -d dashboard

    10. Daily Operations¶

    Morning Health Check¶

    # Quick check that everything is alive
docker compose ps
docker compose run --rm orchestrator doctor
docker compose run --rm orchestrator status --disk

    Running an Experiment¶

    # Standard: fork + run in one container (--no-baseline for fresh runs)
docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/nomad_replay.yaml --no-baseline \
    --name 'alice-nomad-2026-06-08'
"

    Browsing Results¶

    # List completed runs
docker compose run --rm orchestrator logs list
docker compose run --rm orchestrator history

# Or use the browser dashboard
# http://<LAB_WORKSTATION_IP>:8501

    Cleanup¶

    # Weekly: clean old scratch runs
docker compose run --rm orchestrator clean --logs-older-than 7

# Monthly: rebuild history index
docker compose run --rm orchestrator history --rebuild

# As needed: remove all states
docker compose run --rm orchestrator clean --states

    Full Shutdown¶

    # Stop everything (containers + remove volumes)
docker compose down

# To also delete all data:
docker compose down -v

    11. Backup Strategy¶

    What to Back Up¶

    Data Location Frequency Size Critical?
    Experiment logs logs/ Weekly Variable [ok] Yes — primary research output
    Exported bundles datasets/ Per-publish Small [ok] Yes — publication artifacts
    Fork cache .fork_cache/ Never 10-50 GB [fail] Can re-download
    Baselines/snapshots states/ Event-based Small [warn] Nice to have
    Config .env, testbed.local.yaml Events Tiny [ok] Yes — backup manually

    Backup Commands¶

    # Backup experiment logs (compress with date)
tar czf "backup-logs-$(date +%Y%m%d).tar.gz" logs/

# Backup specific run
docker compose run --rm orchestrator export DS-2026-00001
tar czf "backup-export-DS-2026-00001.tar.gz" datasets/DS-2026-00001/

# Automated: add to crontab
# crontab -e
# 0 2 * * 0 cd ~/crucible_testbed && tar czf /backup/logs-$(date +\%Y\%m\%d).tar.gz logs/

    Restore¶

    # Extract backed-up logs
tar xzf backup-logs-20260608.tar.gz

# Rebuild history index after restore
docker compose run --rm orchestrator history --rebuild

    12. Security Best Practices¶

    Critical: .env is GitIgnored¶

    # Verify .env is NOT tracked
git check-ignore .env
# Should output: .env  (if it's properly gitignored)

# NEVER commit .env — it contains real API keys

    File Permissions¶

    # Set correct permissions after any change
sudo chgrp -R crucible ~/crucible_testbed
sudo chmod -R g+rw ~/crucible_testbed
sudo find ~/crucible_testbed -type d -exec chmod g+s {} \;

    Firewall (If Workstation Has External IP)¶

    # Allow SSH
sudo ufw allow ssh

# Allow dashboard — lab network only
sudo ufw allow from 192.168.0.0/16 to any port 8501

# Allow Anvil RPC — localhost only (these are NOT exposed externally by default)
sudo ufw deny 8545
sudo ufw deny 8546
sudo ufw deny 8547

# Deny everything else
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw enable

    Regular Maintenance¶

    # Keep Docker updated
sudo apt update && sudo apt upgrade -y docker-ce docker-ce-cli

# Clean unused Docker resources (disk space)
docker system prune -f --volumes

# Rotate logs
sudo journalctl --vacuum-time=30d

    13. Quick-Start Reference¶

    First-Time Setup (One Command)¶

    # On the lab workstation, after SSH key is configured:
git clone --branch v0.4.0 --single-branch \
  git@github.com:<your-org>/crucible.git ~/crucible_testbed && \
cd ~/crucible_testbed && \
cp .env.example .env && \
nano .env && \
docker compose build orchestrator && \
docker compose run --rm orchestrator doctor && \
echo "Setup complete!"

    Daily Commands Cheatsheet¶

    cd ~/crucible_testbed

# Health check
docker compose ps
docker compose run --rm orchestrator doctor
docker compose run --rm orchestrator status --disk

# Run experiment (--no-baseline until a baseline is captured)
docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/nomad_replay.yaml --no-baseline --name 'my-run'
"

# Browse results
docker compose run --rm orchestrator logs list
docker compose run --rm orchestrator history
# Or: open http://<lab-ip>:8501

# Export & verify
docker compose run --rm orchestrator export DS-2026-XXXXX
docker compose run --rm orchestrator verify datasets/DS-2026-XXXXX

# Cleanup
docker compose run --rm orchestrator clean --logs-older-than 30
docker compose run --rm orchestrator clean --states

# Update to new release
git fetch origin --tags && git checkout v0.X.Y && docker compose build orchestrator

# Full shutdown
docker compose down

    14. Troubleshooting¶

    Anvil Fails: "rate limited" / "failed to fetch network chain ID"¶

    Error: failed to fetch network chain ID
Context:
- Max retries exceeded HTTP error 503 ... "Rate Limited"

    Cause: The RPC provider (e.g., LlamaRPC) is rate-limiting Anvil's archive requests.

    Fix: Use ANKR with a free API key: 1. Sign up at app.ankr.com 2. Get your Ethereum endpoint: https://rpc.ankr.com/eth/YOUR_KEY 3. Set ETH_RPC_URL in .env

    Anvil Fails: "Unauthorized: You must authenticate"¶

    Error: failed to fetch network chain ID
Context:
- server returned an error response: error code -32000: Unauthorized: You must authenticate

    Cause: ANKR requires an API key in the URL.

    Fix: Append your API key: https://rpc.ankr.com/eth/PASTE_YOUR_KEY_HERE

    "No such command 'crucible'"¶

    Cause: The Docker entrypoint already runs uv run crucible "$@". Adding crucible yourself becomes crucible crucible experiment run.

    Fix: Run the subcommand directly: 

    # [ok] Correct
docker compose run --rm orchestrator experiment run ...

# [fail] Wrong
docker compose run --rm orchestrator crucible experiment run ...

    "baseline manifest not found"¶

    ERROR: baseline manifest not found for scenario nomad_pre_exploit

    Cause: No baseline captured yet on this machine.

    Fix: Add --no-baseline flag and fork first: 

    docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/nomad_replay.yaml --no-baseline
"

    "Connection refused" on experiment run¶

    ERROR: Experiment failed: RPC call failed for anvil_dumpState at http://127.0.0.1:8545: Connection refused

    Cause: You ran experiment run without forking first. Anvil isn't running on port 8545.

    Fix: Always fork before running — use the chained pattern: 

    docker compose run --rm --entrypoint bash orchestrator -c "
  rm -f states/*.meta.json && \
  uv run crucible fork --topology single && \
  uv run crucible experiment run experiments/nomad_replay.yaml --no-baseline
"
