    1. Overview & Goals¶

    Crucible is a CLI-first, modular, reproducible security research testbed for blockchain bridge vulnerability research. It enables researchers to:

    • Replay historical bridge exploits at exact block forks with deterministic results
    • Instantly reset the environment to a verified clean baseline
    • Compose single-chain, dual-chain, and full bridge topologies from a single config switch
    • Automate repetitive tasks — setup, fork, reset, run, log, teardown
    • Collect structured, publication-quality telemetry from every experiment run
    • Validate security patches against known attack vectors

    The core philosophy is CLI-first, scenario-as-code, and zero manual setup between experiments.

    1.1 Prerequisites & Environment Setup¶

    Every developer working on Crucible needs the following before anything else runs.

    System requirements: - Python 3.11+ - Docker + Docker Compose v2 - Git - uv — Python package manager (fast, lockfile-based, reproducible)

    Install uv (if not already installed): 

    curl -LsSf https://astral.sh/uv/install.sh | sh
# Verify
uv --version

    Install Foundry (pin the version — do not use latest): 

    curl -L https://foundry.paradigm.xyz | bash
foundryup --version nightly-cafc2606a2187a42b236df4aa65f4e8cdfcea970
# Verify
anvil --version
forge --version

    The exact commit hash above must match what is in foundry.toml. Never run foundryup without a version — it will pull the latest nightly and silently break time-dependent exploits. Crucible enforces this at startup (see §2.1).

    Python environment (using uv): 

    uv sync                          # creates .venv, installs exact versions from uv.lock
source .venv/bin/activate         # Linux/macOS
# .venv\Scripts\activate          # Windows

    Why uv instead of pip? With 4–10 researchers, dependency drift causes "works on my machine" failures. uv.lock pins every dependency to an exact version and hash. uv sync guarantees byte-identical installs across machines. It is also 10–50x faster than pip.

    RPC provider setup (.env — never commit this file): 

    cp .env.example .env

Self-hosted nodes (Erigon, Reth) are preferred for offline research.

    2. Core Design Principles¶

    Principle What It Means in Practice
    Reproducibility First Every experiment is fully reproducible from a single YAML file and a git commit hash
    Immutable Baselines Golden baselines are fingerprinted and verified before every run — never mutated
    Scenario-as-Code Attacks, environments, and experiments are version-controlled artifacts, not shell scripts
    Plug-and-Play Modules Chains, bridges, and attack packs implement standard interfaces; swap freely without touching core
    CLI is Source of Truth Every action the UI can do, the CLI can do — the UI is a wrapper, never the owner
    Automation Everywhere No manual environment preparation; reset is one command, always idempotent
    Full Observability Every action emits a structured JSON log entry; every run produces a complete research artifact bundle

    2.1 Determinism Guarantees¶

    The claim "deterministic replay" is the foundation of Crucible's research credibility. This section specifies exactly how determinism is enforced — not as a hope, but as a set of Anvil flags and Chain Manager rules that are mandatory for every run.

    Problem: By default, Anvil uses wall-clock time for block timestamps, variable gas pricing, and auto-assigns coinbase addresses. Two runs of the same experiment will produce different timestamps, different gas costs, and potentially different transaction ordering. The research claim of "identical state diffs on repeated runs" fails.

    Mandatory Anvil flags for deterministic mode:

    Flag Value Why
    --gas-price 0 Fixes gas price. Eliminates gas cost variation between runs.
    --block-base-fee-per-gas 1000000000 (1 gwei) Fixes EIP-1559 base fee. Prevents dynamic fee fluctuation.
    --auto-impersonate (enabled) Allows sending from any address without private key.
    --chain-id From testbed.yaml Must match the forked chain to avoid replay protection issues.
    --order fifo Process transactions in submission order. Prevents nonce-based reordering.
    --timestamp Fork block timestamp Fix the starting block timestamp to the forked block's timestamp — not wall-clock.

    Determinism configuration in testbed.yaml: 

    chains:
  ethereum:
    rpc_url: "https://eth.llamarpc.com"
    fork_block: 14211979
    chain_id: 1
    port: 8545
    determinism:
      gas_price: 0
      block_base_fee: 1000000000       # 1 gwei — fixed, not dynamic
      timestamp: "fork_block"          # "fork_block" = use the forked block's timestamp
                                       # or an explicit epoch: 1644350000
      coinbase: "0x0000000000000000000000000000000000000000"
      auto_mine: true                  # one tx per block — deterministic ordering
      order: "fifo"                    # transaction processing order

    Chain Manager enforcement: - On start(), the Chain Manager reads the determinism block and translates it to Anvil CLI flags - If determinism is not defined, Crucible applies the defaults shown above — deterministic mode is opt-out, not opt-in - crucible status displays the active determinism flags for each chain: 

    CHAIN       PORT   BLOCK       GAS PRICE   BASE FEE   TIMESTAMP    ORDER
ethereum    8545   14211979    0           1 gwei     fork_block   fifo
polygon     8546   25000000    0           1 gwei     fork_block   fifo

    Foundry version enforcement: - crucible setup checks forge --version against the version pinned in foundry.toml - If versions don't match: abort with a clear error and the exact foundryup command to install the correct version - Override for development: --skip-version-check (never used in CI) - The pinned version is embedded in every run_manifest.json for provenance

    Verification of determinism: Crucible guarantees deterministic local execution after the fork is loaded. It does not automatically guarantee full historical equivalence. Historical equivalence is established by each attack pack's evidence manifest and fidelity level.

    The scenario test suite (§15.3) verifies local execution repeatability by running the same experiment twice and comparing the artifacts. Instead of requiring brittle byte-for-byte JSON equality (which fails on timestamps), it applies canonical comparison rules: - Canonicalized state diffs are identical - Canonicalized balance diffs are identical - Transaction outcomes are identical - Relevant event sequences are identical - Configured invariant/assertion results are identical

    2.2 Reproduction Fidelity Model¶

    For bridge attack research, treating "deterministic replay" as a single binary goal is too coarse. Crucible defines explicit fidelity levels to clarify what is being reproduced:

    Level Name Meaning Academic Use
    L0 Narrative simulation Demonstrates the vulnerability pattern but not historical state Teaching/demo only
    L1 Contract-level reproduction Reproduces vulnerable contract behavior on local deployed contracts Useful for mechanism validation
    L2 Fork-state reproduction Uses historical fork block and real deployed contracts/state Strong academic baseline
    L3 Transaction-path reproduction Replays or closely reconstructs historical exploit transaction sequence Best for single-chain EVM attacks
    L4 Cross-chain causal reproduction Reconstructs source-chain, destination-chain, bridge messages, and final state impact Best possible standard for bridge attacks

    Each attack pack must declare its fidelity level and evidence basis.

    2.3 Threat Model & Research Scope¶

    To evaluate patches and attacks effectively, the testbed's simulation boundaries are explicit.

    Threat Model / In-Scope: - Smart contract vulnerability exploitation (re-entrancy, uninitialized state, logic errors). - On-chain EVM state manipulation and transaction execution. - Bridge message payload forging or replay (when modeled correctly).

    Research Questions crucible is designed to answer: - Can the historical vulnerable state be reconstructed deterministically? - Can the exploit condition be reproduced? - Which state variables and balances changed as a result of the exploit? - Which specific invariants would have detected or prevented the exploit? - Does a proposed patch block the exploit without breaking intended bridge behavior?

    Out of Scope for V1: - Non-EVM consensus simulation: If an attack involves Solana or CosmWasm, the non-EVM portion is simulated via a mock or skipped. - Real validator node execution: We do not run actual relayer/validator binaries. Their signatures or message passing behaviors are impersonated. - Exact public mempool ordering: Sandwitch attacks or MEV requiring millisecond mempool state are not modeled faithfully. - L1/L2 sequencer behavior: Unless explicitly modeled in the attack steps.

    Key architectural decisions:

    • Stateful Core, Stateless Scenarios. The Chain Manager owns infrastructure state. Scenarios only declare what to do — they never manage Anvil processes or ports.
    • Topology-Driven Composition. A YAML topology profile declares which modules are active. Single-chain, dual-chain, and full-bridge are three named profiles — not if branches in the code.
    • Snapshot-Driven Reset. Baseline is captured once after initial setup. Reset is anvil_loadState — no chain restart, under 5 seconds for a soft reset.
    • Separation of Scenario and Experiment. A scenario defines the environment (which chains, which fork blocks, which contracts). An experiment references a scenario plus an attack pack. Scenarios are reusable across many experiments.

    4. Component Specifications¶

    4.1 Chain Manager¶

    File: crucible/chain_manager.py

    Purpose: Start, stop, snapshot, and restore Anvil nodes that fork specific chains at specific blocks. Manages process lifecycle and exposes health checks.

    class ChainManager:
    def start(self, name: str) -> None:
        """Launch anvil subprocess with fork URL and block number from config."""

    def stop(self, name: str) -> None:
        """Terminate the anvil process and remove PID file."""

    def is_running(self, name: str) -> bool:
        """Health check via eth_blockNumber JSON-RPC call."""

    def snapshot(self, name: str, snapshot_path: str) -> None:
        """Call anvil_dumpState, write state JSON to snapshot_path."""

    def restore(self, name: str, snapshot_path: str) -> None:
        """Call anvil_loadState from snapshot_path. Does NOT restart anvil."""

    def get_rpc_url(self, name: str) -> str:
        """Return http://localhost:<port> for the named chain."""

    def advance_time(self, name: str, seconds: int) -> None:
        """evm_increaseTime + evm_mine for time-dependent exploit testing."""

    def impersonate(self, name: str, address: str) -> None:
        """anvil_impersonateAccount — enables sending tx as any address."""

    def set_balance(self, name: str, address: str, value_wei: int) -> None:
        """anvil_setBalance — fund any account without a private key."""

    Internal details: - Each chain runs as subprocess.Popen. PID and metadata (port, chain ID, fork block) stored under states/<chain_name>.meta.json. - Ports and chain IDs come from testbed.yaml. Chain ID must be unique per instance via --chain-id flag. - snapshot/restore use cast rpc calls. State files stored under states/<chain_name>/. - On startup, the manager polls eth_blockNumber with a 30-second timeout before declaring the chain ready.

    Error handling: - If Anvil fails to start (unreachable RPC, unsupported block), raises ChainStartError with the exact Anvil stderr output. - Every command that requires a live chain runs is_running() first.

    4.2 Scenario Engine¶

    File: crucible/scenario_engine.py

    Purpose: Discover, load, and execute attack scenarios from the scenarios/ directory. A scenario is a self-contained directory with a standardised structure.

    Scenario directory structure: 

    scenarios/<name>/
├── scenario.yaml       # Mandatory — metadata, chains, bridge refs, steps
├── steps.py            # Optional — Python step logic
└── Exploit.s.sol       # Optional — Foundry script

    scenario.yaml schema: 

    name: "poly_bridge_reentrancy"
description: "Replay the Poly Network reentrancy exploit — August 2021"
version: "1.0.0"
chains: [ethereum, polygon]         # which chains from testbed.yaml to activate

bridge:                             # optional — resolved at runtime
  name: "PolyNetworkBridge"
  contracts:
    ethereum: "0xVault..."
    polygon:  "0xChildToken..."

default_sender: "0xf39Fd6e51aad88F6F4ce6aB8827279cffFb92266"

steps:
  - type: python_script             # delegates to steps.py::run()

  # OR use inline atomic steps (no Python needed):
  # - type: impersonate
  #   chain: ethereum
  #   account: "0xRelayer..."
  # - type: send_tx
  #   chain: ethereum
  #   to: "bridge.ethereum"         # resolves via bridge registry
  #   data: "0x..."
  #   value: 0
  # - type: advance_time
  #   chain: polygon
  #   seconds: 3600
  # - type: forge_script
  #   script: Exploit.s.sol:ExploitScript
  #   chain: ethereum
  #   extra_args: ["--sender", "0xAttacker..."]

continue_on_failure: false

    Three execution modes:

    Mode When Used Best For
    python_script steps.py present Complex multi-step logic, custom assertions
    YAML atomic steps steps: list in scenario.yaml Simple replays, no coding required
    Foundry script .sol present, no steps: defined Native Forge scripting, vm. cheatcodes

    Python step API example (steps.py): 

    from crucible import chain, log, impersonate, advance_time

def run():
    # Impersonate the bridge relayer
    impersonate("ethereum", "0xRelayerAddress")

    # Send the crafted exploit transaction
    tx = chain("ethereum").send_transaction(
        to="0xVault...",
        data="0x<encoded_payload>",
        value=0,
        sender="0xRelayerAddress"
    )
    log.step("Exploit tx sent", tx_hash=tx.hash, gas_used=tx.gas_used, status=tx.status)

    # Advance time past the challenge window on L2
    advance_time("polygon", 7200)

    # Verify the attacker's resulting balance
    balance = chain("polygon").call(
        to="0xChildToken...",
        data="0x70a08231" + "0xAttacker...".zfill(64)
    )
    log.info("Post-exploit attacker balance", balance=int.from_bytes(balance, "big"))

    YAML atomic steps example: 

    steps:
  - type: impersonate
    chain: ethereum
    account: "0xRelayer..."
  - type: send_tx
    chain: ethereum
    to: "0xVault..."
    data: "0x..."
    value: 1000000000000000000
  - type: advance_time
    chain: polygon
    seconds: 3600
  - type: assert_balance
    chain: polygon
    account: "0xAttacker..."
    min_value: 1000000000000000000

    Symbol resolution: Contract addresses like bridge.ethereum are resolved at runtime by looking up the scenario's bridge block, then falling back to the global bridges.yaml registry. This makes scenarios portable across forks.

    4.3 Bridge Engine¶

    File: crucible/bridge_engine.py

    Purpose: Deploy, configure, and simulate bridge infrastructure between chains. Fully optional — only loaded when the active topology profile includes a bridge.

    class BridgePlugin:
    def metadata(self) -> dict:
        """Return bridge name, version, and supported chain pairs."""

    def deploy(self, chain_a: str, chain_b: str) -> None:
        """Deploy bridge contracts to both chains using Forge scripts."""

    def configure(self, config: dict) -> None:
        """Set validator set, relayer addresses, guardian keys."""

    def connect(self) -> None:
        """Start the relayer / message-passing simulation process."""

    def reset(self) -> None:
        """Reset bridge state to post-deploy baseline."""

    def destroy(self) -> None:

    4.8 Experiment Checkpoint & Resume¶

    File: crucible/checkpoint.py

    Purpose: Multi-phase bridge attack replays can take minutes to hours. If a crash, OOM, or Ctrl+C interrupts a run at phase 4 of 5, the researcher should not have to restart from scratch. Crucible saves phase-level checkpoints automatically.

    Checkpoint lifecycle: 

    Phase 1 (environment_setup) completes
    → anvil_dumpState → states/checkpoints/<run_id>/phase_1.json
    → checkpoint.json updated: phase 1 = complete

Phase 2 (privilege_escalation) completes
    → anvil_dumpState → states/checkpoints/<run_id>/phase_2.json
    → checkpoint.json updated: phase 2 = complete

Phase 3 (state_manipulation) — CRASH (OOM / Ctrl+C / RPC timeout)
    → graceful shutdown handler (§22.1) saves partial state
    → checkpoint.json updated: phase 3 = interrupted

--- later ---

crucible experiment resume <run_id>
    → load checkpoint.json: last completed = phase 2
    → anvil_loadState from phase_2.json
    → re-execute from phase 3 onward
    → on success: delete checkpoint files, produce full artifact bundle

    Implementation rules: 1. Checkpoints are saved after every successfully completed phase — not just on failure 2. On successful experiment completion, checkpoint files are deleted (configurable: keep_checkpoints: true in experiment YAML for debugging) 3.     5.1 RPC Provider Management¶

    Problem: Free-tier RPC providers (LlamaRPC, Ankr, PublicNode, dRPC) enforce 10–25 requests/second. During initial fork, Anvil issues hundreds of eth_getStorageAt calls. Without fallback and rate limiting, experiments fail mid-fork with opaque timeout errors.

    Provider registry in testbed.yaml: 

    providers:
  ethereum:
    - url: "https://eth.llamarpc.com"
      name: llamarpc
      priority: 1                      # tried first
      rate_limit_rps: 25               # max requests per second
      type: free
      archive: true

    - url: "https://rpc.ankr.com/eth"
      name: ankr
      priority: 2                      # fallback
      rate_limit_rps: 15
      type: free
      archive: true

    - url: "https://eth-mainnet.g.alchemy.com/v2/${ALCHEMY_KEY}"
      name: alchemy
      priority: 3                      # used only if ALCHEMY_KEY is set
      rate_limit_rps: 30
      type: paid
      archive: true

  polygon:
    - url: "https://polygon.llamarpc.com"
      name: llamarpc
      priority: 1
      rate_limit_rps: 25
      type: free
      archive: true

    Fallback behavior: 1. Chain Manager tries the highest-priority provider first 2. If the provider returns 429 (rate limited), 5xx (server error), or times out (30s default): - Log a warning: "Provider llamarpc rate limited, falling back to ankr" - Rotate to the next provider in priority order 3. If all providers fail, abort with exit code 5 and a clear error listing what was tried 4. Providers requiring environment variables (e.g., ${ALCHEMY_KEY}) that are not set are silently skipped — not an error

    Rate limiter implementation: - Token bucket algorithm per provider - Configured via rate_limit_rps — defaults to 25 if not specified - Applied transparently by the Chain Manager before every RPC call - Rate limit state is not shared between Anvil instances — each chain has its own limiter

    Pre-flight health check — crucible setup phase: 

    For each chain in testbed.yaml:
  1. Select the highest-priority available provider
  2. Call eth_getBlockByNumber(fork_block, false)
  3. If the call succeeds → provider supports archive state at this block ✓
  4. If the call fails with "missing trie node" or returns null → provider does NOT have archive state
     → Try next provider
     → If no provider has archive state: abort with exit code 5
        "ERROR: No RPC provider for ethereum supports archive state at block 14211979.
         Archive node access is required. See docs/rpc_providers.md"

    Cost estimation — crucible estimate: 

    crucible estimate experiments/wormhole_replay.yaml
# CHAIN       PROVIDER     EST. RPC CALLS    EST. TIME (at rate limit)    COST
# ethereum    llamarpc     ~2,400            ~96 seconds                  free
# polygon     llamarpc     ~800              ~32 seconds                  free
# TOTAL                    ~3,200            ~2 minutes
#
# Note: estimates based on similar scenario profiles. Actual counts may vary.

    The estimator uses historical RPC call counts from previous runs of the same scenario (stored in the experiment index). For first-time scenarios, it uses a heuristic based on the number of steps and chain interactions.

    Operational Modes & RPC Cost Limits: To preserve free-tier RPC quotas and support offline research, Crucible enforces specific operational modes:

    Mode Description Support in V1
    Online fork mode Uses live archive RPC and populates the Anvil fork cache. Required
    Cached fork mode Uses previously populated Anvil cache, bypassing paid RPC during reruns. Required
    Local archive mode Uses a local Erigon/Reth archive node (http://localhost:8545). Supported, but node operation is out-of-scope for Crucible
    Artifact replay mode Uses exported baselines and JSONL traces without fetching chain state. Phase 3 export goal

    To enforce cost control, Crucible reads the rpc_budget from testbed.yaml. If an experiment's estimated RPC calls or USD cost exceeds the budget, the run is aborted before forking unless overridden with --force-budget.

    5.2 Fork Cache Layer¶

    Problem: Re-forking the same (chain, block) pair hits the RPC provider every time. With free-tier providers and multiple researchers, this wastes rate limit budget and makes hard resets unnecessarily slow.

    Solution: Use Anvil's built-in --fork-cache-path flag to cache RPC responses on disk.

    Configuration in testbed.yaml: 

    fork_cache:
  enabled: true
  path: ".fork_cache"               # relative to workspace root
  max_size_gb: 20                   # LRU eviction threshold
  shared: true                      # share cache across researchers on the same machine

    How it works: 1. On first fork of ethereum at block 14211979, Anvil issues RPC calls and caches every response under .fork_cache/ethereum/14211979/ 2. On subsequent forks of the same chain+block, Anvil reads from cache — zero RPC calls 3. This makes hard resets (re-fork from scratch) nearly as fast as soft resets

    Chain Manager integration: 

    # When starting Anvil, the Chain Manager adds the cache path:
anvil_args = [
    "anvil",
    "--fork-url", rpc_url,
    "--fork-block-number", str(fork_block),
    "--fork-cache-path", str(cache_path / chain_name / str(fork_block)),
    "--port", str(port),
]

    Cache invalidation: - Changing fork_block in testbed.yaml does NOT invalidate existing cache entries — different blocks use different cache directories - crucible clean --cache deletes the entire .fork_cache/ directory - The cleanup policy (§7.1) evicts LRU entries when the cache exceeds max_size_gb

    Sharing between researchers: When shared: true, the cache path is absolute (e.g., /shared/crucible_cache/) rather than workspace-relative. Multiple researchers' workspaces can point to the same cache directory, so the first researcher to fork a given block pays the RPC cost — everyone else reads from cache.

    # For shared-machine deployment:
fork_cache:
  enabled: true
  path: "/shared/crucible_cache"    # shared directory with group write permissions
  max_size_gb: 50
  shared: true

    Files: .fork_cache/ is always gitignored. It is machine-local or machine-shared, never committed.

  max_log_age_days: 90           # auto-delete log directories older than this

  max_cache_size_gb: 20          # fork cache eviction threshold
  max_checkpoints: 10            # keep only the last N checkpoint sets per scenario
  warn_at_disk_percent: 85       # emit warning when disk usage exceeds this
  auto_clean_on_setup: false     # if true, run cleanup before every `crucible setup`

    crucible clean --cache Clear the fork cache (.fork_cache/)
    crucible clean --checkpoints Delete all experiment checkpoints
    crucible clean --orphans Kill orphaned Anvil processes + remove stale PID files

    crucible clean --all --dry-run Show the full cleanup plan without executing

    - "bridge_liability_changes"
    - "vulnerable_validation_accepts_invalid_message"

    8.2 Review Gates for Attack Packs¶

    New attack packs are subject to strict academic-quality review gates before inclusion in the benchmark suite:

    1. Manifests: Both plugin.yaml and evidence.yaml pass schema validation.
    2. Evidence: Sources and modeled/excluded assumptions are explicitly declared.
    3. Assertions: At least one positive assertion proves the exploit effect, and one negative assertion proves the baseline was not already compromised.
    4. Repeatability: The determinism test passes on two consecutive runs (canonicalized state diffs match).
    5. Exportability: The exported data bundle includes all necessary reproduction instructions.

    9. Modular Architecture & Plug-and-Play System¶

    This section is the canonical reference for how modularity works in Crucible. Every major subsystem is a replaceable unit. Nothing in crucible/core/ ever imports a specific chain, bridge, or attack by name — it only ever speaks to interfaces.

    9.1 The Module Contract¶

    Every pluggable unit in Crucible — chain, bridge, attack, instrumentation — must satisfy two things:

    1. A plugin.yaml manifest in its root directory
    2. An interface implementation — a Python class inheriting from the correct abstract base

    If both are present and valid, the plugin is discovered and usable. No registration. No config changes. No core code changes.

    9.2 Module Types and Their Interfaces¶

    9.6 Topology as a Composition Rule¶

    The topology profile controls which modules the orchestrator assembles for a given run. When --topology single is set, the ChainManager starts only the chains listed in topologies/single.yaml. The BridgeEngine is never instantiated. When --topology bridge is set, the bridge plugin named in the topology file is loaded, deployed, and connected.

    This means a single attack pack can run against all three topologies if designed to — the orchestrator handles wiring, not the attack.

    10. Comprehensive Logging System¶

    Logging is the primary research output of Crucible. This section specifies every log stream, schema, collection point, and analysis pattern in full. Every developer touching any component must read this section before writing a single line that produces output.

    10.1 Design Principles for Logging¶

    • Every action is logged. There is no "internal" operation that bypasses the logger. Chain start, baseline verify, tx send, revert, gas consumption, bridge message relay, assertion pass/fail — all go through the same bus.
    • Schema is mandatory. Every log entry must have all required fields. Missing a field is a bug, not a style choice.
    • Log first, compute later. Raw data goes to JSONL immediately. Aggregation, diffing, and summarisation are post-processing tasks — they read from the JSONL, they do not replace it.
    • Correlation ID chains everything. Every entry in every log file for a given run carries the same run_id. Every entry for a cross-chain sequence carries the same correlation_id. You must be able to reconstruct the complete sequence of events across chains from the log files alone.

    10.3 Log Emission Architecture¶

    Every component emits through a single LogEmitter instance injected at construction. Components never write directly to files.

    # crucible/logging/emitter.py

class LogEmitter:
    def __init__(self, run_id: str, session_id: str, scenario: str):
        self._run_id    = run_id
        self._writers   = {}     # stream_name → FileWriter

    def emit(self, stream: str, event_type: str, **fields) -> None:
        """Write a structured entry to the named stream."""

    def step(self, message: str, **fields) -> None:
        """Shorthand: emit to run.jsonl with action=step."""

    def tx(self, tx_hash: str, chain: str, **fields) -> None:
        """Shorthand: emit to run.jsonl with action=send_transaction."""

    def event(self, contract: str, event_name: str, args: dict, **fields) -> None:
        """Emit to events.jsonl."""

    def storage_diff(self, contract: str, slot: str,
                     before: str, after: str, **fields) -> None:
        """Emit to storage_diffs.jsonl."""

    def gas(self, step: str, tx_hash: str, gas_used: int, **fields) -> None:
        """Emit to gas_profile.jsonl."""

    def xchain(self, correlation_id: str, source_chain: str,
               dest_chain: str, **fields) -> None:
        """Emit to cross_chain.jsonl."""

    def flush(self) -> None:
        """Flush all writers. Called at end of every phase."""

    10.4 Correlation IDs for Cross-Chain Sequences¶

    When an attack involves a message flowing from Chain A to Chain B via a bridge, every log entry for that message sequence must share a correlation_id. The format is xchain-<NNNN> where NNNN is a per-run incrementing counter.

    The bridge engine assigns correlation_id when it detects a message publication on the source chain. It then tags all relay events and destination chain events with the same ID. Researchers can use this to reconstruct the complete cross-chain timeline:

    # Reconstruct the full sequence for message xchain-0001
cat logs/wormhole_replay/.../run.jsonl \
  logs/wormhole_replay/.../cross_chain.jsonl \
  | jq 'select(.correlation_id=="xchain-0001")' \
  | jq -s 'sort_by(.timestamp)'

    10.8 State Diff Methodology¶

    State diffs (storage_diffs.jsonl) are the primary evidence of an exploit's impact. To ensure accurate academic reporting, Crucible applies the following methodology:

    Scope Selection: - Mode: allowlist_plus_touched. - All explicitly configured contracts (the bridge, target tokens) are diffed. - Any contract touched during the transaction execution is automatically added to the diff scope.

    Comparison and Canonicalization Rules: Byte-for-byte JSON equality of the storage_diffs.jsonl file is not used for determinism checks, as JSON key ordering, internal pointers, and timestamps can introduce false negatives. Instead, Crucible canonicalizes the diffs: 1. Balances: ETH and ERC20 balances of the attacker, bridge vaults, and victims are extracted and compared exactly. 2. Storage Slots: The set of modified slots and their value_after must match exactly across runs. Unknown slots are included in the diff but flagged as unlabeled. 3. Transaction Outcomes: Status codes and revert reasons must match exactly.

├── patch.yaml          # Metadata + what the patch changes
├── patch.sol           # The fixed contract(s)
└── validation.yaml     # Experiment overrides for the patched run

    validation.yaml: 

    base_experiment: experiments/wormhole_replay.yaml
patch: wormhole_guardian_fix

expect:
  attack_outcome:    "fail"    # attack must fail on patched system
  normal_operation:  "pass"    # bridge transfers must still work

    What this does internally: 

    1. Run base experiment (unpatched)   → record outcome as "attack_succeeds"
2. Deploy patch contracts via forge script
3. Soft reset to baseline
4. Re-run same attack against patched contracts
5. Assert: attack_outcome == "fail"
6. Run normal_operation probe (standard bridge transfer)
7. Assert: normal_operation == "pass"
8. Emit side-by-side diff report: patched vs unpatched

import pandas as pd
from pathlib import Path

def load_run(run_id: str, base_path: str = "logs") -> pd.DataFrame:
    """Load run.jsonl for a given run_id into a DataFrame."""
    path = _find_run_dir(run_id, base_path) / "run.jsonl"
    return pd.read_json(path, lines=True)

def load_events(run_id: str, base_path: str = "logs") -> pd.DataFrame:
    """Load events.jsonl — decoded contract events."""
    path = _find_run_dir(run_id, base_path) / "events.jsonl"
    return pd.read_json(path, lines=True)

def load_storage_diffs(run_id: str, base_path: str = "logs") -> pd.DataFrame:
    """Load storage_diffs.jsonl — per-tx storage slot changes."""
    path = _find_run_dir(run_id, base_path) / "storage_diffs.jsonl"
    return pd.read_json(path, lines=True)

def load_gas(run_id: str, base_path: str = "logs") -> pd.DataFrame:
    """Load gas_profile.jsonl — gas accounting per step."""
    path = _find_run_dir(run_id, base_path) / "gas_profile.jsonl"
    return pd.read_json(path, lines=True)

def load_cross_chain(run_id: str, base_path: str = "logs") -> pd.DataFrame:
    """Load cross_chain.jsonl — bridge topology runs only."""
    path = _find_run_dir(run_id, base_path) / "cross_chain.jsonl"
    return pd.read_json(path, lines=True)

def load_manifest(run_id: str, base_path: str = "logs") -> dict:
    """Load run_manifest.json as a dictionary."""
    import json
    path = _find_run_dir(run_id, base_path) / "run_manifest.json"
    return json.loads(path.read_text())

def compare_runs(run_a: str, run_b: str, base_path: str = "logs") -> dict:
    """Compare gas, storage diffs, and outcomes between two runs.
    Returns a dict with keys: gas_diff, storage_diff, outcome_match."""
    gas_a = load_gas(run_a, base_path)
    gas_b = load_gas(run_b, base_path)
    manifest_a = load_manifest(run_a, base_path)
    manifest_b = load_manifest(run_b, base_path)
    return {
        "gas_diff": gas_a["gas_used"].sum() - gas_b["gas_used"].sum(),
        "outcome_match": manifest_a["outcome"] == manifest_b["outcome"],
        "run_a": {"total_gas": gas_a["gas_used"].sum(), "outcome": manifest_a["outcome"]},
        "run_b": {"total_gas": gas_b["gas_used"].sum(), "outcome": manifest_b["outcome"]},
    }

def _find_run_dir(run_id: str, base_path: str) -> Path:
    """Locate the log directory for a run_id by scanning log subdirectories."""
    base = Path(base_path)
    for manifest in base.rglob("run_manifest.json"):
        import json
        data = json.loads(manifest.read_text())
        if data.get("run_id") == run_id:
            return manifest.parent
    raise FileNotFoundError(f"No log directory found for run_id: {run_id}")

    14. Bootstrap Contract & API¶

    bootstrap.py is an optional script executed once during crucible setup, after all chains are forked, before the golden baseline snapshot is taken. Its purpose is to bring the fork into the exact pre-attack state — deploying any missing contracts, funding accounts, configuring bridge state.

    Location: project root — bootstrap.py

    Contract: bootstrap.py must define a run(ctx: ExperimentContext) -> None function. It receives the same context object as steps.py.

    Complete bootstrap template: 

    # bootstrap.py — executed once during `crucible setup`
# Purpose: bring all chains into the desired pre-attack state.
# Rules:
#   - Use cheatcodes only (anvil_setBalance, impersonateAccount)
#   - Do NOT make calls that depend on DEX liquidity or live market state
#   - Do NOT make calls that could fail if external state is missing at the fork block
#   - Always log every action via ctx.log so it appears in the setup phase log

from crucible.sdk import ExperimentContext

def run(ctx: ExperimentContext) -> None:

    eth = ctx.chain("ethereum")
    pol = ctx.chain("polygon")

    # 1. Fund the attacker account on both chains
    eth.set_balance("0xAttacker...", 100 * 10**18)   # 100 ETH
    pol.set_balance("0xAttacker...", 100 * 10**18)   # 100 MATIC
    ctx.log.step("Funded attacker accounts")

    # 2. Deploy a mock token if not already present at fork block
    # (Only needed if the fork pre-dates the token's deployment)
    # deploy_receipt = eth.send_transaction(
    #     to=None,                     # contract creation
    #     data=MOCK_TOKEN_BYTECODE,
    #     sender="0xDeployer...",
    #     value=0
    # )
    # ctx.log.step("Deployed mock token", address=deploy_receipt.contract_address)

    # 3. Configure the bridge guardian set if not set at fork block
    eth.impersonate("0xWormholeGovernance...")
    tx = eth.send_transaction(
        to="0xWormholeCore...",
        data=encode_set_guardian_set(GUARDIAN_KEYS),
        sender="0xWormholeGovernance...",
        value=0
    )
    ctx.log.step("Guardian set configured", tx_hash=tx.hash)

    # 4. Approve bridge to spend attacker's tokens
    eth.impersonate("0xAttacker...")
    eth.send_transaction(
        to="0xWETH...",
        data=encode_approve("0xWormholeBridge...", 2**256 - 1),
        sender="0xAttacker...",
        value=0
    )
    ctx.log.step("Token approval granted")

    Rules: 1. Only use Anvil cheatcodes for state setup — set_balance, impersonate, set_code, set_storage 2. Never make calls that depend on DEX liquidity or external price feeds at the fork block 3. Every action must be logged via ctx.log.step() 4. bootstrap.py must be idempotent — running it twice must not corrupt state 5. If bootstrap fails, crucible setup exits with code 4 and prints the exact error

    16. CLI Exit Codes¶

    Every crucible command exits with a documented code. CI pipelines and shell scripts must rely on these — never parse stdout.

    Code Name Meaning
    0 SUCCESS Command completed successfully
    1 GENERAL_ERROR Unexpected error — check stderr for details
    2 CONFIG_ERROR testbed.yaml or experiment YAML is invalid
    3 BASELINE_MISMATCH Golden baseline verification failed — run aborted
    4 BOOTSTRAP_FAILED bootstrap.py raised an exception during setup
    5 CHAIN_START_FAILED Anvil failed to start or RPC is unreachable
    6 SCENARIO_NOT_FOUND Named scenario or attack pack does not exist
    7 ASSERTION_FAILED expected_state.json assertions failed post-attack
    8 PLUGIN_INVALID A plugin failed manifest or interface validation
    9 EXPERIMENT_FAILED Experiment ran but attack did not produce expected outcome

    Usage in CI: 

    # .github/workflows/nightly.yml
- name: Run Wormhole replay
  run: crucible experiment run experiments/wormhole_replay.yaml
  # exits 0 on success, 3 if baseline drifted, 7 if assertions failed
  # any non-zero exit fails the CI step automatically

    Usage in shell scripts: 

    crucible experiment run experiments/wormhole_replay.yaml
EXIT=$?
case $EXIT in
  0) echo "Attack replayed successfully" ;;
  3) echo "ERROR: Baseline mismatch — environment is dirty" ; exit 1 ;;
  7) echo "ERROR: Post-attack assertions failed" ; exit 1 ;;
  *) echo "ERROR: Unexpected exit code $EXIT" ; exit 1 ;;
esac

    17.5 Multi-Researcher Workspace Isolation¶

    When multiple researchers share the same machine or codebase, isolation is critical. Crucible provides three layers of protection.

    Layer 1: testbed.local.yaml — per-researcher config override

    Each researcher creates a testbed.local.yaml (gitignored) that overrides ports, RPC URLs, and identity without touching the shared testbed.yaml:

    # testbed.local.yaml — Alice's overrides (gitignored, never committed)
researcher: "alice"

chains:
  ethereum:
    port: 8545     # Alice uses default ports
  polygon:
    port: 8546

# Bob's testbed.local.yaml would use:
# chains:
#   ethereum:
#     port: 9545
#   polygon:
#     port: 9546

    Override merge order: testbed.yaml ← testbed.local.yaml ← environment variables ← CLI flags. Each layer overrides the previous. testbed.yaml is the shared baseline; testbed.local.yaml is per-researcher.

    Layer 2: Port allocation convention

    To prevent ad-hoc port selection from colliding, follow this convention:

    Researcher Index Anvil Port Range Description
    0 (default) 8545–8599 Default ports in testbed.yaml
    1 8600–8699 Override in testbed.local.yaml
    2 8700–8799 Override in testbed.local.yaml
    N 8500 + (N+1)×100 Computed

    crucible init --researcher-index 2 scaffolds a testbed.local.yaml with the correct port range.

    22. Technical Gotchas¶

    Archive node required. RPC endpoints must support eth_getBlockByNumber with full historical state. Crucible validates this on startup and exits with code 5 if not satisfied.

    Port conflicts. Chain Manager reads ports from testbed.yaml. If a port is occupied, it exits with code 5 and prints the PID of the conflicting process.

    Chain ID uniqueness. Every Anvil instance must have a unique --chain-id. Without this, cast and web3.py will route transactions to the wrong chain silently. Always set explicitly in testbed.yaml.

    State file size. anvil_dumpState outputs 100–500 MB per chain. The states/ directory is .gitignored. Only baseline YAML fingerprints (a few KB) are committed.

    Time manipulation. evm_increaseTime behaviour varies across Foundry versions. Pin the version in foundry.toml and use the same pin in Docker Compose. Never run foundryup without a version hash.

    Bootstrap must use cheatcodes. bootstrap.py must not rely on DEX liquidity, price feeds, or any external state that may not exist at the fork block. Use anvil_setBalance, anvil_impersonateAccount, anvil_setCode, and anvil_setStorageAt exclusively for setup operations.

    Fork cache invalidation. Changing fork_block in testbed.yaml does not automatically invalidate .fork_cache/. Delete the relevant cache entries manually or run crucible clean --cache.

    Foundry --broadcast against local Anvil. The scenario engine sets ETH_RPC_URL to the Anvil URL before any forge script invocation. Never hardcode RPC URLs in Forge scripts — always read from the environment.

    anvil_dumpState vs evm_snapshot. These are different mechanisms. evm_snapshot returns an integer ID and lives only in memory — it is lost if Anvil restarts. anvil_dumpState writes to disk and survives restarts. Crucible uses anvil_dumpState for all persistent baseline and named snapshots.

    Log schema compliance. If a new log field is added, update crucible/logging/schema.py before opening a PR. The schema test runs on every CI build — a missing field definition will break the build.

    RPC rate limiting. Free-tier RPC providers (LlamaRPC, Ankr, PublicNode) enforce 10–25 requests/second. During initial fork, Anvil issues hundreds of eth_getStorageAt calls. If the provider returns 429 (Too Many Requests), Anvil silently retries but slows to a crawl. Use --fork-retry-backoff and configure the provider rate limit in testbed.yaml (see §5.1).

    Orphaned Anvil processes. If the orchestrator crashes (segfault, OOM, Ctrl+C during startup), Anvil subprocesses may survive as orphans. They hold ports and consume RAM indefinitely. Always run crucible status before starting a new session — it detects orphans via PID files. See §22.1 for the cleanup protocol.

    ABI discovery for event decoding. The logger decodes contract events using ABIs from abis/<protocol>/. If no ABI is found for a contract address, events are logged as raw hex with decoded: false. This is not an error — but decoded events are vastly more useful for research. Always add ABIs for new bridge contracts before running experiments. See §10.7.

    Disk space exhaustion. A single anvil_dumpState produces 100–500 MB. With 6 benchmark attacks × multiple snapshots × 4–10 researchers, the states/ directory can exceed 50 GB. Monitor with crucible status --disk and configure auto-cleanup policies in testbed.yaml. See §7.1.

    Port conflicts between researchers. On a shared machine, two researchers running crucible setup with default ports (8545, 8546) will collide. Each researcher must configure unique ports via testbed.local.yaml (see §17.5) or use the Docker-based isolation strategy.

    22.1 Graceful Shutdown & Orphan Cleanup¶

    Purpose: Ensure that Crucible always cleans up after itself — whether a run succeeds, fails, or crashes mid-experiment.

    Signal Handling¶

    The CLI orchestrator must trap the following signals:

    Signal Trigger Crucible Behavior
    SIGINT Ctrl+C Stop current step, flush logs, save checkpoint, kill Anvil processes, exit 130
    SIGTERM kill <pid>, Docker stop Same as SIGINT — graceful shutdown
    SIGKILL kill -9, OOM killer Cannot trap — orphan cleanup handles this case on next startup

    Shutdown sequence (on SIGINT/SIGTERM): 

    1. Set global shutdown flag — no new steps will start
2. Wait for current step to complete (max 10 seconds, then force-kill)
3. Flush all LogEmitter writers
4. Save checkpoint snapshot if experiment was in progress
   → states/checkpoints/<run_id>/interrupted_phase_<N>.json
5. Log a "run_interrupted" entry to run.jsonl with exit reason
6. Kill all managed Anvil subprocesses (SIGTERM, then SIGKILL after 5s)
7. Remove PID files
8. Write partial run_manifest.json with outcome: "interrupted"
9. Exit with code 130

    Implementation: 

    # crucible/shutdown.py
import signal
import sys

class GracefulShutdown:
    def __init__(self):
        self._shutdown_requested = False
        signal.signal(signal.SIGINT, self._handler)
        signal.signal(signal.SIGTERM, self._handler)

    def _handler(self, signum, frame):
        if self._shutdown_requested:
            # Second signal — force exit
            sys.exit(1)
        self._shutdown_requested = True

    @property
    def requested(self) -> bool:
        return self._shutdown_requested

    Orphan Process Detection & Cleanup¶

    On every crucible status and crucible setup, the Chain Manager checks for orphaned Anvil processes:

    1. Scan states/*.meta.json for PID files
2. For each PID file:
   a. Check if the process is alive (os.kill(pid, 0))
   b. If alive but no active Crucible session owns it → orphan
   c. If dead but PID file exists → stale PID file
3. Report findings:
   [warn] Orphaned Anvil process: PID 12345 (ethereum, port 8545) — started 3 hours ago
   [warn] Stale PID file: states/polygon.meta.json — process no longer running
4. Offer cleanup:
   crucible clean --orphans    # kill orphaned processes + remove stale PID files

    Orphan cleanup is also run automatically at the start of crucible setup. If orphans are found: - In interactive mode: prompt the researcher to confirm cleanup - In non-interactive mode (--yes flag or CI): clean automatically and log a warning

    Lock File for Concurrent Access¶

    To prevent two researchers from accidentally running experiments against the same Anvil instances:

    .crucible.lock — created by crucible setup, removed by crucible clean
Contents: { "researcher": "alice", "pid": 54321, "started": "2026-06-05T10:00:00Z" }

    If crucible setup finds an existing lock file: - Check if the owning process is alive - If alive: abort with message "Testbed is locked by alice (PID 54321, started 2 hours ago). Run crucible clean first or use a separate workspace." - If dead: warn and auto-clean the stale lock
