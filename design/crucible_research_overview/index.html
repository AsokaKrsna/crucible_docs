    Crucible¶

    A Modular Blockchain Bridge Security Research Testbed¶

    Research Group Project | Status: In Development | Version: 1.0

    What We Are Building¶

    Crucible is a reproducible, CLI-first research testbed for systematic security analysis of blockchain bridges. It enables controlled replay of historical bridge exploits, structured telemetry collection, and patch validation — all from a single, version-controlled configuration.

    The primary research objective is to create a rigorous, repeatable experimental platform for studying the attack surface of cross-chain bridge protocols, with a target benchmark suite of six major historical exploits totalling over $1.9 billion in real-world losses.

    Motivation¶

    Cross-chain bridge protocols represent the most financially damaging category of smart contract exploits to date. Despite this, reproducible research infrastructure for studying these attacks is essentially non-existent. Researchers typically reconstruct attacks manually against live testnets — a process that is slow, non-deterministic, and impossible to compare across experiments.

    Crucible addresses this gap by providing: - Exact historical state reconstruction via block-level EVM forking (Foundry/Anvil) - A verified baseline system that guarantees experiment starting conditions are identical across runs and machines - Structured telemetry that produces publication-quality datasets from every run

    The architecture is modular and composable. Researchers select a topology profile — single chain, dual chain, or full bridge — and the system assembles only the necessary components. Adding a new attack requires dropping a folder into the attacks/ directory; no core code changes.

    Key Design Properties¶

    Reproducibility. Every experiment is defined entirely in YAML and anchored to a specific git commit, Foundry version, fork block, and baseline fingerprint. Given these four things, any run can be independently reproduced on any machine.

    Immutable Baselines. Before every experiment, Crucible computes a cryptographic fingerprint of the chain state (block number, state hash, balance hash, storage hash) and compares it against a stored golden baseline. If they do not match, the experiment aborts. This guarantees that every experiment starts from an identical, verified state.

    One-click Reset. Resetting to the clean baseline takes under 5 seconds (Anvil state restore) without restarting any process. This is essential for rapid iteration across experiment phases.

    Scenario-as-Code. Attacks, environments, and experiments are version-controlled artifacts — not shell scripts. Every run produces a run_manifest.json embedding git commit, scenario version, baseline fingerprint, and researcher identity. This is the foundation for publication-quality provenance.

    Research Outputs Per Experiment Run¶

    Every run automatically produces:

    Artifact Contents
    run.jsonl Timestamped, structured log of every action
    events.jsonl Decoded contract events (transfers, bridge messages)
    storage_diffs.jsonl Per-transaction storage slot changes
    run_manifest.json Full provenance record — git, versions, fingerprint, outcome
    report.md Auto-generated human-readable summary

    These outputs are designed for direct ingestion into pandas, jq, or any analytics pipeline. The dataset versioning system assigns each run a monotonic ID (DS-2026-NNNNN) for unambiguous citation in research notes and publications.

    Benchmark Target Suite¶

    The platform is considered complete when all six attacks replay deterministically:

    Exploit Bridge Year Reported Loss
    Guardian signature bypass Wormhole 2022 $326M
    Replica fraud message Nomad 2022 $190M
    Private key compromise Poly Network 2021 $611M
    Multi-sig validator drain Horizon (Harmony) 2022 $100M
    Router contract exploit Multichain 2023 $125M
    Token hub vulnerability BNB Bridge 2022 $570M

    These cases collectively cover the main bridge attack categories: signature/validation bypass, message fraud, key management failure, and smart contract logic errors.

    Technology Stack¶

    Layer Technology
    EVM forking & simulation Foundry / Anvil
    Smart contract scripting Forge scripts (Solidity)
    Orchestration & CLI Python (Typer)
    Structured logging structlog → JSONL
    Container environment Docker Compose
    UI wrapper Streamlit (Phase 4)
    Version control Git — all scenarios, experiments, and baselines

    Development Phases¶

    Phase Scope Outcome
    1 — Core Chain Manager, CLI, baseline system, 3-tier reset, structured logging Running forks with verified baselines
    2 — Scenarios Scenario Engine, topology profiles, bridge engine, experiment runner Executable attack scenarios
    3 — Research Dataset versioning, plugin SDK, all 6 benchmark attacks Publication-ready experiment outputs
    4 — Polish Matrix runner, Streamlit UI, CI integration, export command Fully automated research pipeline

    Scope Boundaries¶

    In scope for V1: Foundry/Anvil integration, historical EVM chain forks, multi-chain environments, bridge simulation, attack replay, snapshot management, structured telemetry, dataset generation, patch validation.

    Deferred to V2: Kubernetes deployment, real-time production monitoring, AI-assisted analysis, non-EVM chains (Solana, Cosmos, Aptos).

    Crucible — controlled conditions for finding where things break.
