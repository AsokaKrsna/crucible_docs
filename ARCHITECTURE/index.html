    Crucible Architecture¶

    Version: 0.4.0 Â· Last Updated: June 2026 Â· Audience: Developers, contributors, reviewers

    Crucible is a modular EVM bridge security research testbed. It forks historical blockchain state using Foundry's Anvil, runs exploit reproductions as plugins, and automatically captures publication-grade artifacts for academic research.

    Looking for a broader overview? See the Technical Overview for system philosophy, core concepts, and design principles, or the Design Decisions document for the rationale behind every major architectural choice.

    Key Design Property¶

    The attack plugin never knows it's instrumented. When start_instrumentation() is called, ChainManager stores the proxy reference and runtime_for() silently returns the proxy URL instead of the direct Anvil URL:

    # chain_manager.py â€” transparent URL substitution
def runtime_for(self, chain_name):
    ...
    if chain_name in self._proxies:
        rpc_url = self._proxies[chain_name].proxy_url  # â† proxy, not Anvil
    return ChainRuntime(chain_name, rpc_url, ...)

    This means any attack â€” Python, Forge script, or cast send â€” produces complete artifacts without per-attack instrumentation code.

    Multi-Chain Proxy Isolation¶

    Each chain gets its own proxy instance with its own handler class:

    Ethereum: Attack â†’ Proxy :9001 â†’ Anvil :8545
BSC:      Attack â†’ Proxy :9002 â†’ Anvil :8547
Polygon:  Attack â†’ Proxy :9003 â†’ Anvil :8546

    The _make_handler() closure captures upstream_url, log, and chain per instance, preventing shared-state bugs across chains.

    Shutdown Sequence¶

    1. proxy.drain()    â†’ ThreadPoolExecutor.shutdown(wait=True)
                      (all pending traces complete)
2. proxy.stop()     â†’ HTTPServer.shutdown() + server_close()
3. log.write_integrity()  â†’ close file handles + hash files

    drain() must be called before stop() and before log handles close. This ensures all async trace tasks complete before artifacts are finalized.

    7 JSONL Artifact Streams¶

    The testbed produces 6 core streams managed by LogEmitter through the standard emit API, plus 1 agent-specific stream written directly by the LLM agent plugin:

    Stream Content Written By
    run.jsonl Experiment lifecycle (phase start/end, errors) LogManager.step()
    events.jsonl Contract events (Transfer, Approval, custom) LogEmitter.event() via proxy
    storage_diffs.jsonl Storage slot changes (contract, slot, before, after) LogEmitter.storage_diff() via proxy
    gas_profile.jsonl Gas used per transaction LogEmitter.gas() via proxy
    balance_diffs.jsonl Balance changes (ETH and token) LogEmitter.balance_diff() via proxy
    cross_chain.jsonl Bridge message correlation IDs LogEmitter.cross_chain()
    agent_reasoning.jsonl LLM agent reasoning traces (observation, action, conversation snapshot) â€” written outside the standard LogEmitter system, so not tracked in log_integrity.json LLMAgentPlugin._log_reasoning_step()

    LogManager is the experiment-level orchestrator. It creates per-chain LogEmitter instances lazily via for_chain(). Attack plugins interact with context.log (which is a LogManager) â€” lifecycle events go to the root emitter, while chain-specific data routes to the per-chain subdirectory.

    Diff Engine â€” Determinism Verification¶

    crucible diff <run-a> <run-b> canonicalizes and compares four metrics:

    flowchart LR
    A["Run A"] --> CA["Canonical Sort"]
    B["Run B"] --> CB["Canonical Sort"]

    CA --> CMP["Compare"]
    CB --> CMP

    CMP --> S["storage_equal: true/false"]
    CMP --> E["events_equal: true/false"]
    CMP --> G["gas_equal: true/false"]
    CMP --> BL["balances_equal: true/false"]

    All four must be true for deterministic reproduction.

    Note: The 6 core streams appear in STREAMS in crucible/logging.py. The 7th (agent_reasoning.jsonl) is emitted by LLMAgentPlugin directly and bypasses integrity hashing â€” this is a known gap.

    17. Key Design Decisions¶

    For a deeper treatment of each decision â€” including the tradeoffs considered and why the current path was chosen â€” see the Design Decisions document.

    CLI-First, UI as Wrapper¶

    Every action is available via crucible <command>. The Streamlit dashboard is a separate read-only viewer. This makes experiments scriptable, CI-able, and Docker-friendly.

    Topology-Driven Composition¶

    Topology YAML files declare which chains to fork. The same attack can run on different topologies. Adding a new chain config doesn't affect existing experiments.

    # topologies/single.yaml â€” Ethereum only
name: single
chains: [ethereum]
bridge: null

# topologies/bridge.yaml â€” Two chains + bridge (demo)
name: bridge
chains: [ethereum, polygon]
bridge: demo_bridge

# topologies/nomad_bridge.yaml â€” Two chains + Nomad bridge plugin
name: nomad_bridge
chains: [ethereum, polygon]
bridge: nomad_bridge

# topologies/wormhole_bridge.yaml â€” Two chains + Wormhole bridge plugin
name: wormhole_bridge
chains: [ethereum, polygon]
bridge: wormhole_bridge

# topologies/poly_network_bridge.yaml â€” Three chains + Poly Network bridge plugin
name: poly_network_bridge
chains: [ethereum, polygon, bsc]
bridge: poly_network_bridge

    Transparent Instrumentation¶

    The instrumentation proxy intercepts all RPC traffic. Researchers don't need to add logging code. Any attack method (Python, Forge, cast send) produces complete artifacts.

    Snapshot-Driven Reset¶

    Baselines are captured once after setup. Soft resets use anvil_loadState â€” no chain restart needed. This enables fast iteration during development.

    Determinism by Default¶

    Deterministic flags (gas_price: 0, order: fifo, timestamp: fork_block) are opt-out, not opt-in. If a researcher does nothing, their experiments are deterministic.

    Plugins for Capabilities¶

    New functionality arrives as plugins, not core rewrites: - attack â€” exploit reproductions - bridge â€” bridge protocol simulation - agent â€” LLM-driven autonomous agents

    Fidelity Levels Are Enforced¶

    The L0â€“L4 fidelity system isn't just documentation â€” publication gates check requirements programmatically. An L2 reproduction without affected_contracts will fail the gate.

    Separation of Scenario and Experiment¶

    Scenarios define environments (chain state, contract setup). Experiments pair a scenario + attack plugin for formal runs. This means the same scenario can be used with different attacks, and the same attack can run against different scenarios.
