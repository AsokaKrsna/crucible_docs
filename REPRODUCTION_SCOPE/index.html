    Reproduction Scope & Capabilities¶

    Purpose: Help researchers understand what Crucible can reproduce, at what fidelity, and what's in vs out of scope before planning a study. Last updated: June 2026

    What Crucible Can Reproduce¶

    Crucible is an EVM-only bridge security research testbed. It reproduces historical bridge exploits by forking blockchain state at a specific block using Foundry's Anvil and running attack plugins against the forked environment.

    Core Capability¶

    Archive RPC → Fork EVM chain at block N → Run attack plugin → Capture artifacts
    Capability Supported? Notes
    Fork any EVM chain [ok] Yes Ethereum, BSC, Polygon, Harmony — any chain with an archive RPC endpoint
    Fork at any historical block [ok] Yes Requires archive node access at that block depth
    Replay exploit transactions [ok] Yes L3: forge exact calldata and replay on fork
    Behavioral reproduction [ok] Yes L2: verify vulnerable state exists and exploit path works
    Multi-chain reproduction [ok] Supported via 3 bridge plugins Nomad (2 chains), Poly Network (3 chains), Wormhole (2 chains) — see bridge topology files
    Deploy fresh bridge contracts on blank chains [ok] Yes NomadBridge, PolyNetworkBridge, WormholeBridge plugins deploy real Solidity contracts to Anvil forks
    Forge cross-chain messages [ok] Yes bridge_plugin.send_message() with forged calldata — captured in cross_chain.jsonl
    Non-EVM chains (Solana, Cosmos) [fail] No Crucible is EVM-only by design
    Off-chain infrastructure [fail] No Relayers, keepers, MPC nodes, governance — must be modeled or excluded

    Fidelity Levels Supported¶

    Level What It Means Crucible Support Example
    L4 Full multi-chain + bridge simulation [ok] Supported via 3 bridge plugins — Nomad (2 chains), Poly Network (3 chains), Wormhole (2 chains). Requires paid archive RPCs for BSC/Polygon at 2021-era blocks Poly Network (all 3 chains), Nomad, Wormhole
    L3 Exact historical transaction replay [ok] Supported — forge calldata, replay on fork Nomad process() call
    L2 Fork-state behavioral reproduction [ok] Primary use case — verify vulnerability exists Wormhole guardian bypass
    L1 Conceptual / narrative reproduction [ok] Supported — deploy vulnerable contracts locally Multichain router permit (MPC compromise)
    L0 Description only [ok] Supported — no code required Documentation-only

    Current Reproduction Status¶

    Complete (L2 — Research-Grade)¶

    These attacks are fully reproducible with determinism proven:

    Attack Chains Fork Block Fidelity Key Limitation
    Nomad Bridge (nomad_replica_fraud) Ethereum 15,259,100 L2 [ok] Doesn't replay exact historical tx (uses reconstructed forged message — proxy ABI resolution needed for exact replay)
    Wormhole Guardian Bypass (wormhole_guardian_bypass) Ethereum 14,125,553 L2 [ok] Root cause was Solana-side (deprecated sysvar); only EVM-side effects reproducible
    Poly Network PrivKey (poly_network_privkey) Ethereum + Polygon + BSC 12,996,693 L2 [ok] 3-chain topology available and determinism proven. BSC + Polygon at exploit blocks require paid archive RPCs for full multi-chain run
    Horizon Multisig Drain (horizon_multisig_drain) Ethereum 15,005,526 L2 [ok] Infrastructure compromise (not a contract bug) — replaying Ethereum tx succeeds but proves nothing about the vulnerability. Harmony chain (non-EVM) not forkable, blocking L4
    BNB Token Hub Proof (bnb_token_hub_proof) BSC 22,166,659 L2 [ok] CrossChain contract (0x...2000) identified as the real handlePackage() target. Function selector corrected to 0x84013b6a. Requires paid BSC archive RPC (block 22M exceeds free tier)

    Complete (L1 — Narrative / Infrastructure)¶

    These attacks are correctly classified at their maximum possible fidelity:

    Attack Chains Fork Block Fidelity Key Limitation
    Multichain Router Permit (multichain_router_permit) Ethereum (narrative) — L1 📝 Maxed at L1. This was an MPC node infrastructure compromise, not a smart contract bug. No on-chain reproduction possible

    Chain Archive RPC Available? Fork Block Range Cost
    Ethereum [ok] Free (LlamaNodes, Ankr) 0–current Free

    GetBlock Starter ($39/month) — covers BSC + Polygon archive. Ethereum works on free LlamaNodes.

    What's In Scope¶

    Attack Types Crucible Can Reproduce¶

    Type Example Works?
    Smart contract logic bugs Nomad committedRoot = 0x00 [ok]
    Access control bypass Wormhole guardian signature verification [ok]
    Hash collision / selector confusion Poly Network verifyHeaderAndExecuteTx / putCurEpochConPubKeyBytes [ok]
    Message forgery (Merkle proof) Nomad, BNB TokenHub proof bypass [ok]
    Compromised key / privilege abuse Horizon multisig drain [warn] (infrastructure, not contract bug)
    Re-entrancy on bridge contracts Standard EVM re-entrancy [ok]
    Flash loan + price manipulation On forked chain with sufficient state [ok]

    Chains Crucible Can Fork¶

    • Any EVM-compatible chain with an archive-capable RPC endpoint
    • Tested: Ethereum (mainnet), BSC, Polygon, Harmony
    • Multi-chain experiments supported via bridge topology

    What's Out of Scope¶

    Category Why
    Non-EVM chains (Solana, Cosmos, Near, etc.) Crucible's ChainManager launches Anvil (EVM-only). Non-EVM reproduction requires separate tooling
    Off-chain infrastructure Relayers, keepers, MPC nodes, governance processes — Crucible can't simulate these. They must be declared in excluded_components in evidence.yaml
    Live mainnet interaction Crucible forks are isolated — no transactions ever reach the real network. By design, not a limitation
    Real-time mempool monitoring Crucible is a replay tool, not a monitoring tool
    Social engineering / phishing Out of scope for a technical testbed

    How To Plan A Study¶

    Step 1: Identify Your Target¶

    # Example: You want to study bridge message verification vulnerabilities
# Which attacks are relevant?
- Nomad Bridge (forged Merkle proof) — [ok] L2 complete
- Wormhole (guardian signature bypass) — [ok] L2 complete  
- BNB TokenHub (IAVL proof bypass) — 🚧 scaffold, needs research fill

    Step 2: Check Chain Requirements¶

    If your target is on... You need...
    Ethereum only Free RPC (LlamaNodes)
    Ethereum + Polygon Paid RPC for Polygon archive
    BSC only Paid RPC for BSC archive
    Harmony Free RPC (Ankr)

    Step 3: Choose Fidelity Level¶

    If you want to... Target fidelity
    Prove a vulnerability exists L2
    Show exactly how the exploit executed L3
    Demonstrate cross-chain impact L4

    Step 4: Find or Create the Attack Pack¶

    # Check if it exists
ls attacks/
crucible plugin list --type attack

# If it's a scaffold, fill in:
crucible plugin validate attacks/<name>  # check what's missing
# Then update evidence.yaml, scenario.yaml, and plugin code

    See docs/ATTACK_PACK_GUIDE.md for the complete guide.

    Step 5: Run With Publication Gates¶

    # Run with publication mode for research-grade gates
crucible experiment run experiments/<name>.yaml --publication-mode

    Reproducibility Guarantees¶

    Claim Detail
    Bit-for-bit identical runs Same topology, same fork block, same attack → same gas, events, storage, balances [ok]
    Cross-provider reproduction Same results across different archive RPC providers (LlamaNodes vs Alchemy) [ok]
    Cross-environment Same results in Docker and native local [ok]
    Cross-machine Same results on different machines with same git commit [ok]

    What's NOT guaranteed identical¶

    • Cache timing (cold vs warm fork)
    • RPC provider response ordering (doesn't affect on-chain state)
    • Cross-chain relay timing (multi-chain experiments)
    • Artifact file ordering (mitigated by canonical sort in diff)
