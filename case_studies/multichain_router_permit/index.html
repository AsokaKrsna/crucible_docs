    Case Study: Multichain MPC Node Compromise¶

    Fidelity Level: L1 (Narrative ΓÇö Infrastructure Compromise) Date: July 7, 2023 Loss: ~$1.4B (total value drained across chains) Chains: Fantom, Moonriver, Dogechain (MPC infrastructure)

    Overview¶

    The Multichain (formerly Anyswap) bridge was exploited on July 7, 2023 when an attacker compromised the project's MPC (Multi-Party Computation) node key shards. This was not a smart contract vulnerability ΓÇö the bridge contracts functioned exactly as designed. The attacker simply had legitimate access to the keys that authorize cross-chain transactions.

    This case study is classified as L1 (narrative only) because:

    • The attack occurred at the infrastructure layer, not the smart contract layer
    • The MPC key shards were stored on a compromised server
    • Transactions were authorized using legitimate protocol mechanisms
    • No on-chain reproduction is possible ΓÇö the contracts have no vulnerability

    Classifying Infrastructure Compromises¶

    Multichain joins other infrastructure-level bridge attacks that cannot be reproduced on-chain:

    Attack Root Cause Max Fidelity Why
    Multichain (Jul 2023) MPC key shard theft L1 Keys were legitimately authorized
    Horizon (Jun 2022) 2-of-5 multisig key compromise L2 Contracts behaved as designed; state is reproducible
    BNB Bridge (Oct 2022) IAVL proof bypass L2 Contract bug ΓÇö reproducible on fork

    Horizon and BNB Bridge achieve L2 because, while the root cause was off-chain, the contract state at the fork block can be demonstrated. Multichain cannot ΓÇö there is no forkable contract state that proves the vulnerability existed.

    Technical Details¶

    The Attack Vector¶

    1. MPC Key Shard Storage: Multichain operated MPC nodes that collectively held key shards for the bridge. N of M shards were required to authorize a cross-chain transaction.

    2. Server Compromise: An attacker gained access to the server hosting the MPC key shards. The exact method of compromise is not publicly confirmed, but theories include:

    3. Server-side vulnerability (SSH, API, or software exploit)
    4. Insider threat

    5. Social engineering of infrastructure staff

    6. Transaction Authorization: With sufficient key shards under their control, the attacker could authorize bridge transactions through the legitimate MPC signing protocol. The bridge smart contracts saw valid signatures and processed the transactions normally.

    7. Multi-Chain Drain: The attacker drained assets from Fantom, Moonriver, and Dogechain ΓÇö all chains where Multichain's MPC nodes had signing authority.

    Why This Cannot Be Reproduced On-Chain¶

    // The bridge contract has no vulnerability ΓÇö it checks:
require(MPCLibrary.verify(signature, message), "invalid signature");

// A legitimate signature from compromised keys passes this check
// The contract cannot distinguish "compromised but valid" from "honest"

    Reproducing this would require: 1. The actual compromised MPC key shards (not publicly available) 2. The MPC node software (proprietary to Multichain) 3. A simulation of the server compromise (security incident, not technical vulnerability)

    None of these are reproducible in an EVM testbed.

    Classification Notes¶

    Why L1 Is Correct¶

    Requirement Status Explanation
    Forkable chain Γ¥î No single chain ΓÇö assets were on Fantom, Moonriver, Dogechain
    Contract vulnerability Γ¥î Contracts have no bug ΓÇö keys were compromised
    Exploit transaction replay Γ¥î Transactions were legitimate, just authorized by wrong people
    Available evidence Γ£à Public postmortems document the infrastructure attack

    What This Pack Does¶

    The Multichain attack pack exists to: 1. Document the incident with all available sources 2. Classify it correctly as L1 infrastructure compromise 3. Bound Crucible's scope transparently ΓÇö not everything can be reproduced 4. Guide future researchers who might ask \"why isn't Multichain reproduced?\"
