    Research Workflow¶

    This document describes how Crucible should be used for academic bridge exploit reproduction.

    Reproduction Standard¶

    Do not describe a benchmark as "historically reproduced" until the attack pack has:

    1. A declared fidelity level.
    2. A complete evidence.yaml.
    3. Explicit fork blocks.
    4. Affected contract addresses.
    5. Historical exploit transactions, if the target fidelity is L3 or L4.
    6. Expected state, balance, and event assertions.
    7. A deterministic rerun comparison.
    8. An exported artifact bundle.

    Crucible can run scaffold packs before those fields are complete, but scaffold runs are engineering validation, not research evidence.

    Fidelity Levels¶

    Level Meaning Requirements
    L0 Narrative simulation only None (description only)
    L1 Local contract-level reproduction Attack plugin returns a result
    L2 Historical fork-state reproduction Chains exist in topology, trace artifacts present
    L3 Historical transaction-path reconstruction All L2 + exact_tx_replay=true, exploit transactions non-empty
    L4 Cross-chain causal reconstruction All L3 + multi-chain topology + bridge plugin + excluded components declared

    L4 bridge plugins available: | Plugin | Topology | Chains | Vulnerability | |---|---|---|---| | nomad_bridge | topologies/nomad_bridge.yaml | ethereum + polygon | committedRoot == 0x00 | | poly_network_bridge | topologies/poly_network_bridge.yaml | ethereum + polygon + bsc | Hash collision selector | | wormhole_bridge | topologies/wormhole_bridge.yaml | ethereum + polygon | Guardian bypass |

    Enforcing fidelity levels at runtime: 

    crucible experiment run experiments/nomad_replay.yaml

    Fidelity enforcement is enabled by default. Use --no-fidelity to skip.

    Attack Pack Lifecycle¶

    1. Scaffold¶

    For a complete step-by-step guide covering research, evidence filling, plugin writing, and validation, see docs/ATTACK_PACK_GUIDE.md.

    Quick path (from tx hash — 10 seconds):

    crucible attack extract <tx_hash> --name my_attack --chain ethereum

    This generates the complete scaffold (plugin.yaml, evidence.yaml, attack.py, scenario.yaml, ABI mapping).

    Manual path:

    attacks/<attack>/
  plugin.yaml
  evidence.yaml
  <attack>.py

    Validation:

    crucible plugin validate attacks/<attack>

    2. Evidence Fill¶

    Add verified:

    • postmortems
    • explorer links
    • contract addresses
    • vulnerable function names
    • fork block numbers
    • exploit transaction hashes
    • state/balance expectations

    3. Scenario Fill¶

    Update scenarios/<scenario>/scenario.yaml with concrete assertions.

    At minimum, every real attack should prove:

    • the vulnerable path succeeds
    • the attacker position improves
    • protected bridge state changes
    • the same action should fail after a patch, if patch validation is in scope

    4. Baseline Capture¶

    Start the topology (or use setup for fork + bootstrap in one step):

    crucible setup --topology bridge              # fork + bootstrap + prepare
# or manually:
crucible fork --topology bridge

    Capture:

    crucible baseline capture <scenario> --topology bridge
crucible baseline verify <scenario>

    5. Formal Run¶

    crucible experiment run experiments/<attack>_replay.yaml
crucible history --rebuild
crucible logs list

    6. Export & Verify¶

    crucible export <run_id>                          # export as directory
crucible export <run_id> --format tar-gz          # export as compressed archive
crucible verify datasets/<run_id>                  # verify bundle integrity (auto-detects bundle vs run)

    Archive datasets/<run_id>/ or the .tar.gz/.zip file.

    Determinism Check¶

    Run the same experiment twice after restoring the same baseline:

    crucible reset --scenario <scenario>
crucible experiment run experiments/<attack>_replay.yaml
crucible reset --scenario <scenario>
crucible experiment run experiments/<attack>_replay.yaml
crucible diff <run_a> <run_b>

    The diff result should show:

    • storage_equal: true
    • events_equal: true
    • gas_equal: true, unless the experiment declares a gas tolerance

    Patch Validation Workflow¶

    Patch validation should use two experiments:

    1. Vulnerable replay: exploit succeeds.
    2. Patched replay: exploit fails and normal bridge behavior still passes.

    The patch artifact should include:

    patches/<name>/
  patch.yaml
  validation.yaml
  contracts-or-scripts/

    Formal patch acceptance requires both positive and negative assertions.
